# PR 설명 작성 (PR Description Writing)

올바른 커밋 범위를 결정하고 PR 제목과 본문을 구성하는 방법입니다. `ce-commit-push-pr` 호출자에 의해 필요 시 로드됩니다 — 미리 로드하지 마십시오.

Step Pre-A는 커밋 범위, diff, 그리고 (기존 PR의 경우) 현재 PR 본문을 결정합니다. Step A부터 H는 Pre-A의 결과물이 컨텍스트에 있다고 가정합니다.

---

## Step Pre-A: PR 커밋 범위 및 diff 결정

설명에 포함되어야 할 커밋과 diff를 결정합니다. 이를 먼저 실행하십시오. Step A 이후의 단계들은 커밋 목록과 전체 diff가 컨텍스트에 있다고 가정합니다.

### 모드 (Mode)

- **Current-branch 모드.** HEAD와 레포의 기본 베이스를 비교하여 설명합니다. 호출자에게 명시적인 PR 참조가 없을 때 사용됩니다 (ce-commit-push-pr 전체 워크플로의 Step 6; PR 참조 없는 설명 전용 모드).
- **PR 모드.** 특정 PR의 커밋 범위를 설명합니다. DU-3(현재 브랜치의 기존 PR) 및 설명 전용 모드(사용자가 PR URL/번호를 입력함)에서 사용됩니다. PR 참조는 단순 번호, `#NN`, `pr:NN` 또는 전체 URL일 수 있습니다 — 호출자는 이를 `gh pr view <ref>`에 직접 전달합니다. (`gh pr view`는 번호와 URL을 기본적으로 수락합니다. `#NN` 및 `pr:NN`은 전달 전 `#` 또는 `pr:`을 제거해야 합니다.)

### PR 메타데이터 결정 (PR 모드 및 기존 PR이 있는 current-branch 모드)

```bash
gh pr view [<ref>] --json baseRefName,headRefOid,baseRefOid,headRepository,headRepositoryOwner,isCrossRepository,url,body,state
```

반환된 `state`가 `OPEN`이 아닌 경우, `"PR <number> is <state>; cannot generate description"`이라고 보고하고 우아하게 종료하십시오 — 설명을 발명하지 마십시오.

### 베이스 브랜치 및 원격 감지

PR 모드(또는 기존 PR이 있는 current-branch 모드)의 경우 위의 메타데이터에서 `baseRefName`을 사용합니다. 기존 PR이 없는 current-branch 모드의 경우 다음 우선순위에 따라 결정합니다:

1. **호출자 제공 베이스** (호출자가 `base:<ref>`를 전달한 경우) — 그대로 사용합니다. 해당 ref는 로컬에서 확인 가능해야 합니다.
2. **레포 기본 브랜치** — `git rev-parse --abbrev-ref origin/HEAD` 실행 후 `origin/` 접두사를 제거합니다.
3. **GitHub 메타데이터** — `gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'`.
4. **일반적인 이름** — `main`, `master`, `develop`, `trunk` 순서로 `git rev-parse --verify origin/<candidate>`를 시도합니다.

해결되지 않으면 사용자에게 베이스 브랜치를 지정해 달라고 요청하십시오.

**베이스 원격(base remote)**의 경우, PR URL에서 `owner/repo`를 파싱하거나(또는 PR이 없는 current-branch 모드에서는 현재 체크아웃의 레포 사용) `git remote -v` fetch URL과 매칭합니다 (`git@github.com:owner/repo` 및 `https://github.com/owner/repo` 형식을 모두 처리하고 `.git`은 제거함). 일치하는 원격 이름이 `<base-remote>`가 됩니다. **일치하는 원격이 없는 경우**(`upstream` 원격이 없는 포크 기반 PR), `origin`을 기본값으로 사용하지 마십시오 — `origin`은 잘못된 베이스와 비교하게 됩니다. 즉시 Case B로 건너뜁니다.

### Case A — 베이스 원격이 일치하는 경우, 로컬 git 시도

**참조(refs)가 아직 로컬에 없을 때만 fetch 하십시오.** 로컬에서 참조가 확인될 때 fetch를 건너뛰면 최근에 이미 fetch된 브랜치에서 스킬을 재실행할 때 오프라인 / 제한된 네트워크 / 만료된 인증 환경에서도 작업이 가능해집니다.

```bash
PR_HEAD_SHA=<headRefOid>   # current-branch 모드의 경우 HEAD 사용

if ! git rev-parse --verify <base-remote>/<baseRefName> >/dev/null 2>&1 \
  || ! git rev-parse --verify $PR_HEAD_SHA >/dev/null 2>&1; then
  git fetch --no-tags <base-remote> <baseRefName> $PR_HEAD_SHA
fi

MERGE_BASE=$(git merge-base <base-remote>/<baseRefName> $PR_HEAD_SHA) \
  && echo '=== COMMITS ===' && git log --oneline "$MERGE_BASE..$PR_HEAD_SHA" \
  && echo '=== DIFF ===' && git diff "$MERGE_BASE...$PR_HEAD_SHA"
```

Current-branch 모드의 경우 `$PR_HEAD_SHA`는 항상 로컬에 있는 `HEAD`이므로 베이스 참조만 fetch 하면 됩니다. 다운스트림 명령에서 명시적인 `$PR_HEAD_SHA`를 사용하면 `FETCH_HEAD`의 다중 참조 순서 문제를 피할 수 있습니다.

`git merge-base` 자체가 실패하는 경우(히스토리가 부족한 shallow clone 또는 완전히 무관한 히스토리), 억지로 진행하지 마십시오 — Case B로 넘어가 API 경로가 diff와 커밋을 생성하게 하십시오.

### Case A 내부 폴백 — SHA fetch 거부

일부 GHES 구성은 tip이 아닌 SHA의 fetch를 허용하지 않습니다. SHA fetch가 거부되면(PR 모드 전용 — current-branch 모드는 항상 로컬에 있는 HEAD를 사용함), `refs/pull/<number>/head`를 통해 PR 헤드를 fetch 하는 방식으로 폴백합니다:

```bash
git fetch --no-tags <base-remote> "refs/pull/<number>/head"
PR_HEAD_SHA=$(awk '/refs\/pull\/[0-9]+\/head/ {print $1; exit}' "$(git rev-parse --git-dir)/FETCH_HEAD")
```

그런 다음 새로운 `$PR_HEAD_SHA`로 merge-base + log + diff를 다시 실행합니다.

### 빈 범위 (Empty range)

결과 커밋 목록이 비어 있는 경우(HEAD가 이미 베이스에 병합되었거나 브랜치에 고유한 커밋이 없음), `"No commits to describe"`라고 보고하고 우아하게 종료하십시오 — 설명을 발명하지 마십시오.

## Step A: 커밋 분류

커밋 목록을 스캔하고 각 커밋을 분류합니다:

- **기능 커밋 (Feature commits)** -- PR의 목적을 구현합니다 (새로운 기능, 의도적인 리팩토링, 디자인 변경). 설명을 주도합니다.
- **수정 커밋 (Fix-up commits)** -- 반복 작업입니다 (코드 리뷰 수정, 린트 수정, 테스트 수정, 리베이스 해결, 스타일 정리). 독자에게는 보이지 않습니다.

설명 크기를 결정할 때 수정 커밋은 마음속으로 제외하십시오: 커밋이 12개이지만 수정 커밋이 9개인 브랜치는 실질적으로 3개 커밋의 PR입니다.

## Step B: 증거(evidence) 포함 여부 결정

본문에 증거 섹션을 포함할지 결정합니다.

**증거 포함이 가능한 경우**는 diff가 워크스페이스에서 시연 가능한 관찰 가능한 동작 변경을 포함할 때입니다: UI, CLI 출력, 실행 가능한 코드를 통한 API 동작, 생성된 아티팩트 또는 워크플로 출력.

**증거 포함이 불가능한 경우:**
- 문서 전용, 마크다운 전용, 변경 로그 전용, 릴리스 메타데이터, CI/설정 전용, 테스트 전용 또는 순수 내부 리팩토링
- 제공되지 않는 자격 증명, 유료/클라우드 서비스, 봇 토큰, 배포 전용 인프라 또는 하드웨어가 필요한 동작

**결정 로직:**

1. **기존 PR 본문에 이미지 임베드가 포함된 `## Demo` 또는 `## Screenshots` 섹션이 있는 경우:** 사용자의 포커스 요청이 이를 새로 고치거나 제거하라고 하지 않는 한 그대로 보존합니다. 보존된 블록을 본문에 포함합니다.
2. **기존 증거 블록이 없는 경우:** 호출자가 이미 증거를 캡처하여 전달하지 않은 한(예: `ce-demo-reel` URL) 증거 섹션을 완전히 생략합니다.

테스트 출력을 "Demo" 또는 "Screenshots"로 레이블링하지 마십시오. 보존된 증거 블록은 Compound Engineering 배지 앞에 배치합니다.

## Step C: 서사 프레임 구성 (Frame the narrative)

PR의 서사 프레임을 명확히 합니다:

1. **Before**: 무엇이 고장 났거나, 제한되었거나, 불가능했는가? (한 문장.)
2. **After**: 이제 무엇이 가능해졌거나 개선되었는가? (한 문장.)
3. **범위 근거 (Scope rationale)** (2개 이상의 분리 가능한 우려 사항이 있는 경우에만): 왜 이것들이 함께 배포되는가? (한 문장.)

이 프레임이 오프닝이 됩니다. 작고 단순한 PR의 경우 "After" 문장 하나가 설명 전체가 될 수도 있습니다.

## Step D: 변경 크기 평가

설명의 깊이를 선택하기 위해 크기(파일 수, diff 볼륨)와 복잡성(디자인 결정, 트레이드오프, 교차적인 우려 사항)을 평가합니다:

| 변경 프로필 | 설명 접근 방식 |
|---|---|
| 작고 단순함 (오타, 설정, 의존성 업데이트) | 1-2문장, 헤더 없음. 약 300자 이내. |
| 작지만 사소하지 않음 (버그 수정, 동작 변경) | 짧은 서사, 약 3-5문장. 두 가지 뚜렷한 우려 사항이 아니면 헤더 없음. |
| 중간 규모 기능 또는 리팩토링 | 서사 프레임(before/after/scope) 작성 후 변경 내용과 이유 설명. 디자인 결정을 언급. |
| 대규모 또는 아키텍처적으로 중요한 변경 | 서사 프레임 + 최대 3-5개의 디자인 결정 언급 + 1-2문장 테스트 요약 + 주요 문서 링크. 약 100라인 목표, 최대 150라인. 메커니즘이 많은 PR의 경우 Summary 레벨의 테이블을 사용하여 목록화하십시오. 각 메커니즘마다 H3 하위 섹션을 생성하지 마십시오. 리뷰어는 목록이 아닌 결정을 면밀히 검토합니다 — 상세 내용은 diff와 스펙 파일이 담고 있습니다. 10개 이상의 하위 섹션을 작성하고 있다면 테이블로 통합하십시오. |
| 성능 개선 | 가능한 경우 개선 전후 측정치를 포함합니다. 마크다운 테이블이 효과적입니다. |

확실하지 않을 때는 짧은 것이 좋습니다. 설명의 무게를 변경의 무게에 맞추십시오. 대규모 PR일수록 더 많은 내용을 담는 것이 아니라 더 많은 선택적 집중이 필요합니다.

## Step E: 작성 원칙 적용

### 작성 음성 (Writing voice)

레포에 문서화된 스타일 선호도가 컨텍스트에 있다면 이를 따르십시오. 그렇지 않은 경우:

- 능동태. 대시(--)를 사용하지 말고 마침표, 쉼표, 콜론 또는 괄호를 사용하십시오.
- 문장 길이를 다양하게 하십시오. 비슷한 길이의 문장을 세 번 연속해서 쓰지 마십시오.
- 주장을 한 뒤 바로 설명하지 마십시오. 독자를 믿으십시오.
- 평이한 영어 사용. 기술 용어는 괜찮지만 비즈니스 용어는 지양하십시오.
- 채우기 문구 지양: "it's worth noting", "importantly", "essentially", "in order to", "leverage", "utilize."
- 숫자는 문자가 아닌 숫자로 표기합니다 ("3 files").

### 작성 원칙

- **가치 우선 (Lead with value)**: 무엇이 이동했거나 이름이 바뀌었는지가 아니라, 무엇이 가능해졌거나 해결되었는지로 시작하십시오. 더 미묘한 실패는 결과("이제 증거 캡처가 웹 앱뿐만 아니라 CLI 도구와 라이브러리에서도 작동합니다") 대신 메커니즘("하드코딩된 캡처 블록을 계층화된 스킬로 교체합니다")으로 시작하는 것입니다.
- **고립된 오프닝 문단 지양**: 설명에서 어디에든 `##` 헤더를 사용한다면 오프닝도 헤더(예: `## Summary`) 아래에 있어야 합니다. 섹션이 없는 짧은 설명의 경우 헤더 없는 문단도 괜찮습니다.
- **여정이 아닌 최종 결과 설명**: 설명은 어떻게 도달했는지가 아닌 최종 상태를 다룹니다. 반복 작업 내역, 디버깅 단계, 중간 실패 또는 개발 중 발견하여 수정한 버그는 포함하지 마십시오. 이는 기존 PR에 대해 재생성할 때도 동일하게 적용됩니다: 마지막 버전 이후 무엇이 바뀌었는지가 아니라 현재 상태에서 다시 작성하십시오. 예외: 디자인 선택을 이해하는 데 필수적인 프로세스 세부 사항.
- **커밋이 충돌할 때는 최종 diff를 신뢰**: 커밋 목록은 보조 컨텍스트이지 소스 오브 트루스(source of truth)가 아닙니다. 커밋이 나중에 수정되거나 되돌려진 중간 단계를 설명한다면, 전체 브랜치 diff를 바탕으로 최종 상태를 설명하십시오.
- **명백하지 않은 것 설명**: diff가 자명하다면 나레이션하지 마십시오. 왜 이 접근 방식을 택했는지, 무엇이 거절되었는지, 리뷰어가 무엇을 주의 깊게 봐야 하는지 등 diff가 보여주지 않는 것에 공간을 할애하십시오.
- **구조가 가치를 더할 때만 사용**: 헤더, 불렛, 테이블은 필수 템플릿 섹션이 아니라 이해를 돕기 위한 도구입니다.
- **데이터를 위한 마크다운 테이블**: 전후 비교, 성능 수치 또는 옵션 트레이드오프는 테이블로 잘 전달됩니다.
- **빈 섹션 지양**: 섹션이 해당하지 않으면 생략하십시오. "N/A"나 "None"을 쓰지 마십시오.
- **테스트 계획 — 명백하지 않을 때만**: 리뷰어가 생각하지 못할 에지 케이스, 확인하기 어려운 동작 또는 특정 설정이 필요한 경우에만 포함하십시오. "테스트 실행"이 유일한 유용한 가이드라면 생략하십시오. 브랜치에 테스트 파일이 추가된 경우 무엇을 다루는지 명시하십시오.
- **Commits 섹션 금지**: GitHub는 이미 자체 탭에서 커밋 목록을 보여줍니다. PR 본문의 Commits 섹션은 컨텍스트 추가 없이 이를 중복할 뿐입니다. 커밋 순서나 배포 근거를 설명하는 주석이 필요한 경우가 아니면 생략하십시오.
- **Review / 프로세스 섹션 금지**: 리뷰어가 어떻게 리뷰해야 하는지 설명하는 섹션(체크리스트 등)을 포함하지 마십시오. 프로세스는 리뷰어가 코드를 평가하는 데 도움이 되지 않습니다. 정밀 조사가 필요한 특정 사항은 해당 변경 사항과 인라인으로 언급하십시오.

### 시각적 소통 (Visual communication)

독자가 산문만으로는 멘탈 모델을 재구성하기 어려울 정도로 구조적으로 복잡한 변경인 경우에만 시각적 도움을 포함하십시오.

**핵심 구분 — 구조(topology) vs. 병렬 변형(parallel variation):**

- 변경 사항에 **위상(topology)**이 있는 경우(지향성 관계 - 호출, 흐름, 의존성, 상태 전환, 데이터 경로를 가진 컴포넌트) **Mermaid 다이어그램**을 사용하십시오. 다이어그램은 테이블이 할 수 없는 방식으로 "A는 B와 통화하고, B는 C와 통화하며, C는 A에게 응답하지 않는다"를 표현합니다.
- 변경 사항에 **단일 형태의 병렬 변형**이 있는 경우(속성은 공유하지만 값이 다른 N개의 항목) **마크다운 테이블**을 사용하십시오. 테이블은 "옵션 1의 비용은 X, 옵션 2는 Y, 옵션 3은 Z"를 깔끔하게 표현합니다.

아키텍처 변경은 거의 항상 위상(컴포넌트 + 엣지)이므로 보통 Mermaid가 정답입니다 — "상호작용하는 컴포넌트" 테이블은 엣지를 잃어버리고 단순 목록이 됩니다. 테이블은 전후 측정치, 옵션 트레이드오프, 플래그 매트릭스, 설정 열거와 같은 순수 병렬 데이터를 위해 남겨두십시오.

**언제 포함할 것인가 (아키텍처/흐름의 경우 테이블 대신 Mermaid 선호):**

| PR 변경 내용... | 시각적 도움 |
|---|---|
| 3개 이상의 상호작용하는 컴포넌트를 건드리는 아키텍처 (컴포넌트 간에 누가 누구를 호출하는지, 누가 무엇을 소유하는지, 어떤 스킬이 누구에게 위임하는지 등 *지향성 관계*가 있음) | **Mermaid** 컴포넌트 또는 상호작용 다이어그램. 테이블로 대체하지 마십시오 — 테이블은 엣지(edge)를 보여줄 수 없습니다. |
| 명백하지 않은 순서를 가진 다단계 워크플로 또는 데이터 흐름 | **Mermaid** 흐름도 |
| 3개 이상의 상태와 복잡한 전이를 가진 상태 머신 | **Mermaid** 상태 다이어그램 |
| 3개 이상의 관련 엔티티가 있는 데이터 모델 변경 | **Mermaid** ERD |
| 전후 성능 또는 동작 측정치 (동일 지표, 다른 값) | **마크다운 테이블** |
| 옵션 또는 플래그 트레이드오프 (변형 간에 평가된 동일 속성) | **마크다운 테이블** |
| 기능 매트릭스 / 호환성 그리드 | **마크다운 테이블** |

**확실하지 않을 때는 자문하십시오: "정보에 엣지(A → B)가 있는가, 아니면 행(속성 × 변형)이 있는가?"** 엣지 → Mermaid. 행 → 테이블. 아키텍처는 정의상 거의 항상 엣지를 가집니다.

**언제 시각적 도움을 건너뛸 것인가:**
- 크기 평가가 "1-2문장"으로 분류될 때
- 산문으로 이미 명확하게 전달될 때
- 다이어그램이 단순히 diff를 시각적으로 재진술할 뿐일 때
- 기계적인 변경 (이름 변경, 의존성 업데이트, 설정, 포맷팅)

**포맷 세부 사항:**
- **Mermaid** (위상 구조의 기본값). 보통 5-10개의 노드, 정말 복잡한 변경의 경우 최대 15개. `TB` 방향을 사용하십시오. 소스 코드는 폴백으로서 읽기 쉬워야 합니다.
- **ASCII 다이어그램** 박스 내에 풍부한 내용이 필요한 주석 달린 흐름의 경우. 최대 80열.
- **마크다운 테이블** 순수 병렬 변형 데이터의 경우에만.
- 별도 섹션이 아닌 관련 있는 위치에 인라인으로 배치하십시오.
- 산문이 시각적 내용과 충돌할 때는 산문이 우선합니다.

포함하기 전 생성된 다이어그램을 변경 내용과 대조하여 검증하십시오.

### 번호 지정 및 참조 (Numbering and references)

PR 설명에서 리스트 항목 앞에 `#`를 붙이지 마십시오 — GitHub은 `#1`, `#2`를 이슈 참조로 해석하여 자동 링크를 생성합니다.

실제 GitHub 이슈나 PR을 참조할 때는 `org/repo#123` 또는 전체 URL을 사용하십시오. 확인된 경우가 아니면 단순 `#123`은 사용하지 마십시오.

### 사용자 포커스 적용

사용자가 포커스 힌트를 제공한 경우(예: "벤치마크를 강조해줘", "이것은 기능이 아니라 마이그레이션으로 읽혀야 해"), 이를 diff에서 도출된 서사와 함께 반영하십시오. 포커스를 대체의 기준이 아닌 조향(steering)의 기준으로 취급하십시오: diff가 지원하지 않는 내용을 발명하지 말고, 단순히 포커스에서 언급하지 않았다는 이유로 diff가 요구하는 중요한 내용을 억제하지 마십시오. 포커스와 diff가 실질적으로 충돌하는 경우(예: 포커스는 "벤치마킹 포함"을 요구하지만 diff에 벤치마크가 없음), 내용을 꾸며내지 말고 사용자에게 충돌 사실을 알리십시오.

## Step F: 제목 작성

제목 형식: `type: description` 또는 `type(scope): description`.

- **Type**은 파일 확장자나 diff 모양이 아닌 의도에 따라 선택됩니다. 새로운 기능은 `feat`, 버그 수정은 `fix`, 동작 보존 리팩토링은 `refactor`, 문서 전용은 `docs`, 도구/유지보수는 `chore`, 성능은 `perf`, 테스트 전용은 `test`. `fix`와 `feat`가 둘 다 어울릴 때는 `fix`를 기본값으로 하십시오: 고장 났거나 누락된 동작을 바로잡는 변경은 코드를 추가하여 구현되더라도 `fix`입니다. `feat`는 사용자가 이전에 수행할 수 없었던 능력을 위해 남겨두십시오. 사용자가 재정의할 수 있습니다.
- **Scope** (선택 사항)는 스킬/에이전트 이름, CLI 영역 또는 공유 영역과 같이 가장 좁고 유용한 레이블입니다. 단일 레이블이 명확성을 더하지 않으면 생략하십시오.
- **Description**은 명령형, 소문자로 시작하며 총 72자 이내로 작성합니다. 마지막에 마침표를 찍지 않습니다.
- 레포의 최근 커밋에 문서화된 제목 컨벤션이 있다면 그에 맞추십시오.

브레이킹 체인지(Breaking changes)는 `!`를 사용하거나(예: `feat!: ...`) 본문 하단에 `BREAKING CHANGE:` 푸터를 작성합니다. 일부 릴리스 도구에서 메이저 버전 자동 업데이트를 트리거하므로 사용자 확인 없이 적용하지 마십시오.

## Step G: 본문 구성

본문을 다음 순서대로 조립하십시오:

1. **오프닝** -- Step D에서 선택한 깊이에 따른 Step C의 서사 프레임. 다른 곳에서 `##` 헤더를 사용한다면 헤더(예: `## Summary`) 아래에 두고, 그렇지 않으면 헤더 없는 문단으로 둡니다.
2. **본문 섹션** -- 해당 변경에 가치를 더하는 섹션만 포함: 변경 내용과 이유, 디자인 결정, 테이블 برای 데이터, 시각적 도움 등. 빈 섹션은 완전히 생략합니다.
3. **테스트 계획** -- 작성 원칙에 따라 명백하지 않은 경우에만 포함합니다. 그렇지 않으면 생략합니다.
4. **증거 블록** -- Step B에서 보존되거나 새로 캡처된 블록이 있는 경우에만 포함합니다. 가짜 내용을 넣거나 플레이스홀더를 두지 마십시오.
5. **Compound Engineering 배지** -- `---` 가로선으로 구분된 배지 푸터를 추가합니다. 배지가 이미 포함된 기존 본문을 재생성하는 경우 생략합니다.

**배지 (Badge):**

```markdown
---

[![Compound Engineering](https://img.shields.io/badge/Built_with-Compound_Engineering-6366f1)](https://github.com/juyohan/compound-engineering-plugin)
![HARNESS](https://img.shields.io/badge/MODEL_SLUG-COLOR?logo=LOGO&logoColor=white)
```

**하네스(Harness) 조회:**

| 하네스 | `LOGO` | `COLOR` |
|---------|--------|---------|
| Claude Code | `claude` | `D97757` |
| Codex | (logo 파라미터 생략) | `000000` |
| Gemini CLI | `googlegemini` | `4285F4` |

**모델 슬러그 (Model slug):** 공백을 언더바(_)로 바꿉니다. 컨텍스트 윈도우와 사고 레벨(thinking level)을 알 수 있다면 괄호 안에 추가합니다. 마크다운 이미지 URL 내부의 리터럴 괄호 `(` 및 `)`는 `%28` 및 `%29`로 URL 인코딩하십시오 — 인코딩되지 않은 괄호는 일부 커밋 메시지 파서(특히 release-please의 컨벤션 커밋 파서)를 손상시켜 커밋 전체를 실패하게 만들고 변경 로그에서 조용히 누락시킵니다. 예시: `Opus_4.6_%281M,_Extended_Thinking%29`, `Sonnet_4.6_%28200K%29`, `Gemini_3.1_Pro`.

## Step H: 압축 패스

적용 전 작성된 본문을 다시 읽고 다음을 잘라내십시오:

- 본문 섹션이 `## Summary`에 있는 내용을 반복한다면 제거하십시오. 요약과 diff만으로 독자에게 전달되어야 합니다.
- "Testing" 또는 "Test plan"이 2문단을 초과하면 불렛 포인트로 압축하십시오.
- "Commits" 섹션이 커밋 로그를 나열한다면 제거하십시오 — GitHub은 자체 탭에서 이를 보여줍니다.
- "Review" 또는 프로세스 지향 섹션이 리뷰 방법을 나열한다면 제거하십시오. 정말 명백하지 않은 리뷰 힌트는 관련 변경 사항과 인라인으로 이동시키십시오.
- 본문에 각 메커니즘을 설명하는 5개 이상의 H3 하위 섹션이 있다면, 하나의 헤더 아래에 메커니즘당 하나의 테이블 행으로 통합하십시오. 산문 형식의 H3 언급은 2-3개의 진정한 디자인 결정을 위해 남겨두십시오.
- 본문이 크기 평가 테이블의 타겟을 30% 이상 초과하면 가장 긴 비요약 섹션을 절반으로 압축하십시오.

**가치 우선 체크 (Value-lead check).** 요약의 첫 문장을 다시 읽으십시오. 무엇이 이동했거나, 이름이 바뀌었거나, 추가되었는지를 설명하고 있다면("이 PR은 3계층 자동 수정을 도입합니다..."), 무엇이 가능해졌거나 무엇이 고장 났다가 고쳐졌는지를 리드하도록 다시 작성하십시오("기본 문서 리뷰는 사용자 판단이 필요한 14개 이상의 발견 사항을 생성했으나, 이 PR은 이를 4-6개로 줄입니다.").

대규모 PR은 포괄성이 아닌 선택적 집중을 통해 가치를 얻습니다.
