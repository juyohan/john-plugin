# 합성 요약 (Synthesis Summary)

**합성(Synthesis) ≠ 플랜 문서.** 합성은 플랜 작성(Phase 5.2) 단계가 입력으로 소비하는 범위/결정 체크포인트입니다. 플랜 자체는 확정된 합성을 바탕으로 작성됩니다. 합성은 결정 수준(decision-level)에서 범위나 플랜 단계의 결정 사항(수정할 파일, 확장할 패턴, 미뤄진 항목 등)을 명시합니다. 플랜 본문은 이러한 결정 사항을 플래닝 규칙에 따라 방향성 있는 스케치(파일 경로, 테스트 시나리오, 접근 방식 설명)로 확장하지만, 결코 정확한 메서드 시그니처, 프레임워크 문법 또는 코드 사양(Phase 4.3에서 플랜 본문 내 금지됨)으로 확장하지 않습니다. 만약 합성이 플랜 미리보기처럼 읽힌다면 잘못 형성된 것입니다 — 범위/결정 전용으로 다시 정리하십시오.

**3개 버킷 구조는 채팅 시의 산출물일 뿐입니다.** 이는 사용자와의 대화에서 범위 확인 역할을 수행한 후, Phase 5.2에서 플랜이 작성될 때 소멸됩니다: Stated 내용은 Requirements에, Inferred 내용은 Key Technical Decisions / Implementation Units(대화형 모드) 또는 `## Assumptions`(자동 모드)에, Out-of-scope 내용은 Scope Boundaries에 반영됩니다. 플랜에는 별도의 `## Synthesis` 섹션이 없으며, 산문 형태의 요약만이 `## Summary`로 임베드됩니다. 라우팅에 대해서는 아래 "확정 후 문서 형태"를 참조하십시오.

이 콘텐츠는 ce-plan에서 합성 요약 단계가 실행될 때 로드됩니다. 두 가지 변형이 있으며, 구조는 공유하지만 시점과 콘텐츠 집중도가 다릅니다:

- **단독 변형 (Solo variant)** (Phase 0.7): Phase 0.4 부트스트랩 및 Phase 0.6 깊이 분류 이후, Phase 1 조사가 시작되기 전에 실행됩니다. 하위 에이전트 발송 비용을 쓰기 전에 범위 오해를 잡아냅니다. 문제 프레임, 의도된 동작, 성공 기준, 포함/제외 범위 등 전체적인 범위를 다룹니다.
- **브레인스토밍 기반 변형 (Brainstorm-sourced variant)** (Phase 5.1.5): Phase 1 조사 이후, Phase 5.2 플랜 작성 전에 실행됩니다. 플랜 단계의 결정 사항(수정할 파일/모듈, 확장하거나 새로 도입할 패턴, 테스트 범위, 리팩토링 범위)에 집중합니다. 브레인스토밍에서 검증된 'WHAT(무엇을)'은 당연한 것으로 간주하고 다시 언급하지 않습니다.

두 변형 모두 3개 버킷 구조, 개방형 산문 피드백, 소프트 컷(soft-cut) 동작 및 문서 형태 라우팅을 공유합니다. 비대화형(headless) 모드에서는 두 변형 모두 합성을 구성하지만 사용자 확정 단계를 건너뜁니다 — 추론된(Inferred) 베팅은 Key Technical Decisions가 아닌 플랜의 `## Assumptions` 섹션으로 라우팅됩니다. 전체 라우팅은 아래 "비대화형 모드 (공통)"를 참조하십시오.

## 3개 버킷 구조 (공통)

모든 합성은 라벨이 지정된 세 개의 버킷으로 구조화됩니다. 항목이 의미상 양쪽에 모두 해당하는 경우 두 버킷에 모두 나타날 수 있습니다 — 포함 후 제외되는 항목을 Inferred로 표시하여 독자가 에이전트의 추론 과정을 볼 수 있게 하십시오.

- **Stated (명시됨)** — 사용자가 직접 말한 내용 (원본 프롬프트, 이전 대화, 대화 답변 또는 상류의 브레인스토밍 문서). 이 항목들은 명확한 사용자 언어 앵커를 가집니다.
- **Inferred (추론됨)** — 에이전트가 공백을 메우기 위해 가정한 내용. 사용자가 명시적으로 언급하지 않은 범위 경계, 의도에서 유추한 성공 기준, 짧은 인터뷰에서 조사하지 못해 내린 기술적 가정 등. "Inferred" 목록은 가장 실행 가능한 버킷입니다 — 여기에 있는 항목들은 사용자가 바로잡을 수 있는 에이전트의 베팅입니다.
- **Out of scope (범위 외)** — 의도적으로 제외된 항목. 에이전트가 고려했으나 포함하지 않기로 결정한 인접 작업, 리팩토링, 있으면 좋은 기능, 향후 작업 항목 등.

## 합성의 구조적 규율 (공통)

두 변형 모두 이 구조적 규칙을 공유합니다. 이는 합성이 범위 체크포인트가 아닌 Phase 5.2(플랜 작성)의 미리보기가 되어버리는 실패 모드를 방지합니다.

**산문 요약은 합성 섹션 내부에 위치해야 합니다.** 리드인(lead-in) 라인 바로 뒤, Stated 버킷 앞에 위치하며 합성 위에 별도의 산문 블록으로 두지 않습니다. 합성 위에 긴 산문(접근 방식 피치, 수정할 파일 불렛, 근거 블록)을 배치하는 것은 구조를 뒤집는 것입니다: 제안 내용에 대한 1~3줄의 해설이 합성이 되는 것이 아니라, 합성이 제안에 대한 각주가 되어버립니다.

**안티 패턴: 플랜 피치로서의 합성.** 플랜 본문 내용 — 파일 경로, 코드 형태, 센티널 문자열, 정확한 오류 메시지, "권장 사항" / "X일 때의 동작" / "왜 이 형태인가" 근거 — 은 어디에 나타나든 합성의 영역이 아닙니다: 버킷 위의 블록, 산문 요약 내부, Inferred 불렛의 주석이나 하위 불렛 어디에도 해당하지 않습니다. 위치 규칙과 내용 규칙은 독립적입니다: 구조적으로 올바른 위치(버킷 불렛 내부)에 있다고 해서 플랜 본문 내용이 정당화되지 않습니다. 이를 작성하고 있다면 즉시 멈추십시오. 해당 내용은 다음 단계인 Phase 5.2(플랜 작성)의 영역입니다. 합성은 범위/결정 체크포인트이며, 세 개의 버킷과 1~3줄의 해설로 구성됩니다. 구현 세부 사항이 합성의 어느 곳으로든 유출되는 것은 Phase 1~4(조사 및 구조화)와 Phase 5.2(플랜 작성)가 합성 확정 단계로 붕괴되었음을 의미합니다.

**수정은 확정이 아닙니다.** 사용자가 수정을 요청하면(아무리 사소한 변경이라도), 변경 사항을 통합하고 수정된 합성을 다시 제시한 후, 플랜을 작성하기 전에 명시적인 확정을 기다리십시오. 루프는 다음과 같습니다:

1. 합성 제시 → 사용자 응답
2. 사용자 확정 → 플랜 작성
3. 사용자 수정 → 통합, 수정된 합성 재제시, 1단계로 이동

플랜 작성(Phase 5.2)은 명시적인 확정이나 소프트 컷 차단 질문의 "진행" 옵션 이후에만 실행됩니다. 에이전트가 충분히 이해했다고 느끼는 작은 수정이라 하더라도 수정 직후에 바로 작성하지 마십시오 — 확정 단계가 합성을 "에이전트의 마지막 제안"이 아닌 **확정된 것**으로 만듭니다.

## 세분성: 결정을 명명하되, 확장하지 말 것 (공통)

각 Inferred 불렛은 사용자가 **코드를 읽지 않고도** 긍정하거나 부정할 수 있어야 합니다. 사용자가 "예" 또는 "대신 X를 원함"이라고 말할 수 있는 수준의 세분성으로 결정을 명명하십시오. 그보다 구체적인 것은 Phase 5.2의 작업이지 합성의 작업이 아닙니다.

**허용됨** (이러한 것들이 내려지는 결정인 경우):
- 파일 / 모듈 이름 — "어디에 둘 것인가"가 선택 사항일 때 "matcher의 skip 필터"
- 패턴 이름 — "기존 패턴 확장 vs 신규 도입"이 선택 사항일 때 "기존 event-skip 패턴 확장"
- 컬럼 / 테이블 이름 — "어떤 소스인가"가 선택 사항일 때 "user-TZ" 또는 "destination-calendar TZ"
- 접근 자세 — "어떤 전략인가"가 선택 사항일 때 "Google 측 폴백을 포함한 DB 측 쿼리"

**허용 안 됨** (변형에 관계없이 항상 플랜 본문의 영역):
- 라인 번호 (`route.ts:249-255`)
- 정확한 메서드 시그니처, 호출 그래프 또는 구현 흐름 ("상단에서, include/exclude 평가 전에, ...을 반환하며")
- 정확한 JSON / 응답 형태 (`{pause, cleanup: {eventsDeleted, eventsFailed, errors}}`)
- HTTP 상태 코드 (`409`, `404`, `403`)
- 정확한 이벤트 / 활동 로그 / 타입 이름 (`userPauseSet/userPauseEdited/...`)
- 오류 메시지나 UI 라벨의 정확한 문구
- SQL 구문 또는 쿼리 본문

변형에 따라 선이 약간 다르게 그어집니다. **Solo (Phase 0.7)**는 더 높은 수준에 머뭅니다 — 브레인스토밍의 'WHAT'이 아직 검증되지 않았으므로 파일/모듈 이름은 일반적으로 너무 구체적입니다. "syncRules 테이블"이 아니라 "rule 엔티티"의 관점에서 이야기하십시오. **Brainstorm-sourced (Phase 5.1.5)**는 파일 / 모듈 / 패턴 / 컬럼 수준이 플랜 단계의 결정 사항인 경우 허용되지만, 구현 흐름의 세부 사항은 허용되지 않습니다.

### 나쁜 예 vs 좋은 예

| 합성 내의 플랜 본문 (틀림) | 결정 수준 (맞음) |
|---|---|
| 시간대 소스: `users.timezone` (IANA), null인 경우 대상 캘린더 TZ로 폴백. 조사 결과 `useTimezoneSync`와 `ProtectionStatsCalculator`가 패턴을 확립함. | 시간대 소스: user-TZ (브레인스토밍의 잠정적 경향을 뒤집음 — 조사 결과 확립된 인프라 및 패턴 전례를 발견함) |
| Skip 필터는 `RuleMatcher.eventMatchesRule` 상단에 include/exclude 평가 전에 들어가며, 기존 `filteredReason` 메커니즘을 사용함. | Skip 필터는 매처 내의 기존 event-skip 패턴을 확장함 (신규 메커니즘 도입 대신) |
| 재활성화 가드: `[ruleId]/route.ts` PATCH에서의 명시적 안전장치 — `isActive: false → true`일 때 기존 핸들러가 `status/pausedAt/pausedReason`을 지움. | 재활성화 가드: isActive 토글의 기존 시스템 일시정지 해제 경로를 통해 일시정지 윈도우 상태가 유지됨 |
| 부분 정리 실패 응답: `{pause, cleanup: {eventsDeleted, eventsFailed, errors}}`; 정리 결과와 관계없이 일시정지 윈도우는 유지됨. | 부분 정리 실패: 일시정지 윈도우 유지; 부분 실패 응답은 기존 rule-edit 전례를 따름 |

테스트 방법: Inferred 불렛을 읽는 사람이 코드를 읽을 필요 없이 이를 긍정하거나 부정할 수 있는가? 만약 불렛을 평가하기 위해 컬럼 이름, 메서드 이름 또는 호출 그래프를 찾아봐야 한다면 세분성이 잘못된 것입니다 — 그것은 플랜 본문 내용입니다.

## 단독 변형 (Solo variant - Phase 0.7)

다음의 경우에만 실행됩니다:
- Phase 0.2에서 상류 브레인스토밍 문서를 찾지 못함
- 그리고 Phase 0.4에서 ce-plan에 머무름 (ce-debug, ce-work 또는 유니버설 플래닝으로 라우팅되지 않음)
- 그리고 Phase 0.5가 통과됨 (미해결 블로커 없음)
- 그리고 Phase 0.1 패스트 패스(정상 재개, 심화 의도)가 아님

각 가드는 SKILL.md의 명시적인 조건문이며 암시적인 것이 아닙니다. R2 단독 변형은 재개/심화, 라우팅 아웃 또는 브레인스토밍 기반 경로에서 실행되지 않습니다.

**콘텐츠 집중**: 전체 범위 합성. Phase 0.4 부트스트랩은 설계상 간결하므로("한두 개의 명확한 질문"), Phase 0.7이 실행되기 전에 에이전트는 상당한 추론을 내린 상태입니다. 따라서 "Inferred" 목록이 여기서는 특히 중요합니다 — 에이전트의 베팅을 명시적으로 표면화하십시오.

**풍부한 컨텍스트 호출에 대한 경고.** 추론 소스가 단순한 Phase 0.4 부트스트랩이 아닌 경우 — 예: 이전의 대화 내 검증 에이전트, 동일 세션 내에서 완료된 인접 작업 단위 또는 대화 내에 이미 존재하는 플래닝 아티팩트 — "베팅을 명시적으로 표면화"하는 것은 해당 자료를 그대로 쏟아내게 만들기 쉽습니다. 이 경우 세분성 규칙은 완화되는 것이 아니라 오히려 강화됩니다: 에이전트는 노출할 자료가 더 많은 것이 아니라 압축할 자료가 더 많은 것입니다. 상류에서 이미 검증된 베팅은 Inferred가 아니라 **Stated**입니다. 세부 사항이 플랜 본문에 속하는 베팅은 상류 컨텍스트가 얼마나 많은 정보를 제공했는지와 관계없이 여기서 결정 수준에서 명명됩니다. 최근 턴에서 상세한 코드, 파일 경로 또는 조사 아티팩트가 생성되었다면, 합성 초안이 과도하게 공유될 것을 예상하고 제시 전에 선제적으로 압축하십시오.

**왜 작성이 아닌 조사 전인가**: 범위가 틀렸다면 조사 노력이 낭비될 것이기 때문입니다. 하위 에이전트 발송(Phase 1.1의 repo-research-analyst, learnings-researcher 등) 전에 범위 오류를 잡아내면 토큰과 시간 비용을 절약할 수 있습니다.

### 프롬프트 템플릿 (solo)

**산문 요약 규율 (모든 티어 필수):** **플랜이 어떤 범위를 타겟팅할 것인지** 설명하는 1~3줄의 평이한 산문 요약으로 시작하십시오. 미래 지향적(무엇이 계획될 것인가)이어야 하며, 회고적(Phase 0.4 부트스트랩에서 무엇이 논의되었는가)이어서는 안 됩니다. 산문의 역할은 사용자가 불렛을 읽기 전에 의도와 패턴 매칭을 하도록 돕는 것입니다 — 단독 호출은 작성 전 대화가 최소화되므로 산문 요약이 특히 중요합니다. Lightweight 티어조차 게슈탈트(gestalt)의 이점이 있습니다. 유일하게 정당한 생략 사례는 정말 사소한 경우(예: 프롬프트 자체가 완전한 범위 문장이었고 합성이 프롬프트를 그대로 반영하는 한두 개의 Stated 불렛인 경우)뿐입니다.

**군더더기 제거 가이드:** 군더더기 없는 평이한 언어로 실제 계획 중인 내용을 먼저 제시하십시오. "포괄적인," "사려 깊은," "실질적인"과 같은 수식어는 사용하지 마십시오. 사용자의 프롬프트를 재진술하지 마십시오. 군더더기 없이 1~3줄로 범위를 말할 수 없다면 합성은 아직 준비되지 않은 것입니다.

```
요청하신 내용과 Phase 0.4 부트스트랩을 바탕으로, 제가 계획하고자 하는 범위는 다음과 같습니다:

[1~3줄 산문 요약 — 플랜이 어떤 범위를 타겟팅할 것인지 평이한 언어로 기술. 모든 티어 필수; 합성이 프롬프트를 반영하는 ≤ 2개의 불렛인 정말 사소한 경우에만 생략.]

**Stated (명시됨)** (사용자 입력 및 대화 내용):
- [항목]

**Inferred (추론됨)** (가정으로 메운 공백 — Phase 0.4는 설계상 간결하므로 이 리스트가 중요합니다. 제가 틀린 부분이 있다면 알려주세요):
- [문제 프레임 추론]
- [성공 기준 추론]
- [범위 경계 추론]
- [기술적 접근 가정]

**Out of scope (범위 외)** (의도적으로 제외됨):
- [인접 작업]
- [리팩토링]
- [있으면 좋은 기능]

의도하신 것과 일치하나요? 추가, 제거, 방향 수정이 필요한 부분이나 제가 틀린 부분을 알려주세요. 확인해 주시면 바로 진행하겠습니다. (이 작업이 처음에 생각하신 것보다 크다면 `/ce-brainstorm`으로 전환할 수도 있습니다 — 여기서 멈추고 로드해 드리겠습니다.)
```

사용자 응답에는 산문을 사용하십시오 (`AskUserQuestion` 메뉴 사용 안 함). 근거는 SKILL.md의 Interaction Rule 5(a)입니다.

## 브레인스토밍 기반 변형 (Brainstorm-sourced variant - Phase 5.1.5)

다음의 경우에만 실행됩니다:
- Phase 0.2에서 상류 브레인스토밍 문서를 찾음 (브레인스토밍 기반 호출)
- 그리고 Phase 0.1 패스트 패스가 아님

**콘텐츠 집중**: 플랜 단계의 결정 사항만 다룹니다. 브레인스토밍 + R1 합성이 이미 '무엇을(WHAT)' 구축할지 검증했으므로, 이 합성은 플랜이 해당 작업을 '어떻게(HOW)' 수행할지 — 브레인스토밍이 내리지 않은 결정들 — 를 표면화합니다.

표면화할 항목:
- **수정할(그리고 수정하지 않을) 파일/모듈** — 구현이 어디까지 닿는지
- **확장할 패턴 vs 새로 도입할 패턴** — 확정된 범위 내에서 에이전트가 내린 아티팩트적 결정 (어느 한쪽으로 치우치지 않고 R2의 콘텐츠에 집중)
- **테스트 범위** — 기존의 테스트되지 않은 코드 중 이 작업의 테스트 범위에 포함되거나 제외되는 것
- **리팩토링 범위** — 인접한 정리 작업 중 후속 작업으로 미뤄지는 것 vs 현재 diff에 포함되는 것
- **교차적 영향** — 인증, 마이그레이션, 공유 타입 등이 수정되는 경우

**요약 섹션이 아닌 문서 본문에서 읽어옴**: 브레인스토밍 문서에는 `## Synthesis` 섹션이 없습니다 (합성은 ce-brainstorm 시의 채팅 산출물이며, 산문 요약만이 `## Summary`로 임베드됨). Phase 5.1.5는 브레인스토밍 문서의 본문 섹션 — Summary, Problem Frame, Requirements, Key Decisions, Scope Boundaries — 과 Phase 1 조사에서 플랜 단계의 결정을 도출합니다. 레거시 `## Synthesis` 섹션이 있는 오래된 브레인스토밍 문서도 잘 작동합니다. 해당 내용은 보조적인 것으로 취급되며 본문 섹션이 우선권을 가집니다.

**왜 조사 전이 아닌 작성 전인가**: 브레인스토밍 문서 + R1 합성이 이미 '무엇을' 검증했으므로 조사가 잘 타겟팅되어 있습니다. 플랜 단계의 결정은 조사 및 구조화(Phase 1~4) 단계에서 나타나므로, 작성 전이 — Phase 5.2가 플랜을 디스크에 기록하기 전 — 가장 비용이 적게 드는 마지막 포착 시점입니다.

### 프롬프트 템플릿 (brainstorm-sourced)

**산문 요약 규율 (모든 티어 필수):** **구현이 작업을 어떻게 접근하는지** 설명하는 1~3줄의 평이한 산문 요약으로 시작하십시오 — 수정되는 파일/모듈, 확장 vs 도입 패턴, 준수되는 범위 경계 등. 미래 지향적(플랜에 무엇이 담길 것인가)이어야 합니다. 브레인스토밍에서 검증된 'WHAT'은 당연한 것으로 간주하며, 산문은 'HOW'를 요약합니다. Lightweight 티어조차 게슈탈트의 이점이 있습니다. 유일하게 정당한 생략 사례는 정말 사소한 경우(예: 합성이 프롬프트를 반영하는 한두 개의 Stated 불렛인 한 줄짜리 리팩토링 등)뿐입니다.

**군더더기 제거 가이드:** 군더더기 없는 평이한 언어로 실제 구현 형태를 먼저 제시하십시오. 수식어를 쓰지 말고, 브레인스토밍의 'WHAT'을 재진술하지 마십시오. 산문 요약이 브레인스토밍의 Problem Frame을 재진술하고 있다면, 플랜 단계의 결정 사항에 집중하도록 다시 작성하십시오.

```
상류의 브레인스토밍과 Phase 1 조사를 바탕으로, 플랜에 반영할 구현 범위는 다음과 같습니다:

[1~3줄 산문 요약 — 구현이 작업을 어떻게 접근하는지(파일/모듈, 패턴, 준수되는 범위) 평이한 언어로 기술. 브레인스토밍에서 검증된 WHAT은 당연한 것으로 간주하며 HOW를 요약합니다. 모든 티어 필수; 합성이 프롬프트를 반영하는 ≤ 2개의 불렛인 정말 사소한 경우에만 생략.]

**Stated (명시됨)** (브레인스토밍 및 조사 결과):
- [브레인스토밍에서 암시적으로 명명되었거나 저장소 조사로 표면화된 파일/모듈]
- [확장 또는 재사용을 위해 식별된 패턴]

**Inferred (추론됨)** (브레인스토밍이 해결하지 못한 공백을 메우는 플랜 단계의 결정):
- [패턴 확장 vs 신규 추상화 선택]
- [테스트 범위 추가 사항]
- [교차적 영향 평가]

**Out of scope (범위 외)** (의도적으로 제외됨):
- [후속 작업으로 미뤄지는 지엽적인 리팩토링]
- [테스트 범위에서 의도적으로 제외된 인접한 미테스트 코드]

구현 방법(HOW)에 대한 이 계획이 의도하신 바와 일치하나요? 추가, 제거 또는 방향 수정이 필요한 부분이 있다면 알려주세요. 확인해 주시면 진행하겠습니다.
```

사용자 응답에는 산문을 사용하십시오. 근거는 Interaction Rule 5(a)입니다.

## 순환 시 소프트 컷 (Soft-cut on circularity - 공통)

라운드별로 사용자가 어떤 Stated/Inferred/Out 항목을 건드렸는지 추적하십시오. 소프트 컷 차단 질문은 **동일한 항목이 두 번 수정되었을 때** (또는 3라운드 수정이 이미 2라운드에서 수정된 항목을 타겟팅할 때)에만 실행됩니다. 라운드 간의 새로운 항목 수정은 제한 없이 진행됩니다.

소프트 컷이 실행되면 플랫폼의 차단형 질문 도구를 사용하여 두 가지 옵션을 제시하십시오:

- `현재 수정된 합성으로 진행`
- `중단 및 방향 수정 — [조사 / 플랜 작성] 전에 추가 논의`

차단형 도구가 없는 경우에만 채팅의 번호 매겨진 목록으로 대체하십시오. 절대 말없이 건너뛰지 마십시오.

## 비대화형 모드 (공통)

스킬이 LFG와 같은 자동화된 워크플로우 또는 모든 `disable-model-invocation` 컨텍스트에서 호출된 경우, 스킬은 비대화형 모드(동기적 사용자 없음)로 실행됩니다. 산출물은 하류 스킬(ce-doc-review, ce-work) 및 인간 리뷰어(PR 리뷰)에 의해 읽힙니다.

**변형별 동작** (시점에 따라 후속 단계가 달라집니다):

- **단독 변형 (Phase 0.7)**: Phase 0.7은 조사 *전*에 실행됩니다. 비대화형 모드에서는 합성을 구성하되 사용자 확정 단계를 건너뜁니다. 정상적으로 Phase 1 조사를 계속합니다. Inferred 내용은 플랜 작성(Phase 5.2)까지 유지되었다가 `## Assumptions`로 라우팅됩니다.
- **브레인스토밍 기반 변형 (Phase 5.1.5)**: Phase 5.1.5는 조사 *후*, 플랜 작성 전에 실행됩니다. 합성을 구성하되 사용자 확정 단계를 건너뜁니다. Phase 5.2 플랜 작성으로 진행합니다. Inferred 내용은 `## Assumptions`로 라우팅됩니다.

**두 변형의 공통 동작:**

- **사용자 프롬프트 및 차단형 질문 없음.** 확정 단계를 건너뜁니다.
- **모드 인식 형태의 내용 라우팅:**
  - **Stated** 내용 → Requirements (사용자가 언급한 제약 사항, 기원의 R-ID가 있다면 추적됨)
  - **Out-of-scope** 내용 → Scope Boundaries
  - **Inferred** 내용 → 플랜의 `## Assumptions` 섹션 — 검증되지 않은 에이전트의 베팅임을 명시적으로 라벨링합니다. Inferred 항목을 Key Technical Decisions나 Implementation Units로 라우팅하지 마십시오. 그렇게 하면 검증되지 않은 베팅을 사용자 확정 결정과 구분할 수 없게 됩니다.

`## Assumptions` 섹션은 비대화형 플랜에만 나타납니다. 대화형 플랜에서는 필요하지 않습니다(Inferred 베팅은 채팅에서 사용자에 의해 수정되어 Key Technical Decisions가 되거나 수정되어 사라지기 때문입니다).

이는 원래 설계가 의도했던 감사 가시성(검증되지 않은 베팅은 권위 있는 내용으로 전파되어서는 안 됨)을 복원하면서도, 이를 숨기지 않고 별도의 라벨 아래 표면화합니다. 하류 리뷰(ce-doc-review, ce-work, 인간 PR 리뷰)는 Assumptions를 구체적으로 정밀 조사할 수 있습니다.

## 자체 리다이렉트 (공통)

사용자 응답이 잘못된 스킬에 있거나 다른 워크플로우를 원한다고 표시하는 경우:

- **단독 변형**: 흔한 리다이렉트 신호는 "생각보다 큰 작업이네요 — 브레인스토밍부터 할게요" (`/ce-brainstorm` 제안), "그냥 수정만 하면 돼요, 플랜은 필요 없어요" (`/ce-work` 제안), 또는 "먼저 조사가 필요해요" (`/ce-debug` 제안) 등이 있습니다.
- **브레인스토밍 기반 변형**: 흔하지는 않지만 가능합니다 — "사실 이 범위는 틀렸어요, 브레인스토밍으로 돌아가죠" (상류 문서를 수정하기 위해 `/ce-brainstorm` 제안).

어떤 경우든: ce-plan을 중단하고 대안 스킬을 제안하며, 세션 내에서 해당 스킬을 로드하겠다고 제안하십시오. 반박하거나 논쟁하지 마십시오 — 사용자의 리다이렉트 신호는 의도적인 선택입니다.

## 확정 후 문서 형태

사용자 확정 후(또는 소프트 컷 결정이 진행된 후), Phase 5.2가 플랜 문서를 작성합니다. 3개 버킷 구조는 플랜의 `## Synthesis` 섹션으로 전달되지 않습니다. 산문 요약만이 플랜 템플릿의 기존 `## Overview` 슬롯(용어 일관성을 위해 `## Summary`로 이름 변경됨)을 대체하여 임베드됩니다. 버킷 내용은 플랜의 본문 섹션으로 분산됩니다:

| 채팅 시 요소 | 플랜 내 위치 |
|---|---|
| 산문 요약 | `## Summary` (1~3줄, 미래 지향적) — 단독 변형: 타겟팅되는 범위; 브레인스토밍 기반: 구현 접근 방식 |
| Stated 불렛 | `## Requirements` (R-ID) 및 맥락을 위한 `## Problem Frame` |
| Inferred 불렛 | `## Key Technical Decisions` (근거 포함) 및 베팅이 구조적 선택을 주도하는 경우 Implementation Units. 비대화형 모드에서는 `## Assumptions`로 라우팅 — 아래 비대화형 모드 참조. |
| Out-of-scope 불렛 | `## Scope Boundaries` — 관련 있는 경우 `### Deferred to Follow-Up Work` 서브 섹션 포함 |

이탤릭체 캡처 컨텍스트 노트(예: "Phase 0.7에서 캡처됨...")는 작성하지 않습니다. 이는 해당 정보를 필요로 하지 않는 독자에게 엔지니어링 프로세스를 유출하는 것입니다.

플랜의 `## Summary`와 `## Problem Frame`은 서로 다른 목적을 가져야 합니다: Summary는 "이 플랜은 무엇을 제안하는가?"에 답합니다 (미래 지향적, 1~3줄). Problem Frame은 "왜 이 제안이 존재하는가?"에 답합니다 (과거 회고적, 문단 형태). Problem Frame에서 제안 내용을 재진술하지 마십시오. Summary를 상황적 컨텍스트로 채우지 마십시오.

## 합성에 속하지 않는 것

- 구현 코드 (import, 정확한 메서드 시그니처, 프레임워크 특화 문법, JSON 형태, 오류 메시지의 정확한 문구 등)
- 브레인스토밍 문서 전체의 재진술 — 합성은 플랜 관점의 요약이지 복사본이 아닙니다.
- 방어적인 가정이나 회피성 표현 — 우려 사항이 실질적이라면 Inferred나 Out으로 명시하고, 단순 추측이라면 삭제하십시오.
- 3개 버킷 외부에서 표면화된 미결 질문 — 합성 시점까지 모든 범위 형성 질문은 **Stated**(이미 묻고 답함), **Inferred**(수정을 위한 에이전트의 베팅) 또는 **Out**(의도적으로 제외됨) 중 하나여야 합니다. 네 번째 상태는 없습니다. 질문이 정말로 기본값으로 결정될 수 없다면 합성을 일시 중지하고 제시 전에 해결하십시오 — 옵션이 제한적이고 의미 있게 구분되는 경우 차단형 객관식 도구를 사용하고, Interaction Rule 5(a)에 따라 옵션 세트가 답변에 편향을 줄 수 있는 경우 산문을 사용하십시오. 답변을 통합한 후 합성을 제시하십시오. 합성 옆에 부유하는 질문들을 절대 함께 제시하지 마십시오 — 이는 사용자에게 명확한 해결 경로를 제공하지 못합니다.
