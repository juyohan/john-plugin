# 공통 패턴

## 스켈레톤(Skeleton) 프로젝트

새로운 기능을 구현할 때:
1. 검증된 스켈레톤 프로젝트를 검색하십시오.
2. 여러 에이전트를 병렬로 실행하여 옵션들을 평가하십시오:
   - 보안 평가
   - 확장성 분석
   - 관련성 점수 산정
   - 구현 계획 수립
3. 가장 적합한 프로젝트를 기반으로 클론하십시오.
4. 검증된 구조 내에서 반복 개발을 진행하십시오.

## 디자인 패턴

### 리포지토리 패턴 (Repository Pattern)

데이터 액세스 로직을 일관된 인터페이스 뒤로 캡슐화합니다:
- 표준 연산 정의: findAll, findById, create, update, delete
- 구체적인 구현체에서 저장소 상세 사항(데이터베이스, API, 파일 등)을 처리합니다.
- 비즈니스 로직은 저장 메커니즘이 아닌 추상 인터페이스에 의존합니다.
- 데이터 소스 교체가 용이하며, 모킹(mock)을 통한 테스트를 단순화합니다.

### API 응답 형식

모든 API 응답에 일관된 엔벨로프(Envelope)를 사용합니다:
- 성공/상태 표시자 포함
- 데이터 페이로드(payload) 포함 (에러 발생 시 null 가능)
- 에러 메시지 필드 포함 (성공 시 null 가능)
- 페이지네이션 응답을 위한 메타데이터 포함 (total, page, limit)
