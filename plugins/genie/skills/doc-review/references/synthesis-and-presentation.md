# Phase 3-5: Synthesis, Presentation, and Next Action (합성, 프레젠테이션 및 다음 단계)

## Phase 3: Synthesize Findings (발견 사항 합성)

모든 에이전트로부터 수집된 발견 사항을 이 파이프라인을 통해 처리합니다. 순서가 중요합니다 — 각 단계는 이전 단계에 의존합니다. 파이프라인은 발견 사항 수명 주기 상태 머신을 구현합니다: **Raised → (Confidence Gate | FYI-eligible | Dropped) → Deduplicated → Classified → SafeAuto | GatedAuto | Manual | FYI**. 각 단계의 경계에서 상태를 재평가하십시오. 이전 단계의 가정을 산문 수준의 지름길로 끌어오지 마십시오.

### 3.1 Validate (검증)

각 에이전트가 반환한 JSON을 발견 사항 스키마에 맞게 확인합니다:

- 스키마에 정의된 필수 필드가 누락된 발견 사항은 제거합니다.
- 잘못된 enum 값을 가진 발견 사항은 제거합니다 (이전 페르소나의 `auto` / `present` 값 포함 — 모든 페르소나 출력이 재생성될 때까지 이들을 잘못된 형식으로 취급하십시오).
- 잘못된 형식의 출력에 대해서는 Coverage 섹션에 에이전트 이름을 기록합니다.

**remap / 검증 진단 내용을 사용자에게 설명하지 마십시오.** 스키마 드리프트 노트("persona X returned unknown enum Y, remapped to Z"), 페르소나 프롬프트 드리프트 코멘터리 및 기타 유효성 검사기 내부 진단은 유지 관리자용 정보입니다. 사용자가 읽는 Phase 4 출력에 포함되어서는 안 됩니다. 페르소나의 출력이 잘못된 경우 사용자에게 보이는 유일한 결과는 Coverage 행의 주석(예: 페르소나의 발견 사항 수가 적게 표시되거나 `malformed` 마커가 표시됨)뿐입니다. 그 외의 모든 것은 내부적으로 유지됩니다.

### 3.2 Confidence Gate (앵커 기반 신뢰도 게이트)

`confidence` 앵커 값을 기준으로 발견 사항을 필터링합니다. 앵커는 `references/findings-schema.json`에 정의되고 페르소나 루브릭(`references/subagent-template.md`)에 포함된 행동 정의를 가진 불연속적인 정수(`0`, `25`, `50`, `75`, `100`)입니다. 이는 이전의 연속적인 0.0-1.0 스케일을 심각도별 게이트로 대체합니다 — 문서 검토 경제성 측면에서 심각도별 임계값 계조는 불필요하며, 거친 앵커는 거짓 정밀도(false-precision) 조작을 방지합니다.

| Anchor | Meaning | Route |
|--------|---------|-------|
| `0`    | 오탐(False positive) 또는 기존 이슈 | 무음 누락(Drop silently) |
| `25`   | 실제일 수 있으나 확인 불가 | 무음 누락(Drop silently) |
| `50`   | 실제임을 확인했으나 사소함 / 자문 / 중요하지 않음 | FYI 하위 섹션에 표시 |
| `75`   | 재차 확인됨, 실제 환경에서 발생할 것임, 정확성에 직접 영향 | 실행 가능한 티어로 진입 (`autofix_class`로 분류) |
| `100`  | 증거가 직접 확인해 줌, 빈번하게 발생할 것임 | 실행 가능한 티어로 진입 (`autofix_class`로 분류) |

- **무음 누락 (Dropped silently)** (앵커 `0` 및 `25`): 이들은 발견 사항, FYI 관찰, 잔류 우려 사항 등 어떤 출력 버킷에도 나타나지 않습니다. 0이 아닐 때 총 누락 수를 Coverage 각주 라인으로 기록합니다: `Dropped: N (anchors 0/25 suppressed)`. 각주는 Coverage 테이블 아래에 `Chains:` 각주와 함께 표시됩니다. 이것이 누락 수 보고를 위한 표준 위치입니다 — 요약 라인이나 페르소나별 Coverage 열이 아닙니다. N이 0이면 각주를 생략합니다.
- **FYI 하위 섹션 (FYI-subsection)** (앵커 `50`): `autofix_class`에 관계없이 프젠테이션 레이어의 FYI 하위 섹션에 표시합니다. 이들은 워크스루나 일괄 작업에 진입하지 않습니다 — 결정을 강제하지 않는 관찰 가치만 가집니다. 자문 관찰("망가지는 것은 없지만...")이 자연스럽게 여기에 해당합니다.
- **실행 가능 (Actionable)** (앵커 `75` 및 `100`): 분류 파이프라인으로 진입합니다. `autofix_class`에 따라 라우팅됩니다 (3.7 참조).

**왜 Anthropic의 ≥ 80 코드 검토 임계값이 아닌 이 임계값인가:** 문서 검토는 코드 검토와 반대의 경제성을 가집니다. 린터(linter)라는 안전장치가 없으며, 검토 자체가 안전장치입니다. 전제 수준의 우려 사항(product-lens, adversarial)은 "동기가 유효한가?"를 실제 사실과 대조하여 확인할 수 없으므로 자연스럽게 앵커 50-75에서 제한됩니다. 라우팅 메뉴가 이미 기각(Skip, Append to Open Questions)을 저비용으로 만들어 주므로, 표시되고 스킵되는 것은 저비용 결과인 반면, 놓치고 배포되는 것은 다운스트림 구현을 망칩니다. 낮게 필터링(`≥ 50`)하고 볼륨은 라우팅 메뉴가 처리하게 하십시오.

### 3.3 Deduplicate (중복 제거)

`normalize(section) + normalize(title)`을 사용하여 각 발견 사항의 지문(fingerprint)을 만듭니다. 정규화: 소문자 변환, 문장 부호 제거, 공백 축소.

페르소나 간에 지문이 일치하는 경우:

- 발견 사항이 상충하는 작업을 권장하는 경우(예: 하나는 삭제, 다른 하나는 유지), 병합하지 마십시오 — 3.5의 모순 해결을 위해 둘 다 유지합니다.
- 그렇지 않으면 병합합니다: 가장 높은 심각도 유지, 가장 높은 신뢰도 앵커 유지(동점일 경우 문서 순서상 먼저 나타나는 발견 사항 유지 — 확률적이 아닌 결정론적 방식), 모든 증거 배열 합치기, 모든 합의된 검토자 기록 (예: "coherence, feasibility")
- **Coverage 할당:** 병합된 발견 사항을 신뢰도 앵커가 가장 높은 페르소나의 몫으로 할당합니다. 앵커가 동점이면 문서 순서상 먼저 나타나는 페르소나에게 할당합니다. 전체 합계가 정확하도록 패배한 페르소나의 Findings 카운트와 해당 라우팅 버킷을 차감합니다.

### 3.3b Same-Persona Premise Redundancy Collapse (동일 페르소나 전제 중복 축소)

단일 페르소나가 서로 다른 섹션에 표현되거나 서로 다른 프레이밍으로 감싸진 동일한 근본 전제를 가진 여러 발견 사항을 제출하는 경우가 있습니다 (예: product-lens가 Motivation, Unit 4b, Key Technical Decisions 및 다른 두 섹션에 대해 "동기가 약함"의 5가지 변형을 제출함). 크로스 페르소나 중복 제거(3.3)는 근본적인 우려 사항이 같더라도 section+title이 다르면 이를 잡아내지 못합니다. N개의 변형을 모두 표시하는 것은 다른 5개 페르소나에 비해 해당 페르소나의 관점을 과도하게 강조하고 P2 Decisions 티어를 거의 중복된 신호로 부풀리게 됩니다.

각 페르소나에 대해 해당 페르소나의 생존한 발견 사항들을 공유된 근본 전제별로 클러스터링합니다. 동일한 페르소나의 3개 이상의 발견 사항이 다음을 공유할 때 클러스터가 형성됩니다:

- 동일한 `finding_type` (error 또는 omission)
- 실질적으로 겹치는 `why_it_matters` 문구 (동일한 우려 사항을 나타내는 핵심 명사/동사, 예: "motivation", "justification", "premise unsupported", "scope creep")
- 동일한 상위 결정에 의해 모두 불필요해질 수정 사항 (예: "트리거링 사건 추가"가 5개의 동기 약화 발견 사항을 모두 무의미하게 만듦)

크기 N ≥ 3인 각 클러스터에 대해:

- 가장 강력한 증거를 가진 단일 발견 사항만 유지합니다 (가장 높은 신뢰도 앵커, 동점일 경우 가장 구체적인 문서 참조를 인용한 것).
- 나머지 N-1개 발견 사항은 원래 앵커에 관계없이 FYI 하위 섹션 상태(앵커 `50`)로 강등합니다.
- 유지된 발견 사항의 Reviewer 열에 해당 페르소나가 N-1개의 관련 변형을 제기했음을 기록합니다 (예: `product-lens (+4 related variants demoted to FYI)`).

이는 3.4 크로스 페르소나 격상 전에 페르소나별로 실행됩니다. *유지된* 발견 사항에 대한 페르소나 간 합의는 여전히 3.4의 앵커 단계 격상 자격을 갖습니다. 강등된 변형은 페르소나 간 격상에 참여하지 않습니다 (축소 후에는 관찰용으로만 존재함).

이 단계에서 페르소나 간 축소를 수행하지 마십시오 — 서로 다른 페르소나가 동일한 우려 사항을 제기하는 것은 크로스 페르소나 격상이 보상하는 바로 그 독립적 신호입니다. 축소는 오직 한 페르소나의 출력 내에서만 적용됩니다.

### 3.4 Cross-Persona Agreement Promotion (페르소나 간 합의 격상)

2개 이상의 독립적인 페르소나가 동일한 병합된 발견 사항(3.3에서 생성)을 지적한 경우, 병합된 발견 사항의 앵커를 한 단계 격상합니다: `50 → 75`, `75 → 100`. 앵커 `100`은 더 이상 격상되지 않습니다(이미 최고치임). 앵커 `0` 또는 `25`인 발견 사항은 이 단계에 도달하지 않습니다(3.2에서 누락됨).

독립적인 입증은 강력한 신호입니다 — 여러 검토자가 동일한 이슈로 수렴하는 것은 단일 검토자의 앵커보다 신뢰할 수 있습니다. 한 단계의 앵커 격상은 의미론적으로 중요합니다 (두 페르소나가 독립적으로 제기한 "확인되었으나 사소함" 발견 사항은 "실제 환경에서 발생할 것임"으로 볼 수 있음). 이는 이전의 `+0.10` 부스트를 대체합니다 — 마법의 숫자 범프는 연속 스케일에 맞춰져 있었으며 더 이상 적용되지 않습니다.

출력물의 Reviewer 열에 격상 내용을 기록하십시오 (예: `coherence, feasibility (+1 anchor)`).

이는 이전의 잔류 우려 사항 격상 단계를 대체합니다. 앵커 `0` / `25`인 발견 사항은 검토 화면으로 다시 격상되지 않습니다. 이들은 Coverage에서 누락 수로만 나타납니다. 누락된 발견 사항이 진정으로 중요하다면 검토자는 격상 규칙에 의존하기보다 더 강력한 증거를 통해 자신의 앵커를 `50` 이상으로 높여야 합니다.

### 3.5 Resolve Contradictions (모순 해결)

페르소나들이 동일한 섹션에 대해 의견이 다른 경우:

- 두 관점을 모두 제시하는 결합된 발견 사항을 만듭니다.
- `autofix_class: manual`로 설정합니다 (모순은 정의상 판단의 영역입니다).
- `finding_type: error`로 설정합니다 (모순은 문서가 누락한 것이 아니라 충돌하게 말하는 것에 관한 것입니다).
- 판결이 아닌 트레이드오프로 프레임합니다.

구체적인 충돌 패턴:

- Coherence는 "일관성을 위해 유지" + scope-guardian은 "단순함을 위해 삭제" → 결합된 발견 사항 생성, 사용자가 결정하게 함
- Feasibility는 "이것은 불가능함" + product-lens는 "이것은 필수적임" → 트레이드오프로 프레임된 P1 발견 사항
- 여러 페르소나가 동일한 이슈를 지적함 (의견 차이 없음) → 3.3 병합에서 처리되며 여기서 다루지 않음

### 3.5b Deterministic Recommended-Action Tie-Break (결정론적 권장 작업 타이브레이크)

모든 병합된 발견 사항은 워크스루(`references/walkthrough.md`)에서 `(recommended)` 옵션을 표시하고, 최선의 판단 경로(`references/bulk-preview.md`)에서 일괄 실행할 대상을 선택하며, 줄기의 예/아니오 프레이밍에 사용되는 정확히 하나의 `recommended_action` 필드를 가집니다. 병합된 발견 사항이 서로 다른 작업을 암시하는 여러 페르소나에 의해 지적된 경우, 동일한 검토 아티팩트가 실행 전체에서 동일한 워크스루 및 최선의 판단 동작을 생성하도록 합성이 `recommended_action`을 결정론적으로 선택합니다.

**타이브레이크 순서 (가장 보수적인 것 우선):** `Skip > Defer > Apply`. 최소 한 명의 기여 페르소나가 암시한 작업 중 이 순서로 스캔하여 처음 나타나는 작업이 승리합니다.

- 기여 페르소나 중 하나라도 Skip을 암시한 경우 → `recommended_action: Skip`
- 그렇지 않고 기여 페르소나 중 하나라도 Defer를 암시한 경우 → `recommended_action: Defer`
- 그렇지 않으면 → `recommended_action: Apply`

**페르소나-작업 매핑.** 페르소나는 분류를 통해 작업을 암시합니다:

- `safe_auto` 또는 `gated_auto` → Apply 암시
- 구체적인 `suggested_fix`와 권장 해결책을 가진 `manual` → Apply 암시 (페르소나가 무엇을 해야 할지에 대한 의견을 가지고 있음)
- 권장 해결책 없이 트레이드오프나 범위 질문으로 표시된 `manual` → Defer 암시 (재검토할 가치가 있으나 지금 조치할 가치는 없음)
- 잔류 우려 사항을 통해 발견 사항을 낮은 신뢰도 또는 억제 대상으로 표시하는 모든 페르소나 → Skip 암시
- 모순 세트(3.5)에서 "있는 그대로 유지 / 변경하지 않음"을 암시하는 페르소나 → Skip 암시

기여 페르소나들이 작업에 대해 모두 침묵하는 경우(예: 권장 사항 없이 관찰로만 지적한 페르소나들의 병합된 `manual` 발견 사항), 병합된 발견 사항이 실행 가능한 `suggested_fix`를 가지고 있는지 여부에 따라 기본값을 선택합니다:

- `suggested_fix` 존재 → 실용적인 기본값으로 `recommended_action: Apply`.
- `suggested_fix` 부재 → `recommended_action: Defer` (워크스루와 최선의 판단 경로는 수정안 없이 Apply를 실행할 수 없습니다. 작업이 없는 발견 사항을 Defer로 라우팅하면 사용자가 무엇을 할지 결정할 수 있는 Open Questions에 표시됩니다).

이 게이트는 타이브레이크의 모든 분기에 적용됩니다: 승리한 작업이 `Apply`이지만 3.6(Promote) 및 3.7(Route) 실행 후 병합된 발견 사항에 `suggested_fix`가 없는 경우 `Defer`로 다운그레이드합니다. 워크스루는 여전히 사용자가 4가지 옵션 중 하나를 선택할 수 있게 합니다. 이 규칙은 오직 에이전트의 기본 권장 사항만 관장하여 최선의 판단 경로와 일괄 미리보기가 실행 불가능한 Apply를 예약하지 않도록 합니다.

**충돌 컨텍스트 표시.** 타이브레이크가 실행될 때(기여 페르소나들이 서로 다른 작업을 암시함), 병합된 발견 사항에 한 줄의 충돌 컨텍스트 문자열을 기록합니다. 워크스루는 이를 R15 충돌 컨텍스트 라인에 렌더링합니다( `references/walkthrough.md` 참조). 예: `Coherence recommends Apply; scope-guardian recommends Skip. Agent's recommendation: Skip.`

**다운스트림 불변성.** 워크스루와 일괄 미리보기는 권장 사항을 다시 계산하지 않습니다 — 그들은 `recommended_action`을 읽고 일치하는 옵션에 `(recommended)`를 렌더링합니다. Best-judgment-the-rest와 라우팅 옵션 B는 범위 내 발견 사항 세트에 대해 `recommended_action`을 일괄 실행합니다. 이는 최선의 판단 결과를 재현 가능하고 감사 가능하게 유지합니다: 동일한 검토 아티팩트는 항상 동일한 일괄 계획을 생성합니다.

### 3.5c Premise-Dependency Chain Linking (전제 조건 의존성 체인 연결)

문서 검토는 종종 팬아웃(fanout)을 생성합니다: 단일 전제 도전("이 작업이 정당화되는가?")이 전제가 거부될 경우 모두 사라지는 다운스트림 발견 사항들("에일리어스 정당성 부족", "추상화 과잉", "마이그레이션 롤백 부족", "명명법이 미래를 차단함")을 생성합니다. 각각을 독립적인 결정으로 표시하면 사용자는 동일한 근본 질문을 N번 다시 심리해야 합니다. 이 단계는 종속된 발견 사항을 루트(root)에 연결하여 프레젠테이션에서 그룹화하고 워크스루가 체인 전체에 단일 루트 결정을 전파할 수 있도록 합니다.

병합된 발견 사항 세트에 대해 3.5b(`recommended_action` 정규화) 이후, 3.6(자동 격상) 이전에 이 단계를 실행합니다.

**1단계: 루트 식별.** 다음 사항이 모두 충족될 때 발견 사항은 루트 후보가 됩니다:

- 심각도가 `P0` 또는 `P1`임 (전제 수준의 이슈는 본질적으로 높은 우선순위를 가짐)
- `autofix_class`가 `manual`임 (루트 자체는 판단을 필요로 함 — safe/gated 루트는 조치 대상이지 전파 대상이 아님)
- `why_it_matters` 또는 `title`이 세부 사항이 아닌 기초적인 전제에 도전함. 신호 문구 (어휘가 아닌 형태): "premise unsupported", "justification missing", "do-nothing baseline not evaluated", "is X justified", "unsupported by evidence", "is the proposed solution the right approach"
- 발견 사항의 `section`이 프레이밍 수준(Problem Frame, Summary, Overview, Why, Motivation, Goals — `Summary`는 새 genie:plan / genie:brainstorm 템플릿 헤더이며, `Overview`는 레거시로 유지됨)이거나, 발견 사항이 명명된 컴포넌트의 존재 여부를 명시적으로 질문함

여러 후보가 기준에 부합하면 모두 승격시키십시오. 위의 기준(P0/P1, manual, 프레이밍 수준 섹션, 전제 도전 신호 문구)은 충분히 제한적이어서 잘 작성된 문서에 대해 이 리스트는 짧을 것입니다. 수치적 캡을 더 씌우지 마십시오. 두 개의 유효한 루트가 존재할 때 하나만 루트로 선택하면 두 번째 루트의 자연스러운 종속 항목들이 독립적인 manual 발견 사항으로 남게 됩니다 — 이는 체인이 해결하고자 하는 바로 그 UX 문제입니다.

**동료(Peer) vs 중첩(Nested) 테스트.** 루트 A의 제안된 수정안을 수락해도 루트 B의 우려 사항이 해결되지 않는 경우(반대도 마찬가지), 두 후보 루트는 동료입니다. 한 루트의 수정안이 다른 루트를 무의미하게 만드는 경우 그들은 중첩된 것이며, 이 경우 흡수된 후보는 생존한 루트의 종속 항목이 되고 동료 루트가 되지 않습니다. 대칭적으로 테스트하십시오: 결정하기 전에 양방향을 모두 확인하십시오.

**비대칭 흡수 시 생존 루트 선택.** 중첩된 경우, 생존 루트는 신뢰도가 높은 쪽이 아니라 자신의 수정안이 다른 쪽을 무의미하게 만드는 쪽입니다. 루트 A의 수정안을 수락하는 것이 루트 B의 우려 사항을 무의미하게 만들지만, 루트 B의 수정안을 수락하는 것이 루트 A의 우려 사항을 그대로 남겨둔다면, 어느 후보가 신뢰도가 더 높든 관계없이 A가 생존 루트가 되고 B는 그 종속 항목이 됩니다. 흡수 방향이 범위(더 넓은 전제가 승리함)를 결정하고, 신뢰도는 범위가 아닌 강도를 결정합니다. 신뢰도는 중첩 여부를 결정하는 것이 아니라 *동료 간*의 타이브레이크에 사용됩니다.

**합리성 진단.** 3개 이상의 후보가 일치하는 경우 기준이 올바르게 적용되고 있는지 재고하십시오 — 단일 문서가 3개 이상의 진정으로 별개인 전제 수준 도전을 포함하는 것은 드문 일입니다. 후보를 무음으로 누락하지 마십시오. 각 후보가 독립적으로 기준을 충족함을 확인(하고 모두 표시)하거나 기준 적용을 엄격히 하십시오. 카운트가 정당하게 높다면 모든 항목을 표시하는 것이 일부를 숨기는 것보다 유용합니다.

일치하는 항목이 없으면 이 단계의 나머지를 건너뜁니다 — 체인이 존재하지 않습니다.

**여러 루트 하에서의 종속 항목 할당.** 여러 루트가 존재하고 후보 종속 항목이 둘 이상의 루트에 연결될 수 있는 경우, 해당 루트의 거부가 종속 항목의 우려 사항을 가장 직접적으로 해소하는 루트에 할당합니다. 모호함이 남으면 신뢰도 앵커가 더 높은 루트에 할당합니다. 앵커가 동점이면 문서 순서상 먼저 나타나는 루트에 할당합니다. 종속 항목은 절대 둘 이상의 루트에 연결되지 않습니다 — 단일 `depends_on` 값.

**2단계: 종속 항목 식별.** 각 후보 루트에 대해 나머지 발견 사항에서 종속 항목을 스캔합니다. 조건은 `references/walkthrough.md`의 전파 트리거와 일치해야 합니다 — 사용자가 루트를 거부(Skip/Defer)할 때 종속 항목이 전파되므로, 의존성은 수락 분기가 아닌 거부 분기에서 정의됩니다. 다음 경우에 발견 사항은 루트의 종속 항목이 됩니다:

- 루트가 명명된 컴포넌트에 대한 기초 전제 — 존재 여부, 제안된 접근 방식의 정학성 또는 작업의 정당성 — 에 도전함. 인식할 형태 (어휘 리스트가 아님 — 문서의 도메인이 실제로 사용하는 것에 매핑): 필요성이 도전받는 호환성 레이어, 정당성이 의심받는 계획된 기능, 근거가 질문받는 추상화, 범위가 분쟁 중인 제안된 변경 사항, 선택이 경합 중인 마이그레이션 대상, 기반이 지원되지 않는 아키텍처 확약 등
- 후보의 `suggested_fix`가 동일한 컴포넌트를 수정, 세부 정보 추가 또는 제약함
- 루트의 전제가 거부될 경우 후보의 우려 사항이 해소됨 — 의미: 사용자가 루트를 거부(Skip/Defer)하면 종속 항목이 대상으로 삼는 컴포넌트가 더 이상 계획의 확정된 부분이 아니므로 종속 항목의 수정안은 작용할 안정적인 대상을 잃게 되고 루트와 함께 일괄 거부됩니다.

치환 체크로 테스트하십시오: "사용자가 루트를 거부(Skip/Defer)해도 종속 항목의 발견 사항이 여전히 사용자가 이번 라운드에 관여하고 싶어 할 실행 가능한 우려 사항을 설명하는가?" 아니라면 — 종속 항목의 전제가 루트와 함께 사라진다면 — 그것은 종속 항목입니다. 그렇다면(발견 사항이 루트 거부와 무관하게 생존하는 문제를 식별한다면) 종속 항목이 아닙니다.

**3단계: 독립성 안전장치.** 발견 사항의 대상 컴포넌트가 루트에 의해 다뤄지더라도 다음 경우에는 연결하지 마십시오:

- 종속 항목이 루트의 해결 여부와 관계없이 존재할 문제를 식별함. 마이그레이션의 롤백 계획, 모듈의 에러 처리, 기능의 테스트 커버리지 — 이들은 컴포넌트가 존재하기만 한다면 수행해야 할 운영상 의무입니다. 그들은 컴포넌트가 어떻게 행동해야 하는지를 설명합니다.
- 종속 항목의 `why_it_matters`가 전제 조건에 의존하지 않는 독립적인 증거(코드베이스 사실, 프레임워크 컨벤션, 프로덕션 데이터)를 인용함.
- 종속 항목이 `safe_auto`임 — 명확하고 올바른 하나의 수정안을 가지고 있으며 루트의 해결 여부와 관계없이 적용되어야 함.

불확실할 때는 연결하지 않는 것을 기본값으로 합니다. 잘못 연결된 체인은 실제 이슈를 숨기지만, 발견 사항을 연결하지 않고 남겨두는 것은 한 번의 추가 결정 비용만 발생시킵니다.

**4단계: 주석 달기.** 각 종속 항목에 `depends_on: <root_finding_id>`를 기록합니다(ID로 섹션 + 정규화된 제목 사용). 각 루트에 `dependents: [<dependent_ids>]`를 기록합니다. `dependents`는 루트당 6개 항목으로 제한합니다 — 6개 이상의 후보가 동일한 루트에 연결되면 심각도, 신뢰도 앵커(내림차순), 문서 순서 순으로 결정론적 타이브레이크를 거쳐 상위 6개만 유지하고 나머지는 연결하지 않은 상태로 둡니다(과도한 체이닝은 독립적인 우려 사항을 가릴 위험이 있음).

이 단계에서 발견 사항의 분류, 라우팅 또는 신뢰도 앵커를 변경하지 마십시오. 연결은 순수하게 주석 작업이며, 워크스루와 프레젠테이션에서 이 주석을 사용하지만 합성 자체는 사용하지 않습니다.

**5단계: Coverage 보고.** Coverage 요약에 라인을 추가합니다: `Chains: N root(s) with M total dependents`. N = 0이면 라인을 생략합니다.

**카운트 불변성 (중요 — 위반 금지).** Coverage 라인의 `M`은 Step 4 완료 후 `depends_on`이 설정된 발견 사항의 수입니다 — 즉, 단계 2(후보), 3(독립성 안전장치) 및 4(캡)를 거친 후의 최종 연결된 수입니다. 단계 2에서 고려된 후보 수가 아닙니다. 동일한 `dependents` 배열이 Coverage 카운팅과 `Dependents (...)` 하위 블록 렌더링 모두에 대한 정보의 근거(source of truth)입니다. 발견 사항이 루트의 `dependents` 배열에 나타나면, 프레젠테이션에서 해당 루트 아래에 중첩되어 나타나야 하며 자신의 심각도 위치에 나타나서는 안 됩니다. 발견 사항이 어떤 루트의 `dependents` 배열에도 나타나지 않으면, 자신의 심각도 위치에 나타나야 하며 어디에도 중첩되어 나타나서는 안 됩니다. 오케스트레이터가 두 개의 서로 다른 근거 값을 사용하는 경우에만 Coverage 카운트와 렌더링이 어긋나게 됩니다 — 정확히 하나, 루트별 Step-4 이후 `dependents` 배열만 존재합니다.

**작업 사례 A (rename 형태).** 리팩토링 계획 검토 결과 11개의 발견 사항이 발생했습니다. 하나는 Problem Frame 섹션의 P0 manual "사용자 대면 증거로 지원되지 않는 이름 변경 전제" — 루트 후보입니다. 나머지 10개를 스캔합니다:

- P1 manual "정당화되지 않은 에일리어스 메커니즘 범위" — 루트는 에일리어스 없는 순수 이름 변경으로 범위를 좁힐 것을 제안함. 종속 항목의 수정안은 에일리어스 인프라를 삭제할 것을 제안함. 연결됨.
- P2 manual "AliasedCommand 추상화 과잉" — 추상화는 에일리어스를 지원하기 위해 존재함. 에일리어스가 삭제되면 추상화도 사라짐. 연결됨.
- P2 manual "이름 변경이 이중 모드 미래를 차단함" — 우려 사항은 이름 변경이 진행될 때만 존재함. 연결됨.
- P2 manual "ID 드리프트: 명령어 vs 아티팩트 이름" — 명칭 비대칭은 이름 변경이 진행될 때만 존재함. 연결됨.
- P1 manual "마이그레이션에 롤백 전략 부족" — 마이그레이션은 범위와 무관하게 롤백이 필요함. 연결되지 않음 (독립성 안전장치).
- P0 gated_auto "마이그레이션과 코드 사이의 배포 순서" — 사용자가 무관하게 확인하는 구체적인 수정안. 연결되지 않음 (안전장치: 자체 해결 경로를 가진 gated_auto).

결과: 1개 루트 + 4개 종속 항목. 사용자는 루트를 먼저 보게 됩니다. 이를 거부하면 4개의 종속 항목이 자동으로 해결된 것으로 전파됩니다. 수동 관여 횟수가 11 → 7 (연결되지 않은 6개 + 눈에 보이는 1개 루트)로 줄어듭니다.

**작업 사례 B (auth 형태).** 새 세션 관리 미들웨어 도입 계획 검토. 하나는 Problem Frame 섹션의 P1 manual "미들웨어 재작성 전제 미지원 — 기존 세션 처리에는 보고된 신뢰성 이슈가 없음". 나머지 발견 사항 스캔:

- P2 manual "기존 요청 컨텍스트 대비 미들웨어 추상화 경계 불분명" — 경계는 미들웨어가 구축될 때만 중요함. 연결됨.
- P2 manual "새 세션 스토어에 대한 롤아웃 전략 미지정" — 롤아웃은 새 스토어가 배포될 때만 중요함. 연결됨.
- P1 gated_auto "세션 교체 시 CSRF 토큰 재생성 누락" — 미들웨어가 올바른 접근 방식인지와 무관하게 계획의 서면 설계에 있는 실제 보안 갭임. 연결되지 않음 (안전장치: gated_auto, 구체적인 수정안이 무관하게 적용됨).
- P2 manual "기존 세션 타임아웃 동작이 테스트에 캡처되지 않음" — 이는 기존의 테스트 커버리지 갭임. 재작성 여부와 무관하게 현재 코드에 존재함. 연결되지 않음 (독립성 안전장치).

결과: 1개 루트 + 2개 종속 항목. 용어와 도메인은 다르지만 사례 A와 동일한 형태이며, 이것이 인식해야 할 패턴입니다.

### 3.6 Promote Auto-Eligible Findings (자동 적용 가능 발견 사항 격상)

`manual` 발견 사항 중 `safe_auto` 또는 `gated_auto`로 격상할 수 있는 항목을 스캔합니다. 발견 사항이 통합된 자동 격상 패턴 중 하나와 일치하면 격상합니다:

- **코드베이스 패턴 해결 (Codebase-pattern-resolved).** `why_it_matters`가 구체적인 기존 코드베이스 패턴(단순히 "베스트 프랙티스"나 "컨벤션"이 아닌 구체적인 파일/함수/사용 참조)을 인용하고, `suggested_fix`가 그 패턴을 따르는 경우. `gated_auto`로 격상합니다 — 사용자가 여전히 확인하지만, 코드베이스 증거가 모호함을 해소합니다.
- **사실적으로 잘못된 동작 (Factually incorrect behavior).** 문서가 사실과 다른 동작을 설명하고 있으며, 올바른 동작이 컨텍스트나 코드베이스에서 유도될 수 있는 경우. `gated_auto`로 격상합니다.
- **표준 보안/신뢰성 제어 누락 (Missing standard security/reliability controls).** 누락 사항이 명백한 갭이며(설명된 시스템에 대한 합법적인 설계 선택이 아님), 수정안이 확립된 관행(HTTPS 강제, 체크섬 검증, 입력 새니타이징, 프라이빗 IP 거부, 이름 변경 시 사용 중단 경고와 함께 폴백)을 따르는 경우. `gated_auto`로 격상합니다.
- **프레임워크 네이티브 API 대체 (Framework-native-API substitutions).** 수동 구현이 일급 프레임워크 동작을 중복 수행하며 프레임워크 API가 인용된 경우. `gated_auto`로 격상합니다.
- **기계적으로 암시되는 완전성 추가 사항 (Mechanically-implied completeness additions).** 누락된 내용이 (상위 목표가 아닌) 문서 자체의 명시적이고 구체적인 결정에서 기계적으로 도출되는 경우. 진정으로 하나의 올바른 추가 사항이 있다면 `safe_auto`로, 추가 사항이 실질적이라면 `gated_auto`로 격상합니다.

발견 사항이 검토자가 알 수 없는 트레이드오프를 저자가 고려했을 법한 범위나 우선순위 변경과 관련된 경우에는 격상하지 마십시오.

**Strawman 다운그레이드 안전장치.** `safe_auto` 발견 사항이 `why_it_matters`에서 치부된 대안들을 명시한 경우(페르소나 템플릿의 strawman 규칙에 따라), 해당 대안들이 진정으로 허수아비인지 확인하십시오. 어떤 대안이라도 페르소나가 너무 공격적으로 기각한 합당한 설계 선택이라면, 수정안이 적용되기 전에 사용자가 트레이드오프를 볼 수 있도록 `gated_auto`로 다운그레이드하십시오.

### 3.7 Route by Autofix Class (Autofix Class에 의한 라우팅)

**심각도와 autofix_class는 독립적입니다.** 올바른 수정안이 명백하다면 P1 발견 사항도 `safe_auto`가 될 수 있습니다. 테스트 기준은 "얼마나 중요한가?"가 아니라 "명확하고 올바른 수정안이 하나뿐인가, 아니면 판단이 필요한가?"입니다.

**앵커와 autofix_class도 독립적입니다.** 앵커는 발견 사항을 어떤 화면(FYI vs 실행 가능)으로 보낼지 결정하고, `autofix_class`는 실행 가능한 화면에서 해당 발견 사항을 어떻게 처리할지 결정합니다. 이 단계에서는 두 가지를 모두 고려합니다.

3.7에 도달한 발견 사항은 이미 3.2에 의해 앵커 `50`, `75` 또는 `100`으로 필터링되었습니다 (앵커 `0` 및 `25`는 제거됨).

| Anchor | Autofix Class | Route |
|--------|---------------|-------|
| `100`  | `safe_auto`   | Phase 4에서 무음 적용. `suggested_fix`가 필수임. 누락 시 `gated_auto`로 강등. |
| `100`  | `gated_auto`  | Apply가 마킹된(`recommended`) 상태로 발견 사항별 워크스루 진입. `suggested_fix`가 필수임. 누락 시 `manual`로 강등. |
| `100`  | `manual`      | 사용자의 판단이 필요한 프레이밍으로 발견 사항별 워크스루 진입. `suggested_fix`는 선택 사항. |
| `75`   | `safe_auto`   | 라우팅 전 `gated_auto`로 강등 — 무음 적용은 증거가 수정안을 직접 확인하는 앵커 `100` 발견 사항을 위해 예약됨. Apply가 마킹된(`recommended`) 상태로 워크스루 진입. |
| `75`   | `gated_auto`  | Apply가 마킹된(`recommended`) 상태로 발견 사항별 워크스루 진입. `suggested_fix`가 필수임. 누락 시 `manual`로 강등. |
| `75`   | `manual`      | 사용자의 판단이 필요한 프레이밍으로 발견 사항별 워크스루 진입. `suggested_fix`는 선택 사항. |
| `50`   | any           | `autofix_class`에 관계없이 FYI 하위 섹션에 표시. 워크스루나 일괄 작업에 진입하지 않음. 이들은 결정이 아닌 관찰임. |

**safe_auto 가능 패턴:** 요약/세부 불일치(본문이 개요보다 우선함), 잘못된 카운트, 문서의 다른 곳에서 유도 가능한 누락된 리스트 항목, 오래된 내부 상호 참조, 용어 혼동, 다이어그램을 산문에 맞춰 기계적으로 업데이트할 수 있는 산문-다이어그램 불일치 (삭제는 절대 수정안이 아님 — 다이어그램은 공간적 이해를 돕는 의도적인 커뮤니케이션 선택이지 중복이 아님), 다른 내용에 의해 기계적으로 암시되는 누락된 단계, 주변 컨텍스트에 의해 암시되는 언급되지 않은 임계값 등.

**gated_auto 가능 패턴:** 코드베이스 패턴으로 해결된 수정 사항, 사실적으로 잘못된 동작, 누락된 표준 보안/신뢰성 제어, 프레임워크 네이티브 API 대체, 명시적 결정에 의해 기계적으로 암시되는 실질적인 완전성 추가 사항 등.

### 3.8 Sort (정렬)

출력을 위해 발견 사항을 정렬합니다: P0 → P1 → P2 → P3 순서로, 그 다음 발견 사항 유형(error가 omission보다 먼저), 그 다음 신뢰도 앵커(내림차순: `100`, `75`, `50`), 마지막으로 문서 순서(섹션 위치)를 결정론적 타이브레이크로 사용합니다.

### 3.9 Suppress Restatements in Residual Concerns and Deferred Questions (잔류 우려 사항 및 지연된 질문의 재진술 억제)

페르소나 출력에는 `findings`와 함께 `residual_risks` 및 `deferred_questions` 배열이 포함됩니다. 실행 가능 티어 세트가 확정된 후(3.7 라우팅 이후), 페르소나들은 종종 자신의 잔류/지연 배열에 동일한 실질적 내용을 다시 표시합니다 — 페르소나 자신의 발견 사항과 자신의 잔류 우려 사항이 동일한 이슈에 대한 것일 수 있습니다. 두 섹션을 그대로 렌더링하면 새로운 신호가 없는 재진술로 출력이 부풀려집니다.

모든 페르소나 출력의 `residual_risk` 및 `deferred_question` 항목마다 확정된 실행 가능 발견 사항 세트(앵커 `75` 또는 `100`인 발견 사항 및 앵커 `50`인 FYI 발견 사항)와 대조합니다. 다음 중 하나라도 해당하면 잔류/지연 항목을 제거합니다:

- **섹션 및 내용 중첩 (Section-and-substance overlap).** 잔류/지연 항목이 실행 가능 발견 사항과 동일한 섹션을 지칭하고, 그 내용이 발견 사항의 `title` 또는 `why_it_matters`와 퍼지 일치(동일한 우려 사항을 나타내는 핵심 명사/동사 공유)하는 경우.
- **실행 가능 발견 사항의 질문 형태.** 실행 가능 발견 사항의 권장 사항에 의해 직접 답변되거나 불필요해지는 주제를 다루는 지연된 질문. 예: 실행 가능 발견 사항 "Motivation cites no real incident" → 지연된 질문 "Is there a concrete triggering event?" — 발견 사항이 이미 이를 제기했으며, 질문은 이를 의문형으로 재진술한 것입니다.

진정으로 새로운 신호(실행 가능 발견 사항이 건드리지 않는 우려 사항이나 질문)를 도입하는 잔류/지연 항목은 제거하지 마십시오. 의심스러울 때는 유지하십시오 — 이 과정은 경계선상의 호출이 아닌 명백한 재진술을 위한 것입니다.

모든 페르소나에 걸친 병합된 세트에 대해 이 과정을 실행합니다. 0이 아닐 때 제거된 수를 Coverage 각주 라인으로 기록합니다: `Restated: N (residual/deferred items suppressed as duplicates of actionable findings)`. 순서: 각주는 Coverage 테이블 아래에 `Dropped:`, `Chains:`, `Restated:` 순서로 각 줄에 표시됩니다. N이 0이면 각주를 생략합니다.

## Phase 4: Apply and Present (적용 및 프레젠테이션)

**사용자 대면 어휘 규칙 (렌더링된 템플릿뿐만 아니라 Phase 4의 모든 사용자 가시 출력에 적용됨).** 내부 enum 값 — `safe_auto`, `gated_auto`, `manual`, `FYI` — 은 스키마 및 합성 산문 내부에 머뭅니다. 섹션 간의 자유 텍스트 내레이션, 전환 서문, 상태 라인, 확인 메시지를 포함하여 사용자가 Phase 4 출력에서 보게 되는 모든 단어는 반드시 사용자 대면 어휘를 사용해야 합니다: "fixes"(`safe_auto`용), "proposed fixes"(`gated_auto`용), "decisions"(앵커 `75` 또는 `100`인 `manual` 발견 사항용), "FYI observations"(앵커 `50`인 모든 발견 사항용). 유일한 예외는 렌더링된 테이블의 `Tier` 열로, 투명성을 위해 내부 enum을 표시하는 것으로 명시되어 있습니다. "safe_auto fixes applied" 또는 "N safe_auto findings"와 같은 내레이션을 내보내지 마십시오 — 대신 "fixes applied" 또는 "N fixes"라고 작성하십시오.

### Apply safe_auto fixes (safe_auto 수정 사항 적용)

**신뢰도 앵커 `100`인** `safe_auto` 발견 사항만 단일 패스로 문서에 적용합니다. 이는 3.7 라우팅 테이블과 일치합니다: 앵커 `100` + `safe_auto`는 무음 적용; 앵커 `75` + `safe_auto`는 3.7에서 `gated_auto`로 강등되어 워크스루에 진입; 앵커 `50` + 모든 `autofix_class`는 FYI로 라우팅되며 절대 자동 적용되지 않습니다.

- 플랫폼의 편집 도구를 사용하여 문서를 인라인으로 편집합니다.
- 렌더링된 출력의 "Applied fixes" 섹션을 위해 무엇이 변경되었는지 추적합니다 (`safe_auto`는 내부 enum이며, 렌더링된 섹션 헤더는 "Applied fixes"입니다).
- 승인을 요청하지 마십시오 — 이들은 하나의 명확하고 올바른 수정안을 가지고 있으며 증거가 직접 확인(앵커 `100`)해 줍니다.
- 앵커 `75` 또는 `50`인 `safe_auto` 발견 사항은 절대 무음 적용하지 마십시오. 발견 사항이 `autofix_class: safe_auto`이고 앵커가 `100` 미만인 상태로 이 단계에 도달했다면 3.7 라우팅 규칙이 올바르게 적용되지 않은 것이므로, 계속하기 전에 해당 발견 사항에 대해 3.7을 다시 실행하십시오.

사용자가 무엇이 변경되었는지 알 수 있도록 출력 요약에 적용된 모든 수정 사항을 나열하십시오. 각 수정의 실질적인 내용(섹션, 변경 내용, 검토자 할당)을 전달할 수 있을 정도로 상세하게 작성하십시오. 이는 내용을 추가하거나 문서의 의미를 건드리는 수정의 경우 특히 중요합니다 — 사용자가 검토 결과가 무엇을 했는지 이해하기 위해 문서를 직접 비교(diff)하게 해서는 안 됩니다.

### Route Remaining Findings (남은 발견 사항 라우팅)

safe_auto 수정 사항이 적용된 후, 남은 발견 사항들은 버킷으로 나뉩니다:

- 신뢰도 앵커 `75` 또는 `100`인 `gated_auto` 및 `manual` 발견 사항 → 라우팅 질문으로 진입 (Unit 5 / `references/walkthrough.md` 참조)
- FYI 하위 섹션 발견 사항 → 프레젠테이션에만 표시되고 라우팅되지 않음
- 남은 실행 가능 발견 사항이 0개인 경우 → 라우팅 질문을 건너뛰고 Phase 5 최종 질문으로 직접 이동

**헤드리스 모드 (Headless mode):** 대화형 질문 도구를 사용하지 마십시오. 호출자가 파싱할 수 있는 구조화된 텍스트 엔벨로프로 모든 발견 사항을 출력하십시오. 내부 enum 값(`safe_auto`, `gated_auto`, `manual`, `FYI`)은 스키마 및 합성 산문에 머뭅니다. 아래 엔벨로프는 사용자 대면 어휘 — "fixes", "Proposed fixes", "Decisions", "FYI observations" — 를 사용하여 헤드리스 출력이 대화형 출력과 동일하게 읽히도록 합니다.

```

- <section>: <what was changed> (<reviewer>)
- <section>: <what was changed> (<reviewer>)
```

**FYI observations, residual concerns, deferred questions의 압축 렌더링 (높은 카운트 모드).** 이 세 버킷의 합계가 5개 이상인 경우, 각 섹션을 한 줄 카운트로 축소하고 항목별 `Why` 확장 없이 촘촘한 불렛 리스트로 표시합니다. 실행 가능 버킷(Proposed fixes / Decisions)은 무관하게 전체 렌더링됩니다. 이는 두 모드가 동일한 모양을 생성하도록 `references/review-output-template.md`의 대화형 모드 규칙을 미러링한 것입니다.

**대화형 모드 (Interactive mode):**

- Errors (설계 긴장, 모순, 잘못된 진술)가 먼저 — 이들은 해결이 필요함
- Omissions (누락된 단계, 부재한 세부 사항, 잊혀진 항목)가 두 번째 — 이들은 추가가 필요함

Coverage 테이블, 적용된 수정 사항, FYI 관찰(별도의 하위 섹션으로), 잔류 우려 사항 및 지연된 질문을 포함합니다.

**모든 테이블은 반드시 파이프 구분 마크다운(`| col | col |`)이어야 합니다. 어떤 상황에서도 Coverage 테이블을 포함하여 ASCII 박스 그리기 문자(`┌ ┬ ┐ ├ ┼ ┤ └ ┴ ┘ │ ─`)를 사용하지 마십시오.** 이 규칙은 렌더링 시점에 템플릿의 서식 요구 사항을 재진술하여 어긋나지 않도록 합니다. 파이프 구분 테이블은 모든 대상 하네스에서 올바르게 렌더링됩니다. 박스 그리기 문자는 일부에서 렌더링을 깨뜨리며 루트 `AGENTS.md`에 기록된 저장소 컨벤션을 위반합니다.

### R29 Rejected-Finding Suppression (라운드 2+ 기 거부된 발견 사항 억제)

오케스트레이터가 동일한 세션에서 동일한 문서에 대해 라운드 2 이상을 실행 중일 때, 결정 프라이머(`SKILL.md` — Decision primer 참조)는 이전 라운드의 모든 Skipped, Deferred 및 Acknowledged 발견 사항을 전달합니다. 합성은 재차 제기된 거부된 발견 사항을 사용자에게 다시 표시하지 않고 억제합니다. 여기서 Acknowledged는 거부 클래스 결정으로 취급됩니다: 사용자가 발견 사항을 보았고, 조치하지 않기로 선택했으며(Apply 안 함, Defer 추가 안 함), 이를 기록에 남기길 원합니다 — 억제 목적상 Skip과 동일합니다.

이번 라운드의 각 발견 사항에 대해 프라이머의 거부 리스트와 비교합니다:

- **일치 조건 (Matching predicate):** R30과 동일 — `normalize(section) + normalize(title)` 지문에 증거 부분 문자열 일치 확인(>50%)을 결합합니다. 이번 라운드 발견 사항이 지문 및 증거 중첩에서 이전 라운드 거부된 발견 사항과 일치하면 이번 라운드 발견 사항을 제거합니다.
- **실질적 차이 예외 (Materially-different exception):** 이전 라운드 이후 발견 사항의 섹션 주변 문서 상태가 변경된 경우(예: 섹션이 편집되어 증거 인용문이 현재 텍스트에 더 이상 나타나지 않음), 발견 사항을 새로운 것으로 취급합니다 — 기본 컨텍스트가 바뀌었으며 우려 사항이 진정으로 다를 수 있습니다. 페르소나의 증거 자체가 이를 드러냅니다: 현재 문서에 나타나지 않는 인용문은 이전 라운드 거부가 더 이상 적용되지 않는다는 신호입니다.
- **억제 시:** 사용자가 무엇이 억제되었는지 볼 수 있도록 Coverage에 "previously rejected, re-raised this round" 노트와 함께 누락 사실을 기록합니다. 사용자는 억제가 잘못되었다고 판단되는 경우 다른 컨텍스트에서 검토를 다시 호출하여 명시적으로 에스컬레이션할 수 있습니다.

이 규칙은 페르소나 레벨이 아닌 합성 시점에 실행됩니다. 페르소나는 서브 에이전트 템플릿의 `{decision_primer}` 변수를 통해 거부된 발견 사항을 다시 제기하지 말라는 권고 지침을 받지만, 오케스트레이터가 권위 있는 게이트입니다 — 페르소나가 프라이머에도 불구하고 다시 제기하면 합성이 발견 사항을 제거합니다.

### R30 Fix-Landed Matching Predicate (수정 반영 일치 조건)

오케스트레이터가 동일한 문서에 대해 라운드 2 이상을 실행 중일 때(Unit 7 다라운드 메모리 참조), 합성은 이전 라운드의 Applied 발견 사항이 실제로 반영되었는지 확인합니다. `normalize(section) + normalize(title)` 지문이 이전 라운드 Applied 발견 사항과 일치하는 각 이번 라운드 발견 사항에 대해 증거 중첩에 따라 분기합니다:

- **강한 일치 — 이전 라운드 증거와 증거 중첩 >50%: 수정 반영 실패 (regression).** 이번 라운드 발견 사항이 이전 라운드 수정 사항이 제거했어야 할 동일한 문제 텍스트를 인용하고 있습니다. 새로운 발견 사항으로 표시하는 대신 보고서에 "fix did not land"로 플래그를 세웁니다. 사용자가 검증이 플래그를 세운 이유를 볼 수 있도록 이전 라운드 발견 사항의 제목과 이번 라운드 페르소나의 증거를 포함합니다.

- **약한 일치 — 증거 중첩 ≤50%: 수정 반영 실패 아님.** 낮은 증거 중첩은 이전의 문제 텍스트가 더 이상 인용되지 않음을 의미하므로 "fix did not land" 플래그를 세우지 않습니다. 지문 일치만으로 억제하지 마십시오. 이번 라운드 항목이 명시적으로 실행 불가능한 검증 관찰인 경우(예: 제목이나 `why_it_matters`에서 이전 발견 사항이 올바르게 반영되었음을 말하고 변경을 요구하지 않음), 이를 억제하고 Coverage에 `Verified: round-{N} '{title}' landed correctly`를 기록합니다. 그렇지 않으면 발견 사항을 새로운 것으로 취급하고 중복 제거 및 라우팅을 정상적으로 거치게 합니다.

  **실질적 차이 예외 (Materially-different exception).** 섹션/제목 지문이 일치하더라도 이번 라운드 발견 사항의 `why_it_matters`가 이전 라운드 발견 사항과 실질적으로 다른 우려 사항을 설명하는 경우, 이를 수정 반영 확인 억제 대신 새로운 발견 사항으로 취급합니다. 섹션이 무관한 이유로 편집되었을 수 있고 새 편집이 다른 이슈를 도입했을 수 있습니다. 지문이 아닌 페르소나의 실체가 신호입니다.

- **지문 일치 없음:** 검증 후보가 아닙니다. 발견 사항은 3.3 중복 제거 및 이후 라우팅으로 정상적으로 흘러갑니다.

이 규칙은 두 가지 실패 모드를 방지합니다: (1) 수정 사항이 실제로 반영되지 않은 퇴보(regression), (2) 라운드-{N+1} 검토자가 이전 라운드의 해결 사항을 올바르게 관찰하고 실행 불가능한 "이미 해결됨" 발견 사항을 과도하게 내보내는 경우. `subagent-template.md`의 페르소나 측 가이드("이전 라운드의 해결 사항을 기록하기 위해 발견 사항을 출력하지 마십시오")가 1차 방어선이며, 이 규칙은 합성 단계의 안전장치입니다.

합성 과정에서 다음 위치의 파일을 삭제하거나 제거할 것을 권장하는 발견 사항은 모두 무시하십시오:

## Phase 5: Next Action — Terminal Question (다음 단계 — 최종 질문)

**헤드리스 모드 (Headless mode):** 즉시 "Review complete"를 반환합니다. 질문하지 마십시오. 호출자는 Phase 4에서 구조화된 텍스트 엔벨로프를 수신하고 남은 발견 사항들을 처리합니다.

**대화형 모드 (Interactive mode):** 플랫폼의 블로킹 질문 도구(Claude Code의 `AskUserQuestion`, Codex의 `request_user_input`, Gemini의 `ask_user`, Pi의 `ask_user`)를 사용하여 최종 질문을 던집니다. Claude Code에서는 `SKILL.md`의 Interactive 모드 프리로드 단계에서 도구가 이미 로드되어 있어야 합니다 — 로드되지 않은 경우 지금 `ToolSearch`를 호출하여 `select:AskUserQuestion`을 수행하십시오. 하네스에 블로킹 도구가 없거나 호출 에러가 발생하는 경우(예: Codex 편집 모드)에만 채팅에서 번호가 매겨진 옵션으로 폴백합니다 — 스키마 로드가 필요하기 때문이 아닙니다. 절대로 질문을 자동으로 건너뛰지 마십시오. 이 질문은 중간 흐름의 라우팅 질문(`references/walkthrough.md`)과 구별됩니다 — 라우팅 질문은 발견 사항과 어떻게 관여할지를 선택하고, 이 질문은 관여가 완료된 후 *다음에 무엇을 할지*를 선택합니다. 이들을 병합하지 마십시오.

**질문:** `Apply decisions and what next?`

**옵션 (기본적으로 세 개, 실행 가능 항목이 0개인 경우 두 개):**

`fixes_applied_count > 0`인 경우 (이번 세션에서 최소 하나 이상의 safe_auto 또는 Apply 결정이 반영됨):

```
A. Apply decisions and proceed to <next stage>
B. Apply decisions and re-review
C. Exit without further action
```

`fixes_applied_count == 0`인 경우 (실행 가능 항목 0개 또는 사용자가 라우팅 옵션 D를 선택 / 모든 워크스루 결정이 Skip인 경우):

```
A. Proceed to <next stage>
B. Exit without further action
```

`<next stage>` 치환은 Phase 1의 문서 유형을 사용합니다:

- 요구사항 문서(Requirements document) → `genie:plan`
- 계획 문서(Plan document) → `genie:work`

**레이블 조정:** 적용할 대기 중인 결정이 없는 경우, 기본 옵션에서 `Apply decisions and` 접두사를 뺍니다 — 레이블은 시스템이 수행하는 작업과 일치해야 합니다. 수정 사항이 대기 중일 때는 `Apply decisions and proceed`, 아무것도 대기 중이지 않을 때는 `Proceed`를 사용합니다.

**호출자 컨텍스트 처리 (암시적):** 최종 질문의 "Proceed to <next stage>" 옵션은 가시적인 대화 상태로부터 에이전트에 의해 컨텍스트에 따라 해석됩니다. doc-review가 다른 스킬의 흐름 내부에서 호출된 경우(예: genie:brainstorm Phase 4 재검토, genie:plan phase 5.3.8), 에이전트는 중첩된 `/genie:plan` 또는 `/genie:work` 디스패치를 실행하지 않습니다 — 자체 로직을 계속 수행하는 호출자의 흐름으로 제어권을 반환합니다. 단독으로 호출된 경우 "Proceed"는 적절한 다음 스킬을 디스패치합니다. 명시적인 호출자 힌트 인자는 필요하지 않습니다. 이 암시적 처리가 실제 환경에서 신뢰할 수 없는 것으로 판명되면 명시적인 `nested:true` 플래그를 후속 조치로 추가할 수 있습니다.

### Iteration limit (반복 제한)

2번의 개선 패스 후에는 완료를 권장하십시오 — 수확 체감의 법칙이 적용될 가능성이 높습니다. 하지만 사용자가 계속하기를 원한다면 허용하십시오. 프라이머가 모든 이전 라운드 결정을 전달하므로 이후 라운드는 반복되는 발견 사항을 깔끔하게 억제합니다.

사용자가 어떤 옵션을 선택했든 관계없이 호출자를 위해 "Review complete"를 터미널 신호로 반환합니다.

## What NOT to Do (하지 말아야 할 것)

- 문서 전체를 다시 작성하지 마십시오.
- 사용자가 논의하지 않은 새로운 섹션이나 요구사항을 추가하지 마십시오.
- 과도하게 엔지니어링하거나 복잡성을 추가하지 마십시오.
- 별도의 검토 파일을 만들거나 메타데이터 섹션을 추가하지 마십시오.
- 호출자 스킬(genie:brainstorm, genie:plan 또는 doc-review를 호출하는 외부 플러그인 스킬)을 수정하지 마십시오.

## Iteration Guidance (반복 지침)

후속 패스에서는 다라운드 결정 프라이머(Unit 7 참조)와 함께 페르소나를 다시 디스패치하고 재합성합니다. 수정된 발견 사항은 현재 문서에서 증거가 사라졌으므로 스스로 억제됩니다. 거부된 발견 사항은 R29 패턴 일치 억제 규칙에 의해 처리됩니다. 적용된 수정 사항 확인은 위의 R30 일치 조건을 사용합니다. 이러한 메커니즘 실행 후에도 발견 사항이 패스 전반에 걸쳐 반복된다면 완료를 권장하십시오.
