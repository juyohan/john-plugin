# 프론트엔드 디자인 (Frontend Design)

## 다중 에이전트 협업 (Multi-Agent Collaboration)

사용자의 입력(`$ARGUMENTS`) 내에 `--add <ai-이름>` 형태의 플래그가 포함되어 있는지 확인하십시오. 
현재 지원되는 외부 AI 인터페이스는 `--add gemini` (또는 `--add gem`)입니다.

만약 해당 플래그가 감지되면, 작업을 단독으로 확정하지 말고 다음 절차를 따르십시오:
1. **의도 파악:** 플래그를 제외한 나머지 문자열을 실제 지시사항으로 간주합니다.
2. **초안 작성:** 본인(주 에이전트)의 지식과 코드베이스 컨텍스트를 바탕으로 작업의 초기 뼈대나 접근법을 생각합니다.
3. **MCP 협업 호출:** `gem` 도구를 호출하여 외부 Gemini 에이전트에게 조언이나 검토를 구합니다.
   - 호출 시 전달할 메시지 예시: "나는 현재 이 작업에 대한 초안을 세우고 있어. 내 초안은 [초안 요약]이야. 이 접근 방식의 기술적 타당성을 검토하고 누락된 에지 케이스나 더 나은 패턴을 조언해줄 수 있어?"
4. **결과 통합:** `gem` 도구가 반환한 피드백을 당신의 최종 결과물에 통합(Synthesis)합니다. 
5. **명시적 표시:** 최종 산출물의 상단 또는 설명 부분에 "이 결과물은 Gemini와의 협업을 통해 검토 및 보완되었습니다."라는 문구를 추가하십시오.

이 협업 절차를 염두에 두고 아래의 본래 스킬 워크플로우를 진행하십시오.


일반적인 AI 미학을 피하고 독특하고 프로덕션 수준의 프론트엔드 인터페이스 생성을 가이드합니다. 이 스킬은 컨텍스트 감지, 디자인 계획, 의도적인 구축, 시각적 검증의 전체 라이프사이클을 다룹니다.

## 권위 계층 (Authority Hierarchy)

이 스킬의 모든 규칙은 절대적인 명령이 아닌 기본 설정(default)입니다.

1. **기존 디자인 시스템 / 코드베이스 패턴** -- 최우선 순위이며 항상 존중됩니다.
2. **사용자의 명시적 지침** -- 스킬의 기본 설정을 덮어씁니다.
3. **스킬 기본 설정** -- 신규 구축 작업이나 사용자가 디자인 가이드를 요청할 때 적용됩니다.

기존 패턴이 확립된 코드베이스에서 작업할 때는 해당 패턴을 따르십시오. 사용자가 기본 설정과 상충되는 방향을 지정하면 사용자를 따르십시오.

## 워크플로우

```
컨텍스트 감지 -> 디자인 계획 -> 구축 -> 시각적 검증
```

## 레이어 0: 컨텍스트 감지 (Context Detection)

디자인 작업을 시작하기 전에 코드베이스에서 기존 디자인 신호를 조사하십시오. 이는 이 스킬의 의견이 반영된 가이드가 얼마나 적용될지를 결정합니다.

### 조사 항목

- **디자인 토큰 / CSS 변수**: `--color-*`, `--spacing-*`, `--font-*` 커스텀 속성, 테마 파일
- **컴포넌트 라이브러리**: shadcn/ui, Material UI, Chakra, Ant Design, Radix 또는 프로젝트 전용 컴포넌트 디렉토리
- **CSS 프레임워크**: `tailwind.config.*`, `styled-components` 테마, Bootstrap 임포트, 일관된 명명 규칙을 가진 CSS 모듈
- **타이포그래피**: HTML/CSS의 폰트 임포트, `@font-face` 선언, Google Fonts 링크
- **색상 팔레트**: 정의된 색상 스케일, 브랜드 색상 파일, 디자인 토큰 내보내기
- **애니메이션 라이브러리**: Framer Motion, GSAP, anime.js, Motion One, Vue Transition 임포트
- **간격 / 레이아웃 패턴**: 일관된 간격 스케일 사용, 그리드 시스템, 레이아웃 컴포넌트

이러한 신호를 스캔하기 위해 플랫폼의 네이티브 파일 검색 및 콘텐츠 검색 도구(예: Claude Code의 Glob/Grep)를 사용하십시오. 일상적인 파일 탐색에 쉘 명령을 사용하지 마십시오.

### 모드 분류

감지된 신호를 바탕으로 모드를 선택하십시오:

- **기존 시스템 (Existing system)** (여러 카테고리에 걸쳐 4개 이상의 신호): 기존 시스템을 따릅니다. 이 스킬의 미학적 의견(타이포그래피, 색상, 모션)은 확립된 시스템에 양보합니다. 구조적 가이드(구성, 카피, 접근성, 검증)는 여전히 적용됩니다.
- **부분 시스템 (Partial system)** (1~3개의 신호): 존재하는 것을 따르되, 컨벤션이 감지되지 않은 영역에만 스킬 기본 설정을 적용합니다. 예를 들어 Tailwind는 설정되어 있지만 컴포넌트 라이브러리가 없는 경우, Tailwind 토큰을 따르면서 컴포넌트 구조에는 스킬 가이드를 적용합니다.
- **신규 구축 (Greenfield)** (신호가 감지되지 않음): 스킬 가이드가 전적으로 적용됩니다.
- **모호함 (Ambiguous)** (신호가 상충되거나 불분명함): 진행하기 전에 사용자에게 확인하십시오.

### 사용자에게 질문하기

컨텍스트가 모호할 때 플랫폼의 차단형 질문 도구를 사용하십시오: Claude Code의 `AskUserQuestion` (스키마가 로드되지 않은 경우 `ToolSearch`로 `select:AskUserQuestion` 먼저 호출), Codex의 `request_user_input`, Gemini의 `ask_user`, Pi의 `ask_user` (`pi-ask-user` 확장 필요). 도구가 없거나 오류가 발생하는 경우에만 채팅 창에 옵션을 제시하십시오. 질문을 소리 없이 건너뛰지 마십시오. 사용자가 선택을 거부하면 "부분 시스템" 모드로 간주하고 보수적으로 진행하십시오.

예시 질문: "[감지된 신호]를 발견했습니다. 기존 디자인 패턴을 따를까요, 아니면 독특한 스타일을 새로 만들까요?"

## 레이어 1: 구축 전 계획 (Pre-Build Planning)

코드를 작성하기 전에 세 가지 짧은 문장을 작성하십시오. 이는 일관성을 만들고 사용자가 코드가 작성되기 전에 방향을 수정할 수 있는 체크포인트를 제공합니다.

1. **시각적 논지 (Visual thesis)** -- 분위기, 소재, 에너지를 설명하는 한 문장
   - 신규 구축 예시: "깔끔한 에디토리얼 느낌, 넓은 여백, 세리프 헤드라인, 차분한 어스 톤(earth tones)" 또는 "데이터 중심의 밀도 높은 대시보드, 모노스페이스 강조, 어두운 표면 계층"
   - 기존 코드베이스: 기존의 미학을 설명하고 새로운 작업이 이를 어떻게 확장할지 설명

2. **콘텐츠 계획 (Content plan)** -- 페이지에 무엇이 어떤 순서로 들어가는지
   - 랜딩 페이지: 히어로, 지원, 상세, CTA
   - 앱: 주요 작업 공간, 네비게이션, 보조 컨텍스트
   - 컴포넌트: 어떤 상태를 가지며 무엇을 전달하는지

3. **상호작용 계획 (Interaction plan)** -- 분위기를 바꾸는 2~3개의 구체적인 모션 아이디어
   - "애니메이션 추가"가 아니라 "히어로 로드 시 순차적 페이드인, 섹션 간 스크롤 시 패럴랙스(parallax) 효과, 카드 호버 시 스케일 업"과 같이 작성
   - 기존 코드베이스에서는 기존 모션 라이브러리를 사용하여 추가되는 인터랙션만 설명

## 레이어 2: 디자인 가이드 핵심

이 원칙들은 모든 컨텍스트 유형에 적용됩니다. 각 원칙은 권위 계층에 따라 기존 디자인 시스템과 사용자의 지침에 양보합니다.

### 타이포그래피 (Typography)

- 독특하고 개성 있는 폰트를 선택하십시오. 기존 코드베이스에서 사용하지 않는 한 흔한 폰트(Inter, Roboto, Arial, 시스템 기본 폰트)는 피하십시오.
- 명확한 이유가 없는 한 최대 두 개의 서체만 사용하십시오. 디스플레이/헤드라인 폰트와 본문용 폰트를 조합하십시오.
- *레이어 0에서 기존 폰트 선택이 감지되면 해당 폰트를 따릅니다.*

### 색상 및 테마 (Color & Theme)

- CSS 변수를 사용하여 응집력 있는 팔레트를 구성하십시오. 차분하고 고르게 분포된 팔레트보다 강렬한 강조색(accent)이 있는 지배적인 색상이 더 효과적입니다.
- 화이트 배경에 보라색, 다크 모드 등에 대한 편향을 갖지 마십시오. 컨텍스트에 따라 밝고 어두운 톤을 다양하게 사용하십시오.
- 제품에 이미 다색 시스템이 있지 않은 한 기본적으로 하나의 강조색을 사용하십시오.
- *기존 색상 토큰이 감지되면 해당 토큰을 따릅니다.*

### 구성 (Composition)

- 컴포넌트가 아닌 구성부터 시작하십시오. 첫 번째 뷰포트를 문서가 아닌 포스터처럼 취급하십시오.
- 테두리(border), 그림자(shadow), 카드와 같은 장식을 추가하기 전에 여백, 정렬, 스케일, 크로핑(cropping), 대비를 활용하십시오.
- 기본적으로 카드 없는 레이아웃을 사용하십시오. 카드는 사용자 상호작용의 컨테이너(클릭 가능한 항목, 드래그 가능한 단위, 선택 가능한 옵션) 역할을 할 때만 허용됩니다. 카드 스타일을 제거해도 이해에 지장이 없다면 카드를 사용하지 말아야 합니다.
- *모든 구성 규칙은 기본 설정입니다. 사용자는 이를 덮어쓸 수 있습니다.*

### 모션 (Motion)

- 시각적으로 중요한 작업의 경우 2~3개의 의도적인 모션을 적용하십시오: 하나의 등장 시퀀스, 하나의 스크롤 연결 또는 깊이 효과, 하나의 호버/리빌(reveal) 전환.
- 프로젝트에 기존 애니메이션 라이브러리가 있다면 이를 사용하십시오.
- 기존 라이브러리가 없는 경우 프레임워크별 기본 설정을 사용하십시오:
  - 범용적인 베이스라인으로 **CSS 애니메이션**
  - React 프로젝트의 경우 **Framer Motion**
  - Vue 프로젝트의 경우 **Vue Transition / Motion One**
  - Svelte 프로젝트의 경우 **Svelte transitions**
- 모션은 빠른 녹화에서도 눈에 띄고, 모바일에서 매끄러우며, 페이지 전체에서 일관되어야 합니다. 순수하게 장식적이기만 하다면 제거하십시오.

### 접근성 (Accessibility)

- 기본적으로 시맨틱 HTML을 사용하십시오: `nav`, `main`, `section`, `article`, `button` 등을 사용하고 모든 것에 div를 남발하지 마십시오.
- WCAG AA 최소 기준을 충족하는 색상 대비.
- 모든 인터랙티브 요소에 가시적인 포커스 상태(focus states) 제공.
- 접근성과 미학은 제대로 구현되었을 때 서로 상충하지 않습니다.

### 이미지 (Imagery)

- 이미지가 필요한 경우 추상적인 그라데이션이나 가짜 3D 객체보다 실제 또는 사실적인 사진을 선호하십시오.
- 텍스트 오버레이를 위해 안정적인 톤의 영역이 있는 이미지를 선택하거나 생성하십시오.
- 환경에서 이미지 생성 도구를 사용할 수 있다면 플레이스홀더 스톡 이미지 대신 컨텍스트에 맞는 시각 자료를 직접 생성하십시오.

## 컨텍스트 모듈

구축 중인 항목에 맞는 모듈을 선택하십시오. 기존 애플리케이션 내부에서 작업할 때는 기능에 관계없이 모듈 C를 기본으로 사용합니다.

### 모듈 A: 랜딩 페이지 및 마케팅 (신규 구축)

**기본 섹션 순서:**
1. 히어로 -- 브랜드/제품, 약속, CTA, 하나의 지배적인 비주얼
2. 지원 -- 하나의 구체적인 기능, 제안 또는 증거 포인트
3. 상세 -- 분위기, 워크플로우, 제품의 깊이 또는 스토리
4. 최종 CTA -- 전환, 시작, 방문 또는 문의

**히어로 규칙 (기본 설정):**
- 대시보드가 아닌 하나의 구성으로 만드십시오. 풀 블리드(full-bleed) 이미지 또는 지배적인 비주얼 평면을 사용하십시오.
- 브랜드가 첫 번째, 헤드라인이 두 번째, 본문이 세 번째, CTA가 네 번째입니다.
- 텍스트 컬럼은 좁게 유지하고 이미지의 차분한 영역에 고정하십시오.
- 명확한 이유가 없는 한 섹션은 총 6개를 넘지 않도록 하십시오.
- 하나의 H1 헤드라인. 스크롤 없이 보이는 영역(above the fold)에 하나의 기본 CTA.

**카피 (Copy):**
- 헤드라인이 의미를 전달하게 하십시오. 보조 카피는 대개 하나의 짧은 문장입니다.
- 디자인 코멘터리가 아닌 제품 언어로 작성하십시오. UI에 프롬프트 언어나 AI 코멘터리를 넣지 마십시오.
- 각 섹션에는 하나의 역할만 부여하십시오: 설명, 증명, 심화 또는 전환.
- 모든 문장은 존재 이유가 있어야 합니다. 카피는 많은 것보다 적은 것을 기본으로 합니다.

### 모듈 B: 앱 및 대시보드 (신규 구축)

**기본 패턴:**
- 차분한 표면 계층, 강력한 타이포그래피와 간격, 적은 색상, 밀도 높지만 읽기 쉬운 정보, 최소한의 장식.
- 주요 작업 공간, 네비게이션, 보조 컨텍스트/인스펙터, 동작이나 상태를 위한 하나의 명확한 강조색을 중심으로 구성하십시오.
- 카드는 해당 카드가 인터랙션(클릭 가능한 항목, 드래그 가능한 단위, 선택 가능한 옵션)일 때만 사용하십시오. 패널이 스타일 없이도 의미를 잃지 않는다면 카드 처리를 제거하십시오.

**카피 (마케팅이 아닌 유틸리티):**
- 약속, 분위기 또는 브랜드 보이스보다 오리엔테이션, 상태 및 동작을 우선시하십시오.
- 섹션 헤더는 해당 영역이 무엇인지 또는 사용자가 거기서 무엇을 할 수 있는지를 말해야 합니다. 좋은 예: "계획 상태", "검색 지표". 나쁜 예: "당신의 잠재력을 깨우세요".
- 문장이 홈페이지 히어로 섹션에 어울릴 법하다면 제품 UI처럼 들릴 때까지 다시 작성하십시오.
- 판별법: 운영자가 헤드라인, 라벨, 숫자만 훑어보고도 페이지를 즉시 이해할 수 있습니까?

### 모듈 C: 컴포넌트 및 기능 (기존 앱 내 기본 설정)

기존 애플리케이션에 추가하는 경우:

- 기존의 시각 언어에 맞추십시오. 이 모듈은 눈에 띄는 것이 아니라 소속감을 주는 것에 집중합니다.
- 주변 코드로부터 간격 스케일, 테두리 반경(border radius), 색상 토큰 및 타이포그래피를 상속받으십시오.
- 인터랙션 품질에 집중하십시오: 명확한 상태(기본, 호버, 액티브, 비활성, 로딩, 에러), 상태 간의 부드러운 전환, 명확한 어포던스(affordance).
- 하나의 새로운 컴포넌트가 새로운 디자인 시스템을 도입해서는 안 됩니다. 기존 앱이 4px 테두리 반경을 사용한다면 8px 컴포넌트를 추가하지 마십시오.

## 엄격한 규칙 및 안티 패턴

### 권장하지 않음 (사용자 오버라이드 가능)

이들은 스킬의 주관적인 의견입니다. 사용자는 이 중 어느 것이든 덮어쓸 수 있습니다.

- 첫인상으로 흔한 SaaS 카드 그리드 사용
- 화이트 배경의 보라색 색상 체계, 다크 모드 편향
- 신규 구축 작업에서 남용되는 폰트(Inter, Roboto, Arial, Space Grotesk, 시스템 기본 폰트) 사용
- 통계, 일정, 알약 모양 클러스터, 로고 클라우드로 가득 찬 히어로 섹션
- 동일한 분위기의 문구를 다른 말로 반복하는 섹션들
- 서사적 목적이 없는 캐러셀(Carousel)
- 여러 개의 경쟁적인 강조색
- 실제 비주얼 콘텐츠 대신 장식적인 그라데이션이나 추상적인 배경 사용
- 디자인 코멘터리처럼 들리는 카피 ("매끄러운 통합을 경험하세요")
- 텍스트가 복잡한 이미지 쪽에 위치한 분할 화면 히어로

### 항상 피함 (품질 하한선)

이들은 어떤 사용자도 원치 않을 진정한 품질 실패 사례입니다.

- 프롬프트 언어나 AI 코멘터리가 UI에 유출됨
- 깨진 대비 -- 이미지나 배경 위의 텍스트를 읽을 수 없음
- 가시적인 포커스 상태가 없는 인터랙티브 요소
- 적절한 HTML 요소가 존재함에도 모든 것을 div로 작성함 (div soup)

## 리트머스 테스트 (Litmus Checks)

시각적 검증으로 넘어가기 전의 빠른 자기 검토입니다. 모든 항목이 모든 컨텍스트에 적용되는 것은 아닙니다 — 관련 있는 항목에 대해 판단력을 발휘하십시오.

- 첫 화면에서 브랜드나 제품이 명확히 드러납니까?
- 하나의 강력한 시각적 앵커(anchor)가 있습니까?
- 헤드라인만 훑어봐도 페이지를 이해할 수 있습니까?
- 각 섹션이 하나의 역할을 합니까?
- 카드가 사용된 곳에 정말로 카드가 필요합니까?
- 모션이 계층 구조나 분위기를 개선합니까, 아니면 그냥 거기 있습니까?
- 모든 장식적인 그림자를 제거해도 디자인이 고급스럽게 느껴집니까?
- 카피가 프롬프트가 아닌 제품처럼 들립니까?
- 새로운 작업이 기존 디자인 시스템과 일치합니까? (모듈 C)

## 시각적 검증 (Visual Verification)

구현 후 시각적으로 확인하십시오. 이는 픽셀 단위의 완벽한 리뷰가 아닌 상식적인 수준의 확인입니다. 한 번의 패스로 진행하며, 눈에 띄는 문제가 있으면 수정하십시오. 괜찮아 보이면 다음으로 넘어가십시오.

### 도구 우선순위

가장 먼저 사용 가능한 옵션을 선택하십시오:

1. **기존 프로젝트 브라우저 도구** -- Playwright, Puppeteer, Cypress 등이 이미 프로젝트 종속성에 있다면 이를 사용하십시오. 검증만을 위해 새로운 종속성을 도입하지 마십시오.
2. **브라우저 MCP 도구** -- 에이전트 환경에서 브라우저 자동화 도구(예: claude-in-chrome)를 사용할 수 있다면 이를 사용하십시오.
3. **agent-browser CLI** -- 다른 대안이 없고 `agent-browser`가 설치되어 있다면 이를 사용하십시오. 설치되어 있지 않다면 사용자에게 알려주십시오: "`agent-browser`가 설치되어 있지 않습니다. `/ce-setup`을 실행하여 필요한 종속성을 설치하십시오." 그 후 다음 옵션으로 건너뛰십시오.
4. **정신적 검토 (Mental review)** -- 브라우저 접근이 불가능한 경우(헤드리스 CI, 설치 권한 없음), 리트머스 테스트를 자기 검토용으로 적용하고 시각적 검증을 건너뛰었음을 명시하십시오.

### 평가 항목

- 결과물이 구축 전 계획의 시각적 논지와 일치합니까?
- 명백한 시각적 문제(깨진 레이아웃, 읽을 수 없는 텍스트, 누락된 이미지)가 있습니까?
- 의도한 컨텍스트 모듈처럼 보입니까(랜딩 페이지는 랜딩 페이지처럼, 대시보드는 대시보드처럼, 컴포넌트는 주변과 어울리게)?

### 범위 제어

한 번의 반복. 스크린샷을 찍고 리트머스 테스트를 기준으로 평가한 뒤, 눈에 띄는 문제를 수정하고 마무리하십시오. 결과물(PR 설명, 대화 출력 등)에 스크린샷을 포함하십시오.

한 번의 패스를 넘어선 반복적인 개선(스크린샷-평가-수정의 여러 라운드)이 필요한 경우, `ce-design-iterator` 에이전트를 참조하십시오.

## 창의적 에너지 (Creative Energy)

이 스킬은 구조를 제공하지만, 목표는 공식적인 결과물이 아닌 AI가 만든 티가 나지 않는 독창적인 작업입니다.

신규 구축 작업의 경우, 대담한 미학적 방향을 설정하십시오. 브루탈리즘, 미니멀리즘, 맥시멀리즘, 레트로 퓨처리즘, 오가닉/내추럴, 럭셔리/리파인드, 플레이풀, 에디토리얼, 아트 데코, 소프트/파스텔, 인더스트리얼 등 다양한 분위기를 고려하거나 컨텍스트에 맞는 새로운 스타일을 발명하십시오. 영감을 위해 이들을 참고하되 프로젝트에 진실된 디자인을 하십시오.

자문해 보십시오: 무엇이 이것을 잊을 수 없게 만드는가? 사람들이 기억할 단 하나는 무엇인가?

구현 복잡도를 미학적 비전에 맞추십시오. 맥시멀리즘 디자인은 정교한 코드와 광범위한 애니메이션 및 효과가 필요합니다. 미니멀리즘 디자인은 절제와 정밀함, 그리고 간격, 타이포그래피, 미묘한 디테일에 대한 세심한 주의가 필요합니다. 우아함은 비전을 잘 실행하는 데서 오는 것이지 강도에서 오는 것이 아닙니다.
