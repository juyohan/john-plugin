# 작업 실행

<input_document> #$ARGUMENTS </input_document>

`--add gemini` (또는 `--add gem`) 플래그가 있으면: gem 도구로 Gemini에 초안 검토 요청, 결과 통합, 산출물 상단에 "Gemini와의 협업을 통해 검토 및 보완되었습니다." 추가.

## 1. 입력 분류

입력이 파일 경로이면 계획 문서 → 2단계로. 그 외 단순 프롬프트:

1. 변경될 파일, 기존 테스트, 로컬 패턴 파악
2. 복잡도 판단:

| 복잡도 | 신호 | 처리 |
|--------|------|------|
| **사소함** | 1-2개 파일, 동작 변경 없음 | 직접 구현. 작업 목록·실행 루프 없음 |
| **소형/중형** | 범위 명확, 10개 파일 미만 | 작업 목록 작성 후 2단계 진행 |
| **대형** | 횡단적·아키텍처, 10개 이상 파일 | `/genie:brainstorm` 또는 `/genie:plan` 권장. 사용자 선택에 따름 |

## 2. 준비

1. **계획 읽기** _(단순 프롬프트는 건너뜀)_
   - 계획을 실행 스크립트가 아닌 결정 결과물로 취급
   - `Implementation Units`, `Files`, `Test Scenarios`, `Verification`, `Scope Boundaries`, `Deferred to Implementation` 섹션 확인
   - 각 단위의 `Execution note` 기록 (테스트 우선 등)
   - 불명확한 점 있으면 지금 질문
   - 실행 중 계획 본문 수정 금지 (배포 시 `status: active → completed` 전환만 허용)

2. **브랜치 설정**
   - 기능 브랜치에 있으면: 이름이 의미 있는지 확인. 불투명한 이름은 변경 제안
   - 기본 브랜치에 있으면 선택:
     - **옵션 A**: `git checkout -b <meaningful-name>`
     - **옵션 B**: worktree 스킬 사용 (병렬 개발 권장)
     - **옵션 C**: 기본 브랜치에서 계속 (사용자 명시 확인 필수)

3. **작업 목록 생성** _(사소함 또는 이미 생성한 경우 건너뜀)_
   - 계획의 구현 단위, 종속성, 파일, 테스트 대상에서 작업 도출
   - 단위에 `U-ID`가 있으면 작업 제목에 접두사로 유지 (예: "U3: 파서 커버리지 추가")

4. **실행 전략 선택**

| 전략 | 사용 시기 |
|------|-----------|
| **인라인** | 1-2개 소규모 작업, 또는 단순 프롬프트 (기본값) |
| **직렬 서브에이전트** | 종속성 있는 3개 이상 작업 |
| **병렬 서브에이전트** | 파일 겹침 없는 3개 이상 작업 |

병렬 파견 전 파일 겹침 확인: 겹침 있고 worktree 격리 불가능하면 직렬로 강등.
worktree 격리 사용 시: `isolation: "worktree"`, `run_in_background: true`.
공유 디렉토리 폴백 시: 서브에이전트는 스테이징·커밋·테스트 실행 금지 — 오케스트레이터가 처리.

## 3. 실행

각 작업에 대해:
1. **멱등성 체크**: 검증 기준이 이미 충족되면 완료 표시 후 건너뜀 (재구현 금지)
2. 관련 파일 및 유사 패턴 읽기
3. **테스트 발견**: 구현 파일의 기존 테스트 파일 먼저 찾기
4. 기존 컨벤션 따라 구현
5. 테스트 추가·업데이트·제거
6. **시스템 전반 테스트 체크**: 콜백·미들웨어·옵저버 2단계 추적, 통합 테스트 최소 1개 (모킹 없이)
7. 테스트 실행 — 실패 즉시 수정
8. 작업 완료 표시
9. **증분 커밋 평가**: 테스트 통과 + 논리적 단위 완료 시
   ```bash
   git add <관련 파일들>   # git add . 지양
   git commit -m "feat(scope): 단위 설명"
   ```

일련의 관련 단위 완료 후 30라인 이상 변경 시: simplify 스킬로 중복 패턴 통합.
Figma 디자인이 있는 UI 작업이면 구현과 일치할 때까지 비교.

**병렬 서브에이전트 배치 완료 후:**
- worktree 격리: 종속성 순서대로 브랜치 머지 → 충돌 시 직렬로 재파견 → 워크트리 제거
- 공유 디렉토리: 실제 수정 파일 교차 확인 → 단위별 스테이징·커밋

## 4. 품질 검사

1. **테스트 및 린팅**
   ```bash
   # 전체 테스트 스위트 실행 (bin/rails test, npm test, pytest, go test 등)
   # 린터 실행 (AGENTS.md 참조)
   ```

2. **코드 리뷰**

   **Tier 1 (기본)**: `/genie:review` 실행. 차단·권장 발견 사항 인라인 처리.

   **Tier 2 (에스컬레이션)** — 아래 중 하나라도 해당 시:
   - 민감한 표면: 인증·결제·마이그레이션·암호화·공개 API·의존성 매니페스트
   - 400라인 이상 변경 + 3개 이상 디렉토리 또는 2개 이상 하위 시스템
   - 1,000라인 이상 변경
   - 계획 또는 작업에서 전체/심층 리뷰 명시 요청

   Tier 2 후 잔여 발견 사항이 있으면 처리 방법 선택:
   - 지금 수정 / AGENTS.md 트래커에 티켓 생성 / 수락 후 진행 / 중단

3. **최종 검증**
   - 모든 작업 완료 표시
   - 테스트 통과, 린팅 통과
   - 계획의 `Requirements` 각 항목 충족 여부 확인
   - `Deferred to Implementation` 질문 해결 여부 확인

4. **운영 검증 계획** (PR 본문에 포함)
   - 로그 쿼리, 모니터링 지표, 정상 신호, 실패 신호 및 롤백 트리거
   - 운영 영향 없으면: "운영 모니터링 불필요" + 이유 한 줄

## 5. 핸드오프

1. 계획 프론트매터 업데이트: `status: active → completed`
2. commit 및 PR 생성은 `/genie:commit` 또는 push-pr 스킬로 핸드오프:
   - PR 본문 컨텍스트로 전달: 계획 요약, 주요 결정, 테스트 노트, 운영 검증 계획, 수락된 잔여 사항
