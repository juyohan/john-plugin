# 요구사항 문서의 시각적 커뮤니케이션 (Visual Communication)

시각적 도구의 사용 여부는 문서의 깊이가 아닌 콘텐츠의 패턴에 따라 결정됩니다. 복잡한 워크플로우를 다루는 경량(Lightweight) 브레인스토밍에는 다이어그램이 필요할 수 있는 반면, 단순한 기능을 다루는 심화(Deep) 브레인스토밍에는 필요 없을 수도 있습니다.

**포함해야 하는 경우:**

| 요구사항이 다음을 설명할 때... | 시각적 도구 | 위치 |
|---|---|---|
| 여러 단계의 사용자 워크플로우 또는 프로세스 | Mermaid 플로우 다이어그램 또는 주석이 포함된 ASCII 플로우 | 문제 정의(Problem Frame) 뒤, 또는 대규모 플로우(노드 10개 초과)의 경우 별도의 `## User Flow` 헤딩 아래 |
| 3개 이상의 동작 모드, 변형, 또는 상태 | 마크다운 비교표 | 요구사항(Requirements) 섹션 내부 |
| 3개 이상의 상호작용 참여자 (사용자 역할, 시스템 구성 요소, 외부 서비스) | Mermaid 또는 ASCII 관계도 | 문제 정의(Problem Frame) 뒤, 또는 별도의 `## Architecture` 헤딩 아래 |
| 여러 경쟁적인 접근 방식 비교 | 비교표 | Phase 2 접근 방식 탐색 단계 내부 |

**생략해야 하는 경우:**
- 산문(prose)만으로도 개념이 명확하게 전달되는 경우
- 다이어그램이 요구사항을 시각적 형태로 단순히 반복할 뿐, 이해를 돕는 추가적인 가치를 제공하지 않는 경우
- 시각적 자료가 구현 아키텍처, 데이터 스키마, 상태 머신, 또는 코드 구조를 설명하는 경우 (이는 `ce-plan`의 영역임)
- 브레인스토밍 주제가 단순하고 선형적이며, 다단계 플로우나 모드 비교, 다자간 상호작용이 없는 경우

**형식 선택:**
- **Mermaid** (기본): 5-15개의 노드가 있는 단순한 플로우에 사용합니다. 박스 내 주석은 생략하고 표준 플로우차트 도형을 사용하십시오. 렌더링된 형태와 소스 형태 모두에서 너비를 좁게 유지할 수 있도록 `TB` (상단에서 하단) 방향을 사용하십시오. 소스 텍스트는 diff 뷰나 터미널에서 읽을 수 있는 폴백(fallback) 역할을 해야 합니다.
- **ASCII/박스 드로잉 다이어그램**: 각 단계의 CLI 명령어, 의사결정 로직 분기, 파일 경로 레이아웃, 다단 공간 배치 등 박스 내에 풍부한 주석이 필요한 플로우에 사용합니다. 단계 내부의 주석이 다이어그램의 핵심 가치일 때 Mermaid보다 표현력이 좋습니다. 코드 블록의 80자 제한을 준수하고 수직 스태킹 방식을 사용하십시오.
- **마크다운 표**: 모드/변형 비교 및 접근 방식 비교에 사용합니다.
- 다이어그램의 크기를 콘텐츠에 맞게 조절하십시오. 단순한 5단계 워크플로우는 5-10개의 노드로 충분합니다. 의사결정 분기와 단계별 주석이 포함된 복잡한 워크플로우는 15-20개의 노드가 필요할 수 있습니다 — 각 노드가 제 역할을 다한다면 괜찮습니다.
- 별도의 섹션이 아닌 관련 내용이 있는 위치에 인라인으로 배치하십시오.
- 개념적 수준(사용자 플로우, 정보 흐름, 모드 비교, 구성 요소별 책임 등)에 집중하십시오. 구현 아키텍처, 데이터 스키마, 코드 구조는 다루지 않습니다.
- 산문이 우선합니다: 시각적 도구와 주변 산문의 내용이 충돌할 경우 산문의 내용을 따릅니다.

시각적 도구를 생성한 후에는 그것이 산문으로 작성된 요구사항을 정확하게 반영하는지(순서의 정확성, 누락된 분기 유무, 병합된 단계 유무 등) 확인하십시오. 검증할 코드가 없는 다이어그램은 코드 기반 다이어그램보다 부정확할 위험이 더 큽니다.
