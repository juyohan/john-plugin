# 코딩 스타일

## 불변성 (치명적 - CRITICAL)

항상 새로운 객체를 생성하고, 기존 객체를 절대 변경(mutate)하지 마십시오:

```
// 의사코드
오답: modify(original, field, value) → 원래 객체를 직접 변경
정답: update(original, field, value) → 변경 사항이 적용된 새로운 복사본 반환
```

사유: 불변 데이터는 숨겨진 부작용을 방지하고, 디버깅을 쉽게 만들며, 안전한 동시성 처리를 가능하게 합니다.

## 핵심 원칙

### KISS (Keep It Simple)

- 실제로 작동하는 가장 단순한 솔루션을 선호합니다.
- 조급한 최적화(premature optimization)를 피합니다.
- 기교보다는 명확성을 위해 최적화합니다.

### DRY (Don't Repeat Yourself)

- 반복되는 로직은 공유 함수나 유틸리티로 추출합니다.
- 복사-붙여넣기로 인한 구현 차이(drift)를 방지합니다.
- 추측에 의한 반복이 아닌, 실제로 반복이 발생할 때 추상화를 도입합니다.

### YAGNI (You Aren't Gonna Need It)

- 필요하기 전에는 기능을 만들거나 추상화하지 마십시오.
- 추측에 근거한 범용성 확보를 피합니다.
- 단순하게 시작하고, 실제 요구가 발생했을 때 리팩토링합니다.

## 파일 조직화

거대한 파일 몇 개보다 작은 파일 여러 개를 지향합니다:
- 높은 응집도, 낮은 결합도 유지
- 일반적으로 200~400라인, 최대 800라인 유지
- 거대 모듈에서 유틸리티 추출
- 타입이 아닌 기능/도메인별로 조직화

## 에러 처리

항상 에러를 포괄적으로 처리하십시오:
- 모든 레벨에서 에러를 명시적으로 처리합니다.
- UI 관련 코드에서는 사용자 친화적인 에러 메시지를 제공합니다.
- 서버 측에서는 상세한 에러 컨텍스트를 로깅합니다.
- 에러를 아무 조치 없이 무시하지 마십시오.

## 입력 검증

항상 시스템 경계에서 검증을 수행하십시오:
- 처리 전 모든 사용자 입력을 검증합니다.
- 가능한 경우 스키마 기반 검증을 사용합니다.
- 명확한 에러 메시지와 함께 조기에 실패(Fail fast)하도록 합니다.
- 외부 데이터(API 응답, 사용자 입력, 파일 내용 등)를 절대 신뢰하지 마십시오.

## 명명 규칙 (Naming Conventions)

- 변수 및 함수: 의미 있는 이름을 가진 `camelCase`
- 불리언(Boolean): `is`, `has`, `should`, `can` 접두사 선호
- 인터페이스, 타입, 컴포넌트: `PascalCase`
- 상수: `UPPER_SNAKE_CASE`
- 커스텀 훅: `use` 접두사를 가진 `camelCase`

## 피해야 할 코드 스멜 (Code Smells)

### 깊은 중첩

로직이 쌓이기 시작하면 중첩된 조건문보다 조기 반환(early returns)을 선호하십시오.

### 매직 넘버 (Magic Numbers)

의미 있는 임계값, 지연 시간, 제한 사항에는 명명된 상수를 사용하십시오.

### 거대 함수

큰 함수는 명확한 책임을 가진 집중된 조각들로 나누십시오.

## 코드 품질 체크리스트

작업을 완료로 표시하기 전 다음을 확인하십시오:
- [ ] 코드가 가독성이 좋고 이름이 적절한가?
- [ ] 함수가 작은가? (<50라인)
- [ ] 파일의 응집도가 높은가? (<800라인)
- [ ] 깊은 중첩이 없는가? (>4단계 지양)
- [ ] 적절한 에러 처리가 되어 있는가?
- [ ] 하드코딩된 값은 없는가? (상수나 설정값 사용)
- [ ] 객체 변경이 없는가? (불변 패턴 사용)
