# 심층 방어 (Defense-in-Depth)

잘못된 상태가 취약한 코드 경로에 도달하여 버그가 발생한 경우, 단 한 개의 계층만 수정하면 다른 코드 경로, 리팩토링 또는 모의 객체(mock)에 의해 동일한 버그가 다시 도입될 여지가 남습니다. 심층 방어는 여러 계층에서 검증을 수행함으로써 버그가 구조적으로 다시 발생하기 어렵게 만듭니다.

모든 버그에 이 방식이 필요한 것은 아닙니다. 다음과 같은 경우에 사용하십시오:

- 근본 원인 패턴이 3개 이상의 다른 파일에 존재하는 경우 (수정 시그니처를 grep 하십시오)
- 운영 환경에서 발생했다면 치명적이었을 버그인 경우
- 호출자와 관계없이 해당 작업 자체가 위험한 경우 (파괴적인 부수 효과, 보안 민감, 되돌릴 수 없음)

재발 가능성이 없는 일회성 로직 에러인 경우에는 건너뛰십시오.

## 4개 계층

해당되는 계층을 선택하십시오. 모든 버그에 4개 계층이 다 필요한 것은 아닙니다.

| 계층 | 목적 | 적용 시점 | 예시 |
|-------|---------|------------|---------|
| 1. 입구 검증 (Entry validation) | API 경계에서 명백히 잘못된 입력을 거부 | 거부되었어야 할 잘못된 데이터를 호출자가 전달하여 버그가 발생한 경우 | 다운스트림 코드가 건드리기 전, `workingDirectory`가 비어 있거나 존재하지 않으면 에러 발생 |
| 2. 불변성 / 비즈니스 로직 확인 | 데이터가 이 작업에 적합한지 강제 | 입구 검증으로 표현할 수 없는 전제 조건이 작업에 있는 경우 | 비밀번호 초기화를 실행하기 전 `user.state === 'verified'`임을 어설션(assert) |
| 3. 환경 가드 (Environment guard) | 위험한 작업이 부적절한 컨텍스트에서 실행되는 것을 거부 | 잘못된 환경에서 실행될 경우 작업이 치명적일 수 있는 경우 | 테스트 환경(`NODE_ENV === 'test'`)에서 OS 임시 디렉토리 외부의 `git init`을 거부 |
| 4. 진단 브레드크럼 (Diagnostic breadcrumb) | 위험한 작업 직전에 포렌식 컨텍스트를 캡처 | 다른 계층이 우회될 수 있으므로 향후의 실패에 대한 증거가 필요한 경우 | `git init` 직전에 `{ directory, cwd, env, stack }`을 로그로 남김 |

## 패턴 적용하기

1. 잘못된 값의 기점부터 이를 전달한 모든 함수를 거쳐 데이터 흐름을 추적하십시오.
2. 체크포인트를 설정하십시오: 이 지점들 중 어느 곳에서 잘못된 값을 더 일찍 거부할 수 있었습니까?
3. 적절한 계층에 가드를 추가하십시오. 각 가드는 다른 계층의 확인을 중복하지 않고, 해당 계층이 책임지는 내용만 정확히 검증하도록 가능한 좁게 설정해야 합니다.
4. 각 가드를 독립적으로 테스트하십시오: 계층 1을 우회하는 케이스를 구성하고 계층 2가 여전히 이를 잡아내는지 확인하십시오.

## 흔한 실수

- **모든 계층에서 동일한 확인을 중복함.** 각 계층은 서로 다른 클래스의 실패를 잡아내야 합니다. 계층 2가 계층 1을 단순히 반복한다면, 두 번째 확인은 소음에 불과합니다.
- **정당화할 버그도 없이 추측만으로 가드를 추가함.** 심층 방어는 관찰된 실패 모드에 대한 대응이지, 일반적인 코드 위생 관리 연습이 아닙니다.
- **계층 4(진단 브레드크럼)를 누락함.** 계층 1-3이 결국 우회되었을 때(언젠가는 우회될 것입니다), 브레드크럼은 다음 버그를 디버깅할 수 있게 해주는 핵심 열쇠가 됩니다.
