# 조사 기법 (Investigation Techniques)

표준적인 코드 추적만으로는 부족할 때 사용하는 심화 조사 기법입니다. 버그가 안정적으로 재현되지 않거나, 타이밍이나 동시성 이슈를 포함하거나, 프레임워크 전용 추적이 필요한 경우 이 내용을 로드하십시오.

---

## 근본 원인 추적 (Root-Cause Tracing)

버그가 호출 스택 깊은 곳에서 나타날 때, 본능적으로 에러가 발생한 지점을 고치려 하게 됩니다. 그것은 증상을 치료하는 것입니다. 대신 호출 체인을 거꾸로 추적하여 잘못된 상태가 시작된 지점을 찾으십시오.

**역방향 추적:**

- 에러 발생 지점에서 시작합니다.
- 각 수준에서 자문하십시오: 이 값은 어디서 왔는가? 누가 이 함수를 호출했는가? 어떤 상태가 전달되었는가?
- 유효했던 상태가 처음으로 유효하지 않게 된 지점을 찾을 때까지 업스트림으로 계속 올라갑니다 — 그곳이 근본 원인입니다.

**활용 예시:**

```
증상: API가 "Cannot read property 'email' of undefined"와 함께 500 에러 반환
크래시 지점: NotificationService의 sendWelcomeEmail(user.email)
누가 호출했나? 사용자 레코드 저장 후 UserController.create()
무엇이 전달되었나? user = await UserRepo.create(params) — 하지만 create()는 중복 키 에러 시 undefined를 반환함
원래 원인: UserRepo.create()가 중복 키 에러를 조용히 삼키고 throw 대신 undefined를 반환함
```

수정은 에러가 나타난 곳(NotificationService)이 아니라 기점(UserRepo.create가 중복 키 에러 시 throw 하도록 수정)에 적용되어야 합니다.

**수동 추적이 막힐 때**는 인스트루멘테이션(instrumentation)을 추가하십시오:

```javascript
// 문제가 되는 작업 직전
const stack = new Error().stack;
console.error('DEBUG [operation]:', { value, cwd: process.cwd(), stack });
```

테스트에서는 로거 출력이 억제될 수 있으므로 `console.error()`를 사용하십시오. 작업이 실패한 후가 아니라 위험한 작업을 수행하기 **전에** 로그를 남기십시오.

## 다중 컴포넌트 경계 인스트루멘테이션 (Multi-Component Boundary Instrumentation)

근본 원인 추적은 하나의 호출 체인을 따라갑니다. 버그가 여러 서브시스템을 가로지를 때 — CI → 빌드 → 사이닝, API → 서비스 → 데이터베이스, 프론트엔드 → API → 백그라운드 워커 등 — 실패 지점을 단일 체인으로 국소화하기 어렵습니다. 대신 한 번의 실행으로 모든 컴포넌트 경계에 인스트루멘테이션을 설치하고, 각 경계에 들어가고 나가는 데이터를 캡처하여 증거가 실패 계층을 지목하게 하십시오.

**형태:**

1. 트리거부터 관찰된 증상까지 데이터가 통과하는 컴포넌트 경계 목록을 작성합니다.
2. 각 경계에서 들어오는 값과 나가는 값을 로그로 남깁니다 — 값, 관련 환경 변수, 경계를 식별하는 짧은 태그를 포함합니다.
3. 시나리오를 한 번 실행합니다.
4. 로그를 선형적으로 읽으며 각 "나가는(exits)" 값과 다음 "들어오는(enters)" 값을 비교합니다.
5. 데이터가 처음으로 예상과 일치하지 않게 되는 경계가 바로 실패한 계층입니다.

**활용 예시 (CI에서의 앱 사이닝):**

```bash
# 계층 1: workflow 환경
echo "=== workflow env ==="
echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

# 계층 2: 빌드 스크립트 환경
echo "=== build script env ==="
echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

# 계층 3: 사이닝 단계 키체인 상태
echo "=== keychain ==="
security list-keychains
security find-identity -v

# 계층 4: 실제 사이닝 호출
codesign --sign "$IDENTITY" --verbose=4 "$APP"
```

단 한 번의 실행으로 로그는 어떤 계층이 값을 유실했는지 정확히 보여줍니다 — secrets → workflow ✓, workflow → build ✗ → 조사의 초점을 사이닝 자체가 아니라 workflow에서 빌드 스크립트로의 상속 문제에 맞춥니다.

**이 기법이 역방향 추적보다 나은 경우:** 증상이 트리거로부터 멀리 떨어져 있을 때(여러 컴포넌트 간격), 컴포넌트의 소유 시스템이 다를 때(CI vs 앱 코드), "호출 스택"이 문자 그대로가 아닌 개념적일 때(메시지 버스, HTTP, 프로세스 경계). 실패 계층이 식별되면 그 내부에서 다시 역방향 추적을 적용할 수 있습니다.

## 회귀 버그를 위한 Git Bisect

버그가 회귀(regression, "이전에는 작동했음")인 경우, 이진 탐색을 통해 문제를 일으킨 커밋을 찾으십시오:

```bash
git bisect start
git bisect bad                    # 현재 커밋은 고장남
git bisect good <known-good-ref> # 작동했던 과거의 커밋
# git bisect가 중간 커밋을 체크아웃함 — 테스트 수행
# 결과에 따라 good 또는 bad 표시, 문제의 커밋을 찾을 때까지 반복
git bisect reset                  # 완료 후 원래 브랜치로 복귀
```

테스트 스크립트를 이용한 자동화된 bisection:

```bash
git bisect start HEAD <known-good-ref>
git bisect run <test-command>
```

테스트 명령은 정상일 때 0, 비정상일 때 0이 아닌 값을 반환해야 합니다.

## 간헐적인 버그 기법 (Intermittent Bug Techniques)

2-3번의 시도 후에도 버그가 안정적으로 재현되지 않을 때:

**로깅 트랩 (Logging traps).** 의심되는 실패 지점에 타겟팅된 로깅을 추가하고 시나리오를 반복해서 실행합니다. 성공한 실행과 실패한 실행 사이에 차이가 있는 상태를 캡처합니다.

**통계적 재현 (Statistical reproduction).** 루프를 이용해 실패 시나리오를 실행하여 재현율을 파악합니다:

```bash
for i in $(seq 1 20); do echo "Run $i:"; <test-command> && echo "PASS" || echo "FAIL"; done
```

5%의 재현율은 버그가 존재함을 확증하며, 타이밍이나 데이터 민감도 문제임을 암시합니다.

**환경 격리.** 변수를 체계적으로 제거하십시오:
- 동일한 테스트, 다른 머신?
- 동일한 테스트, 다른 데이터 시드(seed)?
- 동일한 테스트, 직렬 vs 병렬 실행?
- 동일한 테스트, 네트워크 연결 유무?

**데이터 의존적 트리거.** 특정 데이터에서만 버그가 나타난다면 트리거 조건을 식별하십시오:
- 실패하는 입력의 어떤 점이 고유한가?
- 입력의 크기, 인코딩 또는 에지 값이 중요한가?
- 데이터의 순서가 유의미한가 (정렬됨 vs 무작위)?

**테스트 순서 오염 (Test-order pollution).** 개별 테스트는 단독으로 통과하지만 전체 스위트 실행 시 실패한다면, 테스트 간에 상태가 누출되고 있는 것입니다:

- 실패하는 테스트만 단독 실행 — 통과한다면 오염이 확정됨
- 실패하는 테스트가 포함된 파일만 실행 — 오염이 동일 파일 내인지 파일 간인지 좁힘
- 무작위 테스트 순서로 스위트 실행 (대부분의 러너가 시드 플래그 지원) — 실행마다 실패하는 테스트의 이웃이 달라진다면 글로벌 상태 변경을 의미함
- 이전 테스트들 이분 탐색(Bisect): 이전 테스트들의 앞쪽 절반만 포함하여 실행, 그 다음 뒤쪽 절반, 이런 식으로 범위를 좁힘

격리 후의 흔한 범인들: 모듈 레벨의 상태, 정리(tear down)되지 않은 모의 객체(mock), 삭제되지 않은 임시 파일, 롤백되지 않은 데이터베이스 행, 변경 후 복구되지 않은 환경 변수.

## 재현 시나리오 최소화 (Repro Minimization)

버그가 안정적으로 재현되더라도, 재현 시나리오가 너무 큰 경우가 많습니다 — 500라인의 통합 테스트, 거대한 페이로드, 긴 폼 입력 시퀀스 등. 재현 시나리오를 작게 만들면 이후의 모든 조사 단계가 빨라지고 실제 트리거를 국소화할 수 있습니다.

**델타 디버깅 (수동):**

1. 재현 시나리오를 절반으로 자릅니다.
2. 여전히 실패합니까? 그렇다면 나머지 절반은 버리고 남은 부분에서 반복합니다. 아니라면 실패 동작이 방금 자른 부분에 의존하는 것이므로, 다시 돌려놓고 다른 절반을 자릅니다.
3. 실패를 잃지 않으면서 더 이상 줄일 수 없을 때까지 계속합니다.

**입력 페이로드의 경우:**

- 버그가 지속되는지 확인하면서 필드를 하나씩(또는 절반씩) 제거합니다.
- 버그를 트리거하는 최소 길이에 도달할 때까지 문자열 값을 줄입니다.
- 복잡한 중첩 구조를 재현 가능한 가장 작은 모양으로 교체합니다.

**테스트 시퀀스의 경우:**

- 실패하는 어설션에 영향을 주지 않는 것으로 보이는 설정 단계들을 제거합니다.
- 실제로 무엇이 실행되는지 확인하기 위해 헬퍼 함수를 테스트 내부에 인라인화합니다.
- 어떤 것이 실패하는지, 어떤 상태에서 실패하는지 격리하기 위해 다른 어설션들을 제거합니다.

최소화된 재현 시나리오는 종종 근본 원인을 직접적으로 드러냅니다 — "이 버그는 문자열에 탭 문자가 포함될 때만 발생한다"는 신호는 "이 500라인 통합 테스트에서 버그가 발생한다"는 신호보다 훨씬 강력합니다.

## 프레임워크 전용 디버깅

### Rails
- 콜백 확인: `before_save`, `after_commit`, `around_action` — 이들은 암시적으로 실행되며 상태를 변경할 수 있습니다.
- 미들웨어 체인 확인: `rake middleware`로 전체 스택을 확인합니다.
- Active Record 쿼리 생성 확인: 모든 관계(relation)에 대해 `.to_sql`을 실행해 봅니다.
- 요청 추적을 위해 태그된 로깅과 함께 `Rails.logger.debug`를 사용하십시오.

### Node.js
- 비동기 stack traces: 전체 비동기 호출 체인을 위해 `--async-stack-traces` 플래그와 함께 실행하십시오.
- 처리되지 않은 거부(Unhandled rejections): 프로미스에 누락된 `.catch()`나 `await`가 있는지 확인하십시오.
- 이벤트 루프 지연: 의심되는 작업 전후에 `process.hrtime()`을 측정하십시오.
- 메모리 누수: `--inspect` 플래그 + Chrome DevTools heap snapshots을 활용하십시오.

### Python
- Traceback enrichment: except 블록에서 `traceback.print_exc()` 사용
- 대화형 디버깅을 위한 `pdb.set_trace()` 또는 `breakpoint()`
- 실행 추적을 위한 `sys.settrace()`
- 상세 출력을 위한 `logging.basicConfig(level=logging.DEBUG)`

## 단계별 디버거 vs 인스트루멘테이션 (Stepping Debugger vs Instrumentation)

프린트 디버깅(Print-debugging)은 가장 먼저 손이 가는 기본 도구입니다 — 추가하기 빠르고 많은 경우에 효과적입니다. 하지만 대화형 단계별 디버거(stepping debugger)가 근본 원인에 훨씬 빨리 도달하게 해주는 경우들이 있습니다. 경험 법칙은 다음과 같습니다:

- **단계별 디버거를 사용해야 할 때:** 실패 코드 경로가 국소적이고(특정 함수나 짧은 호출 체인), 버그 재현이 안정적이며, 특정 시점의 정밀한 상태가 필요할 때 — 여러 지역 변수의 값, 구조의 정확한 모양, 루프를 거치며 변하는 상태의 추이 등. 한 번의 중단(break)으로 모든 것을 조사할 수 있습니다.
- **인스트루멘테이션을 사용해야 할 때:** 버그가 간헐적이고, 많은 호출이나 분산된 컴포넌트에 걸쳐 있거나, 실행을 중단하는 것이 시스템에 지장을 주는 컨텍스트일 때 (운영 환경, 타이밍이 중요한 동시성 코드, 장시간 실행되는 프로세스). 인스트루멘테이션은 시간과 환경에 걸쳐 분산된 동작을 포착합니다.

혼합해서 사용하는 경우가 많습니다: 먼저 인스트루멘테이션으로 지점을 국소화한 후, 해당 지점에 디버거를 연결합니다.

**언어별 진입점:**

| 언어 | 대화형 중단점 | 실행 중인 프로세스에 연결 |
|----------|------------------------|---------------------------|
| Python | 코드 내 `breakpoint()`, 또는 `python -m pdb script.py` | `python -m pdb -p <pid>` (Python 3.14+ 전용); 이전 버전에서는 `rpdb` / `remote-pdb`를 사용하고 트리거 후 연결 |
| Node.js | 코드 내 `debugger;` + `node --inspect-brk` 실행 후 Chrome DevTools나 VS Code로 연결 | `kill -SIGUSR1 <pid>`로 실행 중인 프로세스에서 인스펙터 활성화(Linux/macOS), 그 후 9229 기본 포트로 연결 |
| Ruby | `binding.irb` (표준 라이브러리), `binding.pry` (pry 젬), `debugger` (debug 젬), `rdbg` | `debug` 젬이 로드된 상태에서 `rdbg --attach <pid>` |
| Go | `dlv debug` 또는 `dlv test` 실행 후 `break`, `continue`, `print` | `dlv attach <pid>` |
| Rust / C / C++ | `lldb target/debug/binary` 또는 `gdb binary` 실행 후 `break`, `run`, `print` | `lldb -p <pid>` / `gdb -p <pid>` |
| Browser JS | 코드 내 `debugger;`, 또는 DevTools Sources → 중단점 설정 | DevTools가 페이지에 자동으로 연결됨 |

테스트 실행의 경우 대부분의 테스트 러너가 위의 도구들과 연동됩니다 — 예: `node --inspect-brk $(which jest)`, `pytest --pdb`, `rspec` + `binding.pry`, `dlv test`. 사후에 연결을 시도하기보다 러너의 연동 기능을 우선적으로 사용하십시오.

## 레이스 컨디션 조사 (Race Condition Investigation)

타이밍이나 동시성 이슈가 의심될 때:

**타이밍 격리.** 의심되는 지점에 의도적인 지연을 추가하여 레이스 윈도우를 넓히고 재현 가능하게 만듭니다:

```javascript
// 레이스를 노출하기 위해 느린 작업 시뮬레이션
await new Promise(r => setTimeout(r, 100));
```

**공유 가변 상태 (Shared mutable state).** 동기화 없이 여러 스레드나 프로세스가 접근하는 변수, 캐시, 데이터베이스 행을 찾으십시오. 흔한 패턴:
- 글로벌 또는 모듈 레벨의 가변 상태
- 락(lock) 없는 캐시 읽기
- 낙관적 락(optimistic locking) 없이 읽은 후 업데이트되는 데이터베이스 행

**비동기 순서.** 작업이 보장되지 않은 특정 실행 순서를 가정하고 있는지 확인하십시오:
- 의존 관계가 있는 작업들이 포함된 `Promise.all`
- 발행 순서를 가정하는 이벤트 핸들러
- 읽기 일관성을 가정하는 데이터베이스 쓰기

**임의의 지연 대신 조건 기반 대기.** 불안정한(flaky) 테스트는 종종 작업 시간을 추측하는 `setTimeout`/`sleep` 호출을 기반으로 작성됩니다. 이들은 빠른 머신에서는 통과하지만 부하 상황이나 CI에서는 실패합니다. 추측성 지연을 타임아웃이 설정된 조건 폴링(polling)으로 교체하십시오:

```typescript
// 이전: 부하 상황에서 레이스 발생
await new Promise(r => setTimeout(r, 50));
expect(getResult()).toBeDefined();

// 이후: 조건을 기다림
await waitFor(() => getResult() !== undefined, 'result available', 5000);
expect(getResult()).toBeDefined();
```

임의의 지연은 실제 타이밍 동작(디바운스 간격, 스로틀 윈도우)을 테스트할 때만 적절하며, 이 경우 왜 그 특정 시간이 필요한지 주석을 남기십시오.

## 하이젠버그와 관찰자 효과 (Heisenbugs and the Observer Effect)

`console.log`를 추가하거나 디버거를 연결하거나 인스트루멘테이션을 삽입했을 때 버그가 사라진다면, 관찰 행위가 시스템의 동작을 바꾸고 있는 것입니다. 이 현상 자체가 진단 정보입니다 — "해결됨"이라고 결론짓지 마십시오. 버그는 여전히 존재하며, 당신의 관찰 도구가 이를 시야 밖으로 밀어냈을 뿐입니다.

**사라짐 현상이 알려주는 것:**

- **타이밍 민감:** 인스트루멘테이션이 코드를 충분히 느리게 만들어 레이스 컨디션이 더 이상 발생하지 않게 됨. 일반적인 로직이 아니라 동시성, 비동기 순서, 공유 가변 상태를 조사하십시오.
- **가비지 컬렉션(GC) 민감:** 할당된 메모리를 로깅하는 행위가 GC를 트리거하여 증상을 숨김. 메모리 압박, 파이널라이저(finalizer), 객체 수명 주기를 살펴보십시오.
- **최적화 의존적:** 인스트루멘테이션이 잘못된 결과를 내보내던 컴파일러/JIT 최적화를 방지함. 드물지만 실제로 발생합니다 (특히 C/C++/Rust 릴리스 빌드).
- **버퍼링 의존적:** 로그 플러싱(flushing)이 I/O 순서를 바꿈. 종종 다른 곳에 플러싱되지 않은 쓰기 작업이 있음을 나타냅니다.
- **비동기 순서 민감:** 로그 I/O가 마이크로태스크 경계를 도입하여 후속 작업의 순서를 재배치함. 동기적 순서에 암시적으로 의존하는 코드를 찾으십시오.

**교란 없이 조사하는 방법:**

- 비차단형(Non-blocking) 인스트루멘테이션: 메모리 내 링 버퍼(ring buffer)에 기록하고 실패 관찰 후에만 덤프
- 추적 대신 샘플링 프로파일러: 경로에 코드를 주입하지 않고 실행 중인 내용을 외부에서 관찰
- 플랫폼 레벨 인스트루멘테이션: 코드 변경이 필요 없는 `strace`, `dtrace`, eBPF, 플랫폼 프로파일러
- 사후(Post-mortem) 증거: 실행 중 관찰하지 않고 코어 덤프, 힙 스냅샷, 실패 후 캡처된 상태 활용

정의 규칙: 버그가 관찰에 민감하다면, 수정 사항은 관찰 도구를 다시 도입했을 때도 살아남아야 합니다. 인스트루멘테이션이 있을 때만 작동하는 수정은 그 자체가 하이젠버그입니다.

## 브라우저 디버깅

`agent-browser`나 유사 도구로 UI 버그를 조사할 때:

```bash
# 영향받는 페이지 열기
agent-browser open http://localhost:${PORT:-3000}/affected/route

# 현재 상태 캡처
agent-browser snapshot -i

# 페이지와 상호작용
agent-browser click @ref          # 요소 클릭
agent-browser fill @ref "text"    # 폼 필드 입력
agent-browser snapshot -i         # 상호작용 후 상태 캡처

# 시각적 증거 저장
agent-browser screenshot bug-evidence.png
```

**포트 감지:** 프로젝트 지침 파일(`AGENTS.md`, `CLAUDE.md`)에서 포트 참조를 확인하고, 그 다음 `package.json`의 dev 스크립트, `.env` 파일을 확인하십시오. 기본값은 `3000`입니다.

**콘솔 에러:** 브라우저 콘솔 출력에서 JavaScript 에러, 네트워크 요청 실패, CORS 이슈를 확인하십시오. 이들은 코드 추적 전에 UI 버그의 근본 원인을 밝혀주는 경우가 많습니다.

**네트워크 탭:** 실패한 API 요청, 예상치 못한 응답 코드, 누락된 CORS 헤더를 확인하십시오. 백엔드의 422 또는 500 응답은 조사의 범위를 즉시 좁혀줍니다.

## 시스템 전반의 증거 수집 (Evidence Harvesting Across Systems)

버그가 실제 환경 — 운영, 스테이징, 멀티 서비스 설정 — 에 걸쳐 있을 때, 가장 풍부한 증거는 대개 로그, 트레이스, 에러 트래커 페이로드에 이미 존재합니다. 가능하면 처음부터 재현하기보다 이를 활용하십시오.

**단일 요청을 처음부터 끝까지 추적하십시오.** 구체적으로 실패한 요청 하나(에러 트래커의 정확한 타임스탬프, 사용자 ID 또는 이벤트 ID)를 선택하십시오. 그 후:

- 상관관계 ID(correlation ID), 요청 ID, 트레이스 ID, 사용자 ID 등 해당 식별자로 모든 관련 로그 소스를 검색합니다.
- 타임라인을 조립합니다: edge → API → 서비스 → 데이터베이스 → 다운스트림 호출 → 응답.
- 타임라인에 공백(로그 누락)이나 모순(타임스탬프 순서 뒤바뀜, ID 전파 안 됨)이 있는 지점을 확인합니다.

추적된 요청 하나가 수십 번의 재현 시도보다 빠르게 근본 원인을 밝혀주는 경우가 많습니다.

**상관관계 ID (Correlation IDs).** 대부분의 웹 프레임워크는 요청 ID를 자동으로 부여하거나 헤더(`X-Request-ID`, `traceparent`)를 통해 수락합니다. 프로젝트에 이것이 있다면 모든 로그 라인과 다운스트림 호출에 포함되어야 합니다. 이것이 누락되거나 전파되지 않는다면 그 자체가 발견 사항입니다 — 전파 공백은 에이전트가 타임라인을 조립할 수 없음을 의미하며, 다음 장애를 조사할 당번 개발자도 마찬가지일 것입니다.

**타임스탬프 삼각측량 (Timestamp triangulation).** 실패한 작업에 공유 ID가 없는 경우 타임스탬프가 폴백 수단입니다. 모든 로그 쿼리를 관찰된 실패 시점 주변의 좁은 창으로 제한하고, 순서대로 첫 번째 이상 징후를 찾으십시오. 서비스 간 클록 스큐(clock skew)를 주의하십시오 — 두 호스트 사이의 30초 차이는 증거의 순서를 뒤섞어 삼각측량을 방해합니다.

**에러 트래커 페이로드.** Sentry, Bugsnag, Honeybadger, AppSignal 등은 실패 순간의 스택 트레이스, 브레드크럼, 사용자 컨텍스트, 요청 상태, 릴리스 메타데이터를 캡처합니다. 코드를 추적하기 전 전체 페이로드를 읽으십시오 — 정확한 file:line, 변수 상태, 에러에 이르는 브레드크럼이 포함되어 있는 경우가 많습니다. 그룹화 규칙에 의해 빈도나 변형 정보가 가려질 수 있으므로, 대표 사례만 보지 말고 모든 인스턴스를 확장해서 확인하십시오.

**APM / 분산 트레이스.** 프로젝트에 Datadog APM, Honeycomb, New Relic 또는 OpenTelemetry 수집기가 있다면, 트레이스 뷰는 서비스 간 호출 트리와 타이밍을 보여줍니다. 다음을 찾으십시오: 예상보다 긴 스팬(블로킹 또는 느린 의존성), 체인 중간의 실패한 스팬, 존재해야 하지만 없는 스팬 (인스트루멘테이션 누락도 버그를 가릴 수 있습니다).

**조사 전 보존.** 에러 트래커와 로그 시스템은 보관 기간이 있습니다. 긴 조사를 시작하기 전, 핵심 증거(이벤트 ID, 트레이스 ID, 전체 스택 트레이스, 브레드크럼)를 내보내거나 스냅샷을 찍어 세션 도중에 사라지지 않게 하십시오.

## 시스템 경계 확인 (System Boundary Checks)

많은 버그가 애플리케이션과 실행 시스템(네트워크, 데이터베이스, 파일 시스템, OS) 사이의 경계에 존재합니다. 이 경계들을 빠르게 확인하면 깊은 코드 추적 전에 많은 의심 카테고리를 제거할 수 있습니다.

**네트워크.**

- DNS 확인: `dig <host>`, `nslookup <host>`, `host <host>` — 이름이 해당 호스트에서 예상한 대로 해석됩니까?
- 도달 가능성: `curl -v https://host/path` — 전체 헤더, 리다이렉트, TLS 에러 확인
- 상태 코드 및 헤더: 4xx/5xx 응답, 예상치 못한 리다이렉트, 누락된 CORS 헤더, 예상 밖의 콘텐츠 인코딩 확인
- 연결 상태: `ss -tan` / `netstat -an` / `lsof -i` — 열린 연결, 리스닝 포트, TIME_WAIT 또는 CLOSE_WAIT 상태의 연결
- TLS: `openssl s_client -connect host:443` — 인증서 체인, 만료 여부, SNI 불일치 확인

**데이터베이스.**

- 쿼리 플랜: 의심되는 쿼리에 대해 `EXPLAIN` / `EXPLAIN ANALYZE` 수행 — 예상한 인덱스를 사용합니까, 아니면 큰 테이블을 스캔합니까?
- 느린 쿼리 로그 / 최근 쿼리: 대부분의 DB는 상위 N개의 느린 쿼리를 보여줍니다 — 실패하는 쿼리가 거기 나타나는 경우가 많습니다.
- 락 및 트랜잭션: 락/트랜잭션 테이블(`pg_locks`, `information_schema.innodb_trx`, `sys.dm_tran_locks`) 검사 — 작업이 오래 유지된 락을 기다리고 있습니까?
- 커넥션 풀: 앱이 풀을 소진하고 있습니까? 커넥션 누수가 있습니까?
- 복제 지연 (읽기 복제본 사용 시): 쓰기 직후의 읽기가 아직 따라잡지 못한 복제본을 참조할 수 있습니다.

**파일 시스템.**

- 존재 여부 및 권한: `ls -la <path>` — 파일이 존재하며 실행 사용자가 읽기/쓰기가 가능합니까?
- 대소문자 구분: Linux에서만 (macOS가 아닌) 나타나는 버그는 대소문자 불일치인 경우가 많습니다.
- 열린 핸들: `lsof <path>` 또는 `lsof -p <pid>` — 무언가 여전히 파일을 잡고 있어 쓰기나 삭제(unlink)를 방해합니까?
- 디스크 공간: `df -h` — 공간 부족 에러는 때로 다른 곳에서 모호한 쓰기 실패로 나타납니다.
- 파일 워칭 / inotify 제한: EMFILE이나 "too many open files" 에러는 코드 누수가 아니라 inotify/FD 제한 문제일 수 있습니다.
- 경로 구분자 및 인코딩: 유닉스 코드의 윈도우 스타일 경로, 또는 비 UTF-8 로케일에서의 UTF-8 경로 문제.

**프로세스 및 시그널.** 프로세스가 실제로 당신이 생각하는 버전인지 확인하십시오 (`ps aux | grep` 후 pid와 빌드 시간 교차 확인). 좀비 프로세스, 고립된 워커, 이전 코드로 재시작된 프로세스 등은 모두 코드 버그로 위장합니다.

## 버그 클래스 패턴 체크리스트 (Bug-Class Pattern Checklist)

깊은 추적 전에 이 체크리스트를 훑어보십시오. 많은 버그는 인식 가능한 클래스에 속하며, 클래스는 어디를 먼저 봐야 할지 알려줍니다. 관찰된 증상이 다음 패턴 중 하나와 일치하는지 확인하십시오:

- **시간 및 시간대:** 자정 무렵의 off-by-hours 에러, 서머타임(DST) 전환기 전용 실패, 초(epoch)/밀리초 혼동, naive vs timezone-aware datetime 혼용, UTC vs 로컬 오판
- **인코딩 및 로케일:** 출력물의 글자 깨짐(mojibake), 바이트 vs 문자 길이 off-by-one, 파서를 깨뜨리는 파일 시작 부분의 BOM, 누락된 비 ASCII 문자, 로케일 민감 비교로 인한 일관성 없는 결과
- **부동 소수점 정밀도:** 같아야 하지만 같지 않은 비교, 계산을 통해 전파되어 다운스트림 결과를 오염시키는 NaN, 정밀도를 잃는 매우 크거나 작은 숫자
- **정수 오버플로 / 언더플로:** 범위가 제한된 정수 타입의 wraparound, 임의 정밀도 정수가 없는 언어에서의 `int32` 오버플로, 양수를 가정한 곳에서의 음수 값
- **Off-by-one 및 경계값:** 빈 컬렉션 에지 케이스, 첫 번째 또는 마지막 요소 누락, inclusive vs exclusive 범위 불일치, 펜스포스트(fencepost) 에러
- **캐시 실효성 (Cache staleness):** 변경 직후에는 정상이나 일정 시간 후 오작동, 재시작이나 캐시 플러시로 해결됨; HTTP 캐시, CDN 캐시, 앱 레벨 메모이제이션, 브라우저 서비스 워커 포함
- **권한 / 인증:** 특정 사용자에게만 발생, 인증 레이어 없는 개발 환경에선 정상이나 운영에선 실패, 슈퍼유저로는 정상이나 실제 운영 계정으로는 실패
- **의존성 또는 버전 드리프트:** 특정 머신에서만 실패, 매니페스트와 일치하지 않는 lockfile, 동작이 변한 전이적(transitive) 의존성 업데이트, 다른 런타임 버전에서 빌드된 네이티브 모듈
- **경로 / 대소문자 민감도:** macOS에선 정상이나 Linux에선 실패(대소문자), Linux에선 정상이나 Windows에선 실패(경로 구분자, `CON`/`PRN` 같은 예약어)
- **동시성 / 순서:** 직렬 테스트에선 정상이나 병렬에선 실패; 무작위 실행 시 결과가 달라짐
- **오래된 빌드 아티팩트:** `dist/`, `.next/`, 컴파일된 `.pyc`, 생성된 코드, Docker 이미지 레이어 — 깨끗하게 다시 빌드한 후 재현되는지 확인
- **관찰자 효과 (하이젠버그):** 버그가 로깅, 디버거, 프로파일러 연결 시 사라짐 — 위의 하이젠버그 섹션 참조
- **TOCTOU (time-of-check vs time-of-use):** 확인 시점에는 정상이었으나 실제 사용 시점 이전에 상태가 변경됨
