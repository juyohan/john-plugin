# Diff 범위 규칙 (Diff Scope Rules)

이 규칙은 모든 리뷰어에게 적용됩니다. "리뷰할 코드"와 기존 컨텍스트(pre-existing context)를 구분하는 기준을 정의합니다.

## 범위 탐색 (Scope Discovery)

다음 우선순위에 따라 리뷰할 diff를 결정합니다:

1. **사용자 지정 범위.** 호출자가 `BASE:`, `FILES:`, 또는 `DIFF:` 마커를 전달한 경우 해당 범위를 그대로 사용합니다.
2. **작업 복사본 변경 사항.** 스테이징되지 않았거나 스테이징된 변경 사항(`git diff HEAD`가 비어 있지 않음)이 있는 경우 이를 리뷰합니다.
3. **베이스 브랜치 대비 푸시되지 않은 커밋.** 작업 복사본이 깨끗한 경우, `git diff $(git merge-base HEAD <base>)..HEAD`를 리뷰합니다. 여기서 `<base>`는 기본 브랜치(main 또는 master)입니다.

SKILL.md의 scope 단계에서 탐색을 처리하고 확인된 diff를 전달하므로, 직접 git 명령을 실행할 필요는 없습니다.

## 발견 사항 분류 티어 (Finding Classification Tiers)

보고하는 모든 발견 사항은 diff와의 관계에 따라 다음 세 가지 티어 중 하나로 분류됩니다:

### 기본 (직접 변경된 코드 - Primary)

diff에서 추가되거나 수정된 라인입니다. 리뷰의 주요 초점입니다. 이 라인들에 대해서는 완전한 신뢰도로 발견 사항을 보고하십시오.

### 보조 (직접적인 주변 코드 - Secondary)

변경되지 않았지만 변경된 라인과 동일한 함수, 메서드 또는 블록 내에 있는 코드입니다. 변경 사항으로 인해 주변 컨텍스트를 읽어야만 보이는 버그가 도입된 경우 보고하되, 이슈가 새로운 코드와 기존 코드 간의 상호작용에서 발생했음을 명시하십시오.

### 기존 코드 (이 diff와 무관함 - Pre-existing)

diff가 건드리지 않았고 상호작용도 하지 않는 변경되지 않은 코드의 이슈입니다. 출력에서 `"pre_existing": true`로 표시하십시오. 이들은 별도로 보고되며 리뷰 판정(verdict)에는 포함되지 않습니다.

**규칙:** 주변 파일이 포함되지 않은 동일한 diff에서도 똑같은 이슈를 지적할 것이라면 그것은 '기존 코드' 이슈입니다. diff로 인해 해당 이슈가 *새롭게 관련성*을 갖게 된 경우(예: 새로운 호출자가 기존의 버그가 있는 함수를 호출함)는 '보조' 이슈입니다.
