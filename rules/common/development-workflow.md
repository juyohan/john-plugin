# 개발 워크플로우

> 이 파일은 [common/git-workflow.md](./git-workflow.md)을 확장하여 Git 작업 전 단계의 전체 기능 개발 프로세스를 다룹니다.

기능 구현 워크플로우는 조사, 기획, TDD, 코드 리뷰를 거쳐 Git에 커밋하는 개발 파이프라인을 설명합니다.

## 기능 구현 워크플로우

0. **조사 및 재사용 (Research & Reuse)** _(새로운 구현 전 필수 사항)_
   - **GitHub 코드 검색 우선:** 새로운 코드를 작성하기 전에 `gh search repos` 및 `gh search code`를 실행하여 기존 구현 사례, 템플릿, 패턴을 먼저 찾으십시오.
   - **라이브러리 문서 확인:** 구현 전 Context7이나 주요 벤더 문서를 사용하여 API 동작, 패키지 사용법, 버전별 상세 사항을 확인하십시오.
   - **Exa는 앞의 두 단계가 불충분할 때만 사용:** GitHub 검색과 공식 문서 확인 후, 더 넓은 범위의 웹 조사나 탐색이 필요할 때만 Exa를 사용하십시오.
   - **패키지 레지스트리 확인:** 유틸리티 코드를 직접 작성하기 전에 npm, PyPI, crates.io 등의 레지스트리를 검색하십시오. 직접 만든 솔루션보다 검증된 라이브러리를 선호하십시오.
   - **조정 가능한 구현체 검색:** 문제의 80% 이상을 해결할 수 있고 포크(fork), 포팅(port), 또는 래핑(wrap)할 수 있는 오픈 소스 프로젝트를 찾으십시오.
   - 요구사항을 충족한다면 완전히 새로운 코드를 작성하기보다 검증된 접근 방식을 채택하거나 포팅하는 것을 선호하십시오.

1. **기획 우선 (Plan First)**
   - **planner** 에이전트를 사용하여 구현 계획을 수립하십시오.
   - 코딩 전에 기획 문서(PRD, 아키텍처, 시스템 설계, 기술 문서, 작업 목록 등)를 생성하십시오.
   - 의존성 및 리스크를 식별하십시오.
   - 단계를 나누어 구성하십시오.

2. **TDD 접근 방식**
   - **tdd-guide** 에이전트를 활용하십시오.
   - 테스트를 먼저 작성하십시오 (RED).
   - 테스트를 통과하도록 구현하십시오 (GREEN).
   - 리팩토링을 수행하십시오 (IMPROVE).
   - 커버리지 80% 이상을 확인하십시오.

3. **코드 리뷰**
   - 코드 작성 직후 **code-reviewer** 에이전트를 사용하십시오.
   - '치명적(CRITICAL)' 및 '높음(HIGH)' 이슈를 해결하십시오.
   - 가능한 경우 '중간(MEDIUM)' 이슈도 수정하십시오.

4. **커밋 및 푸시 (Commit & Push)**
   - 상세한 커밋 메시지를 작성하십시오.
   - Conventional Commits 형식을 따르십시오.
   - 커밋 메시지 형식 및 PR 프로세스는 [git-workflow.md](./git-workflow.md)를 참조하십시오.

5. **리뷰 전 점검 사항**
   - 모든 자동화된 검사(CI/CD) 통과 여부를 확인하십시오.
   - 모든 머지 충돌을 해결하십시오.
   - 브랜치가 대상 브랜치와 최신 상태를 유지하는지 확인하십시오.
   - 이러한 체크를 마친 후에만 리뷰를 요청하십시오.
