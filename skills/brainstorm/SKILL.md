# 브레인스토밍

<feature_description> #$ARGUMENTS </feature_description>

기능 설명이 비어 있으면 물어보십시오: "무엇을 탐색하고 싶으신가요? 기능, 문제 또는 개선 사항을 설명해 주세요."

`--add gemini` (또는 `--add gem`) 플래그가 있으면: gem 도구로 Gemini에게 검토를 요청하고 결과를 통합하십시오. 산출물 상단에 "Gemini와의 협업을 통해 검토 및 보완되었습니다."를 추가하십시오.

`--deep` 플래그가 있으면: `references/deep-product.md`를 읽고 해당 지침을 섹션 2, 4, 5, 7에 추가로 적용하십시오.

비소프트웨어 주제이면: 섹션 3(컨텍스트 스캔)과 4(Gap 압박 테스트)의 소프트웨어 전용 항목을 건너뛰고 동일한 상호작용 원칙으로 촉진자 역할을 수행하십시오.

## 1. 상호작용 원칙

**한 번에 질문 하나. AskUserQuestion 도구를 우선 사용한다.**

- 메시지당 질문을 하나만 하십시오. 하위 질문이 관련되어 보여도 예외 없이 하나만 하십시오.
- AskUserQuestion 사용 시 `ToolSearch select:AskUserQuestion`으로 스키마를 먼저 로드하십시오.
- 단일 선택 방식을 기본으로 사용하십시오. 답변이 본질적으로 서술적이거나 선택지를 만들기 어려울 때만 자유 텍스트를 사용하십시오.
- 질문을 소리 없이 건너뛰지 마십시오.

## 2. 범위 분류

**기능 설명을 읽고 세 단계 중 하나로 분류한다.**

- **Lightweight**: 작고 범위가 명확하며 모호성이 낮음.
- **Standard**: 일반적인 기능 또는 몇 가지 결정이 필요한 한정된 리팩토링.
- **Deep**: 여러 영역에 걸친 전략적·모호성이 높은 작업. 기존 제품 형상을 상속받는 *feature* 모드와, 제품 형상 자체를 새로 정의해야 하는 *product* 모드로 구분하십시오.
- 범위가 불확실하면 질문 하나로 명확히 한 뒤 분류하십시오.

## 3. 컨텍스트 스캔

**Standard/Deep에서 코드베이스를 먼저 읽는다.**

- Lightweight: 주제를 검색하여 중복 여부만 확인하십시오.
- Standard/Deep: `AGENTS.md`, `STRATEGY.md`(있으면), 관련 브레인스토밍·플랜·스킬 파일을 읽으십시오.
- 무언가가 존재한다고 주장하기 전에 소스 파일을 직접 확인하십시오. 확인 안 된 사항은 "미확인 가정"으로 표시하십시오.
- 스키마·엔드포인트·마이그레이션 등 구현 상세는 계획 단계로 미루십시오.

## 4. Gap 압박 테스트

**접근 방식 제안 전, 해당 격차를 개별 서술형 질문으로 확인한다(메뉴 형식 금지).**

- **Evidence (증거)**: 요청에 실제 사용자 행동(비용 지불, 워크라운드 구축 등)의 근거가 없을 때.
- **Specificity (구체성)**: 대상자가 너무 추상적으로 설명되어 임의 가정이 필요할 때.
- **Counterfactual (반사실)**: 현재 워크라운드나 이것이 없을 때의 비용이 드러나지 않을 때.
- **Attachment (집착)**: 가치보다 특정 해결 방식에 집중하며 더 작은 형태를 검토하지 않을 때.
- **Durability (지속성)**: Deep-product 전용. 가치 제안이 변할 수 있는 세상의 현재 상태에 의존할 때.
- 실제 존재하는 격차만 질문하십시오. 각 격차는 개별적인 직접 서술형 질문으로 대화에 걸쳐 순차적으로 던지십시오.

## 5. 접근 방식 탐색

**2~3가지 제안, 비자명한 각도 하나 포함, 추천안은 나중에 명시한다.**

- 접근 방식을 먼저 제시하고 추천안은 마지막에 설명하십시오.
- 각 접근 방식에 2~3문장 설명, 장단점, 주요 리스크, 적합한 상황을 포함하십시오.
- 메커니즘·제품 형상 수준으로 설명하십시오. 컬럼명·파일경로·클래스명 등 구현 상세는 포함하지 마십시오.
- 한 접근 방식이 압도적으로 좋으면 메뉴를 생략하고 직접 추천하십시오.

## 6. Synthesis Summary

**문서 작성 전, 3-bucket 요약을 제시하고 확인을 받는다.**

- **Stated**: 대화에서 명시적으로 결정된 사항.
- **Inferred**: 에이전트가 채운 가정 — 사용자가 반론할 수 있는 베팅.
- **Out**: 명시적으로 제외된 범위.
- 요약을 제시한 후 턴을 종료하십시오. 사용자 확인 후 섹션 7로 진행하십시오.
- Headless 모드: 확인 없이 진행하고 Inferred 항목을 문서의 `## Assumptions` 섹션에 기록하십시오.

## 7. 요구사항 문서 출력

**`docs/brainstorms/YYYY/MM/DD-<제목>.md` 에 저장한다.**

- 저장 전 `mkdir -p docs/brainstorms/YYYY/MM`를 실행하십시오.
- 포함할 섹션(모든 계층): Summary, Problem Frame, Requirements(R-IDs), Success Criteria, Scope Boundaries, Key Decisions.
- Standard/Deep 추가: Actors(A-IDs), Key Flows(F-IDs), Acceptance Examples(AE-IDs).
- Deep-product 추가: Assumptions 섹션(Headless 모드의 Inferred 항목 포함).
- 파일 참조는 저장소 상대 경로만 사용하십시오(절대 경로 금지).
- Lightweight에서 보존할 결정사항이 없으면 문서를 생략하십시오.
- 다이어그램: 다단계 워크플로우·3개 이상 모드·3개 이상 참여자일 때만 Mermaid 또는 마크다운 표를 인라인으로 추가하십시오.

## 8. 핸드오프

**문서 경로와 핵심 결정사항을 요약하고 다음 단계를 제시한다.**

- `/genie:plan` (권장) — 구현 가드레일 수립.
- 에이전트 문서 리뷰 — 요구사항 문서 검토.
- `/genie:work` — Lightweight 범위에서만 즉시 구현.
- 추가 질문 — 불확실한 사항이 남아 있을 때.
- 지금은 여기까지 — 브레인스토밍만으로 충분할 때.
