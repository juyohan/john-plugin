# 종합 요약 (Synthesis Summary)

**종합(Synthesis) ≠ 요구사항 문서.** 종합은 요구사항 문서의 프리뷰나 초안, 혹은 대체물이 아닙니다 — 이는 문서 작성을 위해 입력값으로 소비되는 범위 체크포인트입니다. 요구사항 문서 자체는 확인된 종합 내용을 바탕으로 Phase 3에서 작성됩니다. 종합과 요구사항 문서는 모두 '범위(scope)'에만 집중합니다 — 구현 세부 사항(파일 경로, 코드 형태, 정확한 에러 문구 등)은 요구사항 문서가 아닌 나중 단계(ce-plan)의 몫입니다. 만약 종합 내용이 요구사항 문서 프리뷰처럼 읽힌다면 잘못 구성된 것입니다 — 범위 전용으로 다시 조정하십시오.

**3개 버킷 구조는 채팅 시의 산출물일 뿐입니다.** 대화를 통해 사용자와 범위를 확정하는 역할을 수행한 뒤, Phase 3에서 문서를 작성할 때 해체됩니다: '명시됨(Stated)' 내용은 Requirements 섹션으로, '추론됨(Inferred)' 내용은 Key Decisions(대화형 모드) 또는 `## Assumptions`(비대화형 모드)로, '범위 제외(Out-of-scope)' 내용은 Scope Boundaries로 이동합니다. 요구사항 문서에는 `## Synthesis`라는 별도의 섹션이 존재하지 않으며 — 오직 산출된 산문 형태의 요약만이 `## Summary`로 포함됩니다. 라우팅 규칙은 아래의 "확인 후 문서 형태"를 참조하십시오.

이 파일은 Phase 2.5가 실행될 때 — 즉, Phase 2(접근 방식 선택)가 완료된 후 Phase 3(요구사항 문서 작성)이 시작되기 전에 로드됩니다. 종합 단계는 문서가 생성되기 전 사용자가 에이전트의 해석을 교정할 수 있는 마지막 기회입니다. 이는 두 가지 목적을 수행합니다: 종합 내용 확인(사용자가 대화 중에 많은 개별 사항에 동의했지만, 전체를 한꺼번에 본 적은 없음)과 전환 체크포인트("이제 문서를 작성하겠습니다") 역할입니다.

경량(Lightweight)을 포함한 **모든 등급**에서 실행됩니다. Phase 0.1b의 비소프트웨어(universal-brainstorming) 경로에서는 Phase 2.5를 완전히 건너뛰십시오 — 해당 플로우에는 자체적인 퍼실리테이션 패턴이 있습니다. 비대화형(headless) 모드에서도 Phase 2.5는 실행됩니다 — 종합 내용은 구성되지만 사용자 확인 과정만 생략됩니다. 추론된 베팅(Inferred bets)은 문서의 `## Assumptions` 섹션으로 라우팅됩니다. 자세한 내용은 아래의 "비대화형 모드"를 참조하십시오.

## 3개 버킷 구조 (Three-bucket structure)

모든 종합 내용은 라벨이 지정된 세 개의 버킷으로 구성됩니다. 항목의 성격에 따라 두 버킷에 동시에 나타날 수 있습니다 — 포함 후 제외하는 경우 '추론됨(Inferred)'으로 표시하여 독자가 에이전트의 추론 과정을 알 수 있게 하십시오.

- **명시됨 (Stated)** — 사용자가 직접 언급한 내용 (원래 프롬프트, 이전 대화, 대화 답변, Phase 2의 접근 방식 선택 등). 이 항목들은 사용자의 언어라는 명확한 근거를 가집니다.
- **추론됨 (Inferred)** — 공백을 메우기 위해 에이전트가 가정한 내용. 사용자가 명시적으로 언급하지 않은 범위 경계, 의도에서 유추한 성공 기준, 짧은 인터뷰에서 다루지 않아 발생한 기술적 가정 등. '추론됨' 목록은 가장 중요한 버킷입니다 — 여기에 포함된 항목은 사용자가 교정할 수 있는 에이전트의 베팅이기 때문입니다.
- **범위 제외 (Out of scope)** — 의도적으로 제외된 항목. 에이전트가 고려했으나 포함하지 않기로 결정한 인접 작업, 리팩토링, 있으면 좋은 기능(nice-to-haves), 향후 작업 항목 등. 제외 사항을 명시함으로써 사용자가 실제로 포함시키고 싶었던 내용이 빠졌는지 확인할 수 있게 합니다.

## 등급별 출력 (Tier-shaped output)

경량(Lightweight) 등급은 한 문단과 짧은 불렛 목록으로 작성합니다. 표준, 심화 기능, 심화 제품 등급은 몇 개의 문단과 버킷별 명시적 목록으로 작성합니다.

종합 단계는 채팅 시의 산출물입니다. 버킷은 사용자가 훑어보고 교정하거나 확인하기 위해 존재하며 — 요구사항 문서로 그대로 옮겨지지 않습니다. 확인 후에는 산문 형태의 요약만 문서에 `## Summary`로 포함되며, 버킷의 내용은 아래의 "확정 후 문서 형태" 규칙에 따라 문서 본문의 각 섹션으로 해체되어 흡수됩니다.

## 프롬프트 템플릿

이는 방향을 제시하는 가이드입니다 — 대화 맥락에 맞춰 표현을 조절하십시오. 상호작용 규칙 5(a)에 따라 산문 형태의 피드백을 유도하십시오 (옵션 세트를 제공하면 유효한 교정 범위에 대한 에이전트의 프레임이 노출되어 답변이 편향될 수 있습니다).

**산문 요약 원칙 (모든 등급 필수):** 종합 내용은 요구사항 문서에서 **제안할 내용을 한눈에 알 수 있도록 1-3행의 산문 요약**으로 시작하십시오. 과거 회상(지금까지 논의한 내용)이 아닌 미래 지향적(문서에 포함될 내용)이어야 합니다. 산문 요약의 역할은 사용자가 불렛을 읽기 전에 자신의 의도와 일치하는지 직관적으로 판단하게 돕는 것입니다 — 개별 불렛에는 동의하더라도 전체적인 방향성에는 동의하지 않을 수 있는데, 산문 요약이 이러한 핵심을 표면화합니다. 경량 등급이라도 전체상을 보여주는 것이 좋습니다. 총 불렛이 2-3개를 넘는 종합 내용이라면 상단에 1-3행의 요약을 포함하십시오. 생략 가능한 유일한 경우는 극히 사소한 사례(예: 사용자의 프롬프트 자체가 "47행 오타 수정"과 같은 완전한 범위 선언이었고 종합 내용이 이를 단순히 반복하는 1-2개의 불렛인 경우)뿐입니다.

**산문 요약은 종합 섹션 내부**, 도입부 바로 뒤이자 '명시됨' 버킷 앞에 위치해야 합니다 — 종합 섹션 위의 별도 블록으로 두지 마십시오. 종합 섹션 위로 방대한 산문(접근 방식 피치, 동작 불렛, 이론적 근거 등)을 배치하는 것은 구조를 뒤집는 것입니다. 종합 내용이 제안 사항에 대한 주석이 되는 것이 아니라, 제안 사항이 종합 내용에 대한 1-3행의 설명이 되어야 합니다.

**불필요한 수식어 지양:** 산문 요약이 "이것은 ~에 대한 실질적인 제안으로..." 또는 "이 종합 내용은 ~에 대한 중요한 우려 사항을 다루며..."로 시작한다면 멈추고 다시 쓰십시오. 제안하는 핵심 내용을 평이한 언어로 바로 기술하십시오. 수식어("포괄적인", "사려 깊은", "실질적인")를 사용하지 마십시오. 사용자가 방금 겪은 대화 맥락을 재진술하지 마십시오. 채워 넣기용 문구 없이 1-3행으로 작업 내용을 설명할 수 없다면 아직 종합 단계로 넘어올 준비가 되지 않은 것입니다.

**안티 패턴: 피치(pitch) 형태의 종합.** 만약 종합 섹션 상단에 파일 경로, JSON 형태, 에러 메시지 등이 포함된 "권장 사항" / "X일 때의 동작" / "이러한 형태인 이유" 블록을 작성하고 있다면 멈추십시오. 해당 내용은 Phase 3(문서 작성)의 영역입니다 — 종합 단계가 아닌 다음 단계에서 작성될 요구사항 문서 본문에 들어가야 할 내용입니다. 종합은 범위 체크포인트입니다: 3개의 버킷과 1-3행의 요약이 전부입니다. 구현 세부 사항이 종합 단계로 유출되는 것은 Phase 2(접근 방식 선택)와 Phase 3(문서 작성)이 Phase 2.5로 뭉개졌다는 신호입니다.

```
대화 내용과 선택하신 접근 방식을 바탕으로, 요구사항 문서에 담을 범위를 다음과 같이 정리했습니다.

[1-3행의 산문 요약 — 제안하는 내용을 평이한 언어로 작성하십시오. 모든 등급 필수; 종합 내용이 프롬프트를 반복하는 2개 이하의 불렛인 극히 사소한 경우에만 생략 가능.]

**명시됨 (Stated)** (제공해주신 입력값과 대화 내용 기반):
- [항목]
- [항목]

**추론됨 (Inferred)** (공백을 메우기 위해 가정한 내용 — 잘못된 부분이 있다면 말씀해 주세요):
- [항목]
- [항목]

**범위 제외 (Out of scope)** (의도적으로 제외한 항목):
- [항목]
- [항목]

이 내용이 의도하신 바와 일치하나요? 추가, 삭제, 방향 수정이 필요한 부분이나 제가 잘못 이해한 부분이 있다면 말씀해 주세요. 문제가 없다면 확인을 부탁드립니다. (앞서 말씀하신 내용이 종합에 정확히 반영되었더라도, 마음이 바뀌었거나 새로운 컨텍스트가 생겼거나 명시되지 않은 가정을 교정하고 싶으시다면 언제든 이견을 주셔도 좋습니다.)
```

사용자 응답에는 산문을 사용하십시오 (`AskUserQuestion` 메뉴 사용 금지). 근거는 SKILL.md의 상호작용 규칙 5(a)입니다 — 옵션 세트는 어떤 차원이 중요한지를 암시하여 답변을 편향시킬 수 있습니다.

## 수정 후 재제시; 확인 후에만 문서 작성

수정은 확인이 아닙니다. 사용자의 수정 요청(예: "`--foo`를 `--bar`로 변경"과 같은 간단한 수정이라도)이 있은 후에는 변경 사항을 통합하고, 수정된 종합 내용을 다시 제시하여 반영되었음을 보여준 뒤, 명시적인 확인을 받은 후에 문서를 작성하십시오. 루프 순서는 다음과 같습니다:

1. 종합 내용 제시 → 사용자 응답
2. 사용자 확인 → 문서 작성
3. 사용자 수정 요청 → 반영 후 수정된 종합 내용 재제시, 단계 1로 복귀

문서 작성은 명시적인 확인이 있거나, 순환 수정 방지용 차단 질문(아래 참조)에서 "진행" 옵션을 선택한 후에만 트리거됩니다. 확인 단계는 이 종합 내용을 "에이전트의 마지막 제안"이 아닌 **확정된(confirmed)** 상태로 만듭니다 — 수정 요청 직후에는 에이전트가 완벽히 이해했다고 느끼더라도 즉시 문서를 작성하지 마십시오.

## 세분화: 결정 사항을 명명하되 확장하지 마십시오

'명시됨' 및 '추론됨'의 각 불렛은 사용자가 **코드를 읽거나 구현 세부 사항을 찾아보지 않고도** 긍정하거나 거부할 수 있어야 합니다. 사용자가 판단할 수 있는 범위 수준에서 결정 사항의 이름을 정하십시오 — 그보다 구체적인 내용은 Phase 3(문서 작성)의 콘텐츠입니다.

**종합 단계 불렛에 허용되지 않는 내용** (항상 문서 본문용):
- 구현 경로, 파일 이름, 메서드 이름 또는 클래스 참조
- 정확한 JSON / 스키마 형태 또는 필드 이름
- HTTP 상태 코드 또는 유선 포맷(wire format)
- 에러 메시지나 UI 레이블의 정확한 문구
- SQL 문법, 쿼리 본문 또는 특정 컬럼 참조

### 잘못된 예시 vs 올바른 예시

| 종합에 포함된 문서 본문 내용 (잘못됨) | 결정 수준의 내용 (올바름) |
|---|---|
| Manifest 검색 시 `~/printing-press/manuscripts/<api-slug>/`를 탐색하고 `research/` 아래에 `*-absorb-manifest.md`를 포함하는 최신 실행을 선택함 | 검색 규칙: 해당 API에 대한 가장 최근의 이전 실행 결과물 선택 |
| `.printing-press.json`의 `absorb_source_run` 필드 아래에 기록됨 | 출처 기록: 각 실행은 어떤 이전 Manifest를 재사용했는지 기록함 |
| 스킬 실행 시 "Reusing absorb manifest from run \<run-id\>" 출력 | 플래그 실행 시 트랜스크립트에 명시적인 재사용 신호 노출 |

테스트 방법: 불렛을 읽는 사람이 아키텍처 관점이 아닌 제품 / 범위 관점에서 이를 긍정하거나 거부할 수 있어야 합니다. 파일 경로 나 스키마 세부 사항을 평가해야 한다면 세분화 수준이 잘못된 것입니다.

## 순환 수정 시의 차단 (반복 횟수가 아닌 순환 여부 기준)

사용자가 라운드당 어떤 항목을 건드렸는지 추적하십시오. 차단 질문은 **동일한 항목이 두 번 수정되었을 때** (또는 3라운드 수정이 이미 2라운드에서 수정된 항목을 겨냥할 때) 트리거됩니다. 라운드를 거치며 새로운 항목을 수정하는 것은 제한 없이 진행됩니다 — 종합 내용의 서로 다른 측면(예: 사용자가 '명시됨'에 대해 이견을 낸 뒤 다음 라운드에서 '추론됨'에 대해 이견을 냄)을 교정하는 것은 이 메커니즘이 지원해야 할 핵심 동작입니다.

차단 질문이 실행될 때, 플랫폼의 차단형 질문 도구(`AskUserQuestion`, `request_user_input`, `ask_user`)를 사용하여 다음 두 옵션을 제시하십시오:

- `현재 수정된 종합 내용으로 진행`
- `중단 후 방향 재설정 — 문서 작성 전 대화 더 나누기`

차단 도구가 없거나 호출 에러가 발생하는 경우에만 채팅창에 숫자가 매겨진 목록으로 렌더링하십시오. 임의로 생략하지 마십시오.

---

## 비대화형 모드 (Headless mode)

LFG나 `disable-model-invocation` 컨텍스트와 같은 자동화된 워크플로우에서 스킬이 호출되는 경우, 비대화형 모드(실시간 사용자 없음)로 실행됩니다. 이것이 검증되지 않음을 의미하지는 않습니다 — 산출물은 다음 단계의 스킬(ce-doc-review, ce-plan)과 사람 리뷰어(PR 리뷰)에 의해 읽힙니다. 검토의 장이 채팅 이력에서 산출물 자체로 옮겨가는 것입니다.

동작 방식:

- 대화형 모드와 동일하게 **종합 내용을 구성**하십시오. 강제 기능(forcing function)은 유지됩니다. 에이전트는 자신의 범위 해석을 명시적으로 기술해야 합니다.
- **사용자 프롬프트 없음; 차단 질문 없음.** 확인 단계를 건너뜁니다.
- **모드 인식형으로 문서에 내용 라우팅:**
  - **명시됨 (Stated)** 내용 → 요구사항 (사용자가 실제 명시한 제약 사항)
  - **범위 제외 (Out-of-scope)** 내용 → 범위 경계 (의도적 제외 사항)
  - **추론됨 (Inferred)** 내용 → `## Assumptions` 섹션 — 나중 단계의 리뷰에서 정밀하게 검토해야 할 검증되지 않은 에이전트의 베팅임을 명시적으로 표시합니다. 추론된 항목을 Key Decisions나 Requirements로 라우팅하지 마십시오. 그렇게 되면 검증되지 않은 베팅과 사용자 확인을 거친 결정 사항을 구분할 수 없게 됩니다.

`## Assumptions` 섹션은 비대화형 문서에서만 나타납니다. 대화형 모드에서는 추론된 베팅이 채팅을 통해 교정되어 결정 사항이 되므로 이 섹션은 존재하지 않습니다.

이를 통해 원래 설계 의도였던 감사 가시성(검증되지 않은 베팅이 확정적 내용으로 전파되어서는 안 됨)을 복원하면서도, 이를 숨기지 않고 별도의 라벨 아래에 표면화합니다. 다음 단계의 리뷰에서는 Assumptions 섹션을 집중적으로 검토할 수 있습니다.

---

## 자가 리다이렉션 (Self-redirect)

사용자의 응답이 잘못된 스킬에 있거나 다른 워크플로우를 원한다는 신호인 경우 (예: "이건 너무 간단해요, 그냥 /ce-work 해주세요" 또는 "이건 생각이 더 필요해요, 브레인스토밍을 다르게 해볼게요"):

- ce-brainstorm을 중단하십시오.
- 사용자가 원하는 것으로 보이는 대안 스킬을 제안하십시오 (예: `/ce-work`, `/ce-debug`).
- 세션 내에서 해당 스킬을 로드할 것을 제안하십시오.
- 반박하거나 설득하려 하지 마십시오 — 사용자의 리다이렉션 신호는 의도적인 선택입니다.

이 지원 기능은 종합 단계가 정직한 체크포인트이기 때문에 존재합니다. 사용자가 종합 내용을 읽고 스킬 선택이 잘못되었음을 깨달았다면, 리다이렉션하는 것이 올바른 조치입니다.

---

## 확인 후 문서 형태 (Doc shape after confirmation)

사용자 확인 후(또는 차단 질문에서 진행이 결정된 후), Phase 3에서 요구사항 문서를 작성합니다. 3개 버킷 구조는 요구사항 문서에 `## Synthesis`라는 섹션으로 그대로 옮겨지지 않습니다. 오직 산문 요약만 상단에 `## Summary`로 포함됩니다. 버킷의 내용은 문서 본문의 각 섹션으로 해체되어 흡수됩니다:

| 종합 단계 요소 | 요구사항 문서 내 위치 |
|---|---|
| 산문 요약 | `## Summary` (1-3행, 미래 지향적, 제안 내용) |
| 명시됨(Stated) 불렛 | `## Requirements` (번호가 매겨진 R-ID, 상세 내용) 및 필요한 경우 내러티브 컨텍스트를 위한 `## Problem Frame` |
| 추론됨(Inferred) 불렛 | `## Key Decisions` (이론적 근거 포함) — 대화에서 수용된 베팅은 문서에서 결정 사항이 됨 |
| 범위 제외(Out-of-scope) 불렛 | `## Scope Boundaries` |

이탤릭체로 된 캡처 컨텍스트 노트(예: "Phase 2.5에서 캡처됨...")는 포함하지 마십시오. 독자에게 필요 없는 엔지니어링 공정 정보를 산출물에 노출하는 셈입니다.

문서의 `## Summary`와 `## Problem Frame`은 서로 다른 목적을 수행해야 합니다 — 규칙은 `references/requirements-capture.md`의 "Summary vs Problem Frame discipline"을 참조하십시오.

## 종합 단계가 불필요한 경우

사용자의 프롬프트 자체가 이미 완전한 범위 선언이었던 극히 사소한 경량(Lightweight) 사례(예: "47행 오타 수정")의 경우, 종합 내용은 대부분 '명시됨'이며 '추론됨'이나 '제외' 항목이 없습니다. 전환 체크포인트는 여전히 가치가 있지만("이제 문서를 작성하겠습니다. 확인이나 중단을 해주세요"), 출력 내용은 군더더기 없이 한 문단으로 유지하십시오. 철저해 보이기 위해 버킷을 억지로 채우지 마십시오.

---

## 종합 단계에 포함되지 않는 것

- 구현 세부 사항 (라이브러리, 스키마, 파일 경로, JSON 형태, 정확한 에러 메시지 문구) — 이것들은 범위 수준의 종합이 아닌 Phase 3 기획 시의 콘텐츠입니다.
- 대화 전체의 재진술 — 종합은 요약이지 트랜스크립트가 아닙니다.
- 방어적인 가정과 모호한 표현 — 우려 사항이 실재한다면 '추론됨'이나 '범위 제외'로 명시하고, 단순한 추측이라면 제외하십시오.
- 3개 버킷 외부에 노출된 미해결 질문 — 종합 단계에 도달했을 때 모든 범위 형성 질문은 **명시됨** (이전에 묻고 답함), **추론됨** (교정을 위한 에이전트의 베팅), 또는 **범위 제외** (의도적 제외) 중 하나에 속해야 합니다. 네 번째 상태는 없습니다. 만약 질문에 대한 기본값을 정할 수 없다면, 종합 제시를 멈추고 먼저 질문을 해결하십시오. 옵션이 제한적이고 유의미하게 구분될 때는 차단형 다지선다 도구를 사용하고, 옵션 세트가 답변을 편향시킬 수 있을 때는 상호작용 규칙 5(a)에 따라 산문 형태의 질문을 사용하십시오. 답변을 통합한 뒤 종합 내용을 제시하십시오. 종합 내용 옆에 해결되지 않은 질문을 나란히 두지 마십시오 — 이는 사용자에게 명확한 해결 경로를 제공하지 못합니다.
