# Java & 헥사고날 아키텍처 코드 스멜 (Code Smells)

> **파일 규약 참고:** 이 파일(`code-smells.md`)은 Java 전용 추가 파일입니다. 다른 언어 디렉토리에는 대응하는 파일이 없으며, 헥사고날 아키텍처 관련 안티 패턴이 `coding-style.md`에 담기 너무 길어 분리되었습니다. 일반 코드 스멜은 `coding-style.md`를 참조하십시오.

> 이 파일은 프로젝트의 품질과 아키텍처 일관성을 유지하기 위해 피해야 할 안티 패턴을 정의합니다.

## 1. 아키텍처 및 레이어링 스멜

### 리포지토리 인터페이스 미준수 (MANDATORY)
- **스멜**: `domain` 레이어에서 구체적인 저장소 구현체에 직접 의존하거나, 인터페이스 없이 구현체만 존재하는 경우.
- **해결**: 반드시 `domain/repository` 또는 `application/port/out`에 **인터페이스**를 정의하고, 구현체는 `adapter/out/persistence` 레이어에서 관리하십시오.

### 컨트롤러 내 비즈니스 로직 포함 (MANDATORY)
- **스멜**: `adapter/in/controller` 내에서 복잡한 계산, 조건문, 또는 여러 서비스를 조합하는 비즈니스 로직이 포함된 경우.
- **해결**: 컨트롤러는 요청 수신 및 응답 반환의 책임만 가져야 합니다. 모든 비즈니스 로직은 `application/service` 또는 `domain/service`로 이동하십시오.

### 도메인 모델의 외부 노출 (MANDATORY)
- **스멜**: 컨트롤러가 `domain` 모델(Entity/VO) 또는 `persistence` 엔티티를 직접 반환하는 경우.
- **해결**: API 응답은 반드시 전용 **반환용 DTO(Response DTO)**를 통해 수행하십시오. 이는 내부 모델의 변경이 외부 API 스펙에 영향을 주는 것을 방지합니다.

## 2. 헥사고날 & CQRS 관련 스멜

### 레이어 간 의존성 역전
- **스멜**: `domain` 레이어가 `adapter` 또는 `application` 레이어를 참조하는 경우.
- **해결**: 의존성은 항상 `Adapter -> Application -> Domain` 방향으로만 흘러야 합니다.

### 혼합된 Command와 Query
- **스멜**: 하나의 서비스 메서드에서 상태 변경(Command)과 복잡한 대량 데이터 조회(Query)를 동시에 수행하는 경우.
- **해결**: CQRS 원칙에 따라 상태 변경 로직과 조회 로직을 분리하고, 조회 전용 DTO와 최적화된 쿼리를 사용하십시오.

## 3. 코드 품질 및 클린 코드

### 무분별한 Setter 및 가변성
- **스멜**: 도메인 객체에 public setter를 열어두어 어디서든 상태를 변경할 수 있게 만든 경우.
- **해결**: 생성자나 의미 있는 비즈니스 메서드를 사용하고, 가능한 한 `final` 필드와 `record`를 활용하여 불변성을 유지하십시오.

### 불명확한 예외 처리
- **스멜**: 원인 파악이 어려운 일반적인 예외(RuntimeException 등)를 던지거나, 로그 없이 예외를 삼키는 경우.
- **해결**: 도메인 전용 예외(예: `UserNotFoundException`)를 정의하고, 에러 메시지에 컨텍스트 정보를 포함하십시오.

### 매직 넘버 및 하드코딩
- **스멜**: 코드 내에 의미를 알 수 없는 숫자나 문자열 리터럴이 흩어져 있는 경우.
- **해결**: 의미 있는 이름을 가진 상수로 정의하여 사용하십시오.
